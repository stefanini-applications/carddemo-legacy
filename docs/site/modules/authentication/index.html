🔐 Authentication

User Signon, Credential Validation & Session Management

📋 Overview

Module ID: authentication

Purpose: Controls all user access to CardDemo. Validates user credentials against the USRSEC VSAM file, establishes user session context via COMMAREA, and determines user role (Admin vs. Regular User) for the entire session.

Business Context: Entry point for all application users. A successful login populates CARDDEMO-COMMAREA with user ID and user type, which all downstream programs rely on for authorization checks.

🎯 User Story Templates

As a regular user, I want to sign in with my user ID and password so that I can access my account information.
As an admin, I want to sign in with admin credentials so that I can access user management functions.
As a security admin, I want the system to reject invalid credentials so that unauthorized users cannot access the application.
As a user, I want to sign out (PF3) from the signon screen so that my session is terminated safely.

📊 Story Complexity

Simple 1-2 pts Medium 3-5 pts Complex 5-8 pts

Simple (1-2 pts)

  • Display signon screen (COSGN00.bms)
  • Clear signon fields (PF4)
  • Return thank-you on PF3 logout

Medium (3-5 pts)

  • Validate user ID + password against USRSEC
  • Set COMMAREA user type (admin vs. user)
  • Route to correct menu after login

Complex (5-8 pts)

  • Implement password change flow
  • Add account lockout after N failed attempts
  • Integrate with RACF for enhanced security

✅ Acceptance Criteria Patterns

Login Success: Given valid User ID and Password, when user presses ENTER, then COMMAREA is populated with USER-ID and USER-TYPE, and user is routed to appropriate menu (CM00 for users, CA00 for admins).
Login Failure: Given invalid credentials, when user presses ENTER, then error message is displayed in ERRMSGO field and credentials are cleared.
Empty Fields: Given blank User ID or Password, when ENTER is pressed, then validation error is shown and cursor is positioned on the offending field.
Logout: Given user presses PF3, then thank-you message is displayed and CICS RETURN is called with transaction CC00.
Invalid Key: Given any key other than ENTER or PF3, then message "Invalid key pressed" is displayed.

COMMAREA Set by Authentication

01 CARDDEMO-COMMAREA.
   10 CDEMO-USER-ID      PIC X(08)  -- authenticated user ID
   10 CDEMO-USER-TYPE    PIC X(01)  -- 'A'=admin, 'U'=user
   10 CDEMO-TO-TRANID    PIC X(04)  -- next transaction to invoke
   10 CDEMO-TO-PROGRAM   PIC X(08)  -- next program to invoke

USRSEC Record Structure

01 SEC-USER-DATA.
   05 SEC-USR-ID      PIC X(08)  -- VSAM key, login user ID
   05 SEC-USR-FNAME   PIC X(20)  -- first name
   05 SEC-USR-LNAME   PIC X(20)  -- last name
   05 SEC-USR-PWD     PIC X(08)  -- password (plaintext)
   05 SEC-USR-TYPE    PIC X(01)  -- 'A'=admin, 'U'=user

🔗 Dependencies

  • USRSEC VSAM: Must be loaded via DUSRSECJ before first use
  • COCOM01Y.cpy: Shared COMMAREA — used by ALL other modules
  • CICS COMMAREA: Passed to every subsequent transaction
  • COMEN01C (CM00): Regular user landing after successful login
  • COADM01C (CA00): Admin user landing after successful login

⚡ Development Acceleration

  • COMMAREA pattern: Established in COCOM01Y.cpy — all programs follow the same session handoff
  • BMS map pattern: COSGN00.bms + COSGN00.CPY — copy pattern for any new screen
  • EXEC CICS FILE READ: Standard VSAM read pattern with RESP/REAS error handling
  • AID key evaluation: EVALUATE EIBAID pattern used in all online programs

⚠️ Known Issues / Modernization Notes

  • Passwords stored in plaintext (SEC-USR-PWD PIC X(08)) — modernization must implement hashing
  • No account lockout on repeated failed attempts — security risk for production
  • Default credentials: ADMIN001/PASSWORD and USER0001/PASSWORD — must be changed in production
  • User type 'A' vs 'U' is case-sensitive — must match exactly
