🗄️ Transaction Type DB2 Optional Module

DB2 Transaction Type Reference Data Management — CTTU / CTLI

📋 Overview

Module ID: transaction-type-db2

Purpose: Optional extension that manages transaction type reference data in DB2 using embedded static SQL. Admin users can add, update, list, and delete transaction type codes via CICS transactions. Demonstrates DB2 integration patterns including cursors, CRUD operations, and SQLCA error handling.

ℹ️ Installation required: Requires DB2 setup via CREADB21 job. See app/app-transaction-type-db2/README.md. After setup, TRANEXTR batch job syncs DB2 data to VSAM for use by base application.

🎯 User Story Templates

As an admin, I want to add a new transaction type code in DB2 so that new transaction categories are available in the system.
As an admin, I want to list and update existing transaction types so that reference data stays accurate.
As an admin, I want to delete obsolete transaction types from DB2 so that the reference table stays clean.
As a batch system, I want TRANEXTR to extract DB2 transaction types to VSAM so that online programs can use the latest reference data.

✅ Acceptance Criteria Patterns

List Types (CTLI): When CTLI loads, then all transaction types in DB2 are listed using forward cursor; PF7/PF8 navigates backward/forward through cursor results.
Add Type (CTTU): Given unique transaction type code, description, and category, when ENTER pressed, then SQL INSERT executes and SQLCA SQLCODE = 0 indicates success.
Duplicate Key: Given transaction type code already exists in DB2, then SQLCODE = -803 (duplicate key) is handled and user-friendly error message displayed.
Delete Type (CTLI): Given admin marks type for deletion, when confirmed, then SQL DELETE executes; if code is referenced by transactions, referential integrity error is shown.
TRANEXTR Sync: After DB2 update, when TRANEXTR batch runs, then VSAM TRANTYPE file is refreshed with current DB2 transaction type data.

DB2 Integration Patterns Demonstrated

-- Static embedded SQL with host variables
EXEC SQL
    INSERT INTO TRANTYPE (TRAN-TYPE-CD, TRAN-TYPE-DESC)
    VALUES (:WS-TRAN-TYPE-CD, :WS-TRAN-TYPE-DESC)
END-EXEC
-- Check SQLCA
IF SQLCODE = 0 THEN success ELSE handle error

-- Forward cursor processing
EXEC SQL DECLARE TYPE-CURSOR CURSOR FOR
    SELECT * FROM TRANTYPE ORDER BY TRAN-TYPE-CD
END-EXEC
EXEC SQL OPEN TYPE-CURSOR END-EXEC
EXEC SQL FETCH TYPE-CURSOR INTO :host-vars END-EXEC

🔗 Dependencies

  • DB2: Must be installed and CREADB21 job run first
  • DB2 precompiler: COBOL programs require DB2 precompile step in build
  • transactions: TRAN-TYPE-CD in TRANFILE references these codes
  • batch-processing: TRANEXTR syncs DB2 to VSAM TRANTYPE file
  • user-management (admin access): CA00 admin menu hosts CTTU/CTLI options

⚡ Development Acceleration

  • SQLCA pattern: Established in COTRTUPC/COTRTLIC — copy for any DB2 program
  • Cursor forward/backward: COTRTLIC shows both FETCH and FETCH PRIOR patterns
  • DB2 DCL files: DECLARE TABLE statements in app/app-transaction-type-db2/dcl/
  • DDL files: Table definitions in app/app-transaction-type-db2/ddl/ — ready to extend
