authorization

The authorization module implements CardDemo pending authorization processing across MQ, CICS, IMS DB, and DB2 fraud logging. It introduces three online transactions (CP00, CPVS, CPVD) and one batch purge pipeline (CBPAUP0J/CBPAUP0C).

Module Overview and Purpose

This module extends the core CardDemo transaction system with pending authorization capture and review flows. Requests arrive through MQ, are evaluated in CICS, and are stored in IMS as pending authorization data. Fraud decisions are captured interactively by analysts in CPVD and persisted into DB2 table CARDDEMO.AUTHFRDS.

It is designed as a demonstrator of hybrid mainframe integration, combining: CICS online interaction, MQ trigger-based ingestion, IMS HIDAM hierarchical persistence, and DB2 fraud event reporting.

Primary Objective

Evaluate and store pending card authorization requests with low-latency response behavior.

Secondary Objective

Enable analyst-led fraud tagging and DB2 audit trail entries through CPVD PF5 action.

Operational Objective

Purge expired IMS pending records via batch process to control dataset growth.

Architectural Context

The module sits on top of core platform assets and uses VSAM account/customer/card data to enrich and validate incoming authorization requests. 

External Client
  -> MQ Request Queue (AWS.M2.CARDDEMO.PAUTH.REQUEST)
  -> CICS Trigger (CP00 / COPAUA0C)
  -> VSAM lookups (CCXREF, ACCTDAT, CUSTDAT)
  -> Decision + MQ reply payload
  -> IMS persist (PAUTSUM0 / PAUTDTL1)

Analyst 3270
  -> CPVS (COPAUS0C) summary
  -> CPVD (COPAUS1C) detail
  -> LINK COPAUS2C for DB2 fraud add/update
  -> IMS detail REPL + CICS SYNCPOINT

Batch Window
  -> CBPAUP0J runs CBPAUP0C (DFSRRC00 BMP)
  -> removes expired PAUTDTL1
  -> removes empty PAUTSUM0

Consistency Model

  • CP00 reads MQ input and replies with decision payload using message correlation id.
  • IMS summary/detail updates are executed after response payload assembly.
  • CPVD fraud action uses CICS unit-of-work semantics (SYNCPOINT/ROLLBACK).
  • DB2 fraud insert/update success is required before IMS fraud tag commit in CPVD flow.

Data and Business Rules

Decision Rules in CP00

  1. Attempt card lookup in CCXREF; if not found, decline reason 3100.
  2. If account context is present, compute available amount using: PA-CREDIT-LIMIT - PA-CREDIT-BALANCE (IMS summary) or ACCT-CREDIT-LIMIT - ACCT-CURR-BAL (VSAM fallback).
  3. If transaction amount exceeds available amount, decline reason 4100.
  4. Approved responses carry code 00; declined responses carry code 05.
  5. Default unknown decline reason is 9000.

Fraud Marking Rules (CPVD + COPAUS2C)

  • PF5 toggles between PA-FRAUD-CONFIRMED (F) and PA-FRAUD-REMOVED (R).
  • DB2 insert into AUTHFRDS occurs first; duplicate key (SQLCODE -803) triggers UPDATE path.
  • Only after DB2 update success does CPVD REPL IMS detail and issue SYNCPOINT.
  • On failure, CPVD performs SYNCPOINT ROLLBACK and displays error message.

Retention Rules (CBPAUP0C)

  • Expiration check uses CURRENT-YYDDD - (99999 - PA-AUTH-DATE-9C).
  • If day difference exceeds configured expiry threshold, detail record is deleted.
  • Summary counters/amounts are decremented during detail deletion.
  • When summary counts reach zero, summary root segment is deleted.
  • Periodic checkpoint frequency is controlled by runtime SYSIN.

Development Guidelines and Patterns

Pattern 1: IMS Access Envelope

  • Schedule PSB with EXEC DLI SCHD.
  • Handle TC (already scheduled) by TERM + re-SCHD.
  • Use GU for root locate and GNP for child traversal.
  • Map non-success status to message area and/or error log queue.

Pattern 2: CICS UI Re-entry Contract

  • First entry initializes map and commarea.
  • Re-entry reads map input via EXEC CICS RECEIVE.
  • PF key dispatch drives back, paging, detail, and fraud action flows.
  • Program returns with TRANSID and updated commarea state.

Pattern 3: DB2 Idempotent Fraud Write

  • Attempt INSERT first.
  • On duplicate key (SQLCODE = -803) perform UPDATE.
  • Return concise action status message to caller.

Pattern 4: Operational Safety in Batch Purge

  • Checkpoint after configurable number of summaries.
  • Emit counters for read/deleted summary/detail records.
  • ABEND path sets non-zero return code.

Do/Don't Recommendations

Do Don't
Preserve correlation id when constructing MQ reply descriptors. Do not generate a new arbitrary correlation id for replies.
Keep IMS and DB2 updates under explicit syncpoint control in CPVD. Do not commit partial fraud state when DB2 update fails.
Retain reason-code dictionary parity between CP00 and CPVD. Do not add new reason code only on one side.
Use copybooks for request/response segment contracts. Do not handcraft duplicate field layouts in each program.

Example 5: PF5 Fraud Toggle with DB2/IMS Coordination

IF PA-FRAUD-CONFIRMED
   SET PA-FRAUD-REMOVED TO TRUE
   SET WS-REMOVE-FRAUD  TO TRUE
ELSE
   SET PA-FRAUD-CONFIRMED TO TRUE
   SET WS-REPORT-FRAUD    TO TRUE
END-IF

EXEC CICS LINK PROGRAM(WS-PGM-AUTH-FRAUD)
     COMMAREA(WS-FRAUD-DATA)
END-EXEC

IF WS-FRD-UPDT-SUCCESS
   EXEC DLI REPL USING PCB(PAUT-PCB-NUM)
       SEGMENT (PAUTDTL1)
       FROM (PENDING-AUTH-DETAILS)
   END-EXEC
   EXEC CICS SYNCPOINT END-EXEC
ELSE
   EXEC CICS SYNCPOINT ROLLBACK END-EXEC
END-IF

Example 6: Duplicate-Key DB2 Update Path

EXEC SQL
   INSERT INTO CARDDEMO.AUTHFRDS (...)
   VALUES (...)
END-EXEC

IF SQLCODE = ZERO
   SET WS-FRD-UPDT-SUCCESS TO TRUE
ELSE
   IF SQLCODE = -803
      EXEC SQL
         UPDATE CARDDEMO.AUTHFRDS
            SET AUTH_FRAUD = :AUTH-FRAUD,
                FRAUD_RPT_DATE = CURRENT DATE
          WHERE CARD_NUM = :CARD-NUM
            AND AUTH_TS = TIMESTAMP_FORMAT(:AUTH-TS, 'YY-MM-DD HH24.MI.SSNNNNNN')
      END-EXEC
   END-IF
END-IF

Example 7: Expiry Check in Batch Purge

COMPUTE WS-AUTH-DATE = 99999 - PA-AUTH-DATE-9C
COMPUTE WS-DAY-DIFF = CURRENT-YYDDD - WS-AUTH-DATE

IF WS-DAY-DIFF >= WS-EXPIRY-DAYS
   SET QUALIFIED-FOR-DELETE TO TRUE
   EXEC DLI DLET USING PCB(PAUT-PCB-NUM)
       SEGMENT (PAUTDTL1)
       FROM (PENDING-AUTH-DETAILS)
   END-EXEC
END-IF

Operations and Batch Lifecycle

Install/Deploy Checklist

  1. Load IMS DBDs/PSBs for DBPAUTP0, DBPAUTX0, PSBPAUTB, PSBPAUTL.
  2. Create DB2 table/index from AUTHFRDS.ddl and XAUTHFRD.ddl.
  3. Compile and NEWCOPY programs COPAUA0C, COPAUS0C, COPAUS1C, COPAUS2C, CBPAUP0C.
  4. Install CICS resources from CRDDEMO2.csd.
  5. Bind DB2 plan and install DB2TRAN(CPVDTRAN) for CPVD.
  6. Verify MQ queue manager trigger wiring for CP00 invocation.
  7. Schedule CBPAUP0J in maintenance window.

Operational Signals

  • COPAUA0C logs error records to transient data queue CSSL.
  • CBPAUP0C writes summary counters and checkpoint progress to SYSOUT.
  • CPVD UI messages indicate fraud add/remove success or rollback failure.

Failure Modes

  • IMS schedule failure (TE/BA) blocks read/write for CP00/CPVS/CPVD.
  • MQ open/get/put failures prevent CP00 request lifecycle completion.
  • DB2 binding or table issues block fraud persistence in CPVD PF5 path.
  • Missing VSAM xref/account/customer entries force decline path and reason 3100.

Validation and Testing Notes

Recommended Functional Scenarios

  1. Valid request under credit limit -> expect response 00 and IMS detail insert.
  2. Request above available amount -> expect response 05 reason 4100.
  3. Unknown card in xref -> expect response 05 reason 3100.
  4. CPVS PF7/PF8 navigation across multiple pages of pending detail entries.
  5. CPVD PF5 first press inserts fraud row in DB2 and tags IMS detail F.
  6. CPVD PF5 second press updates DB2 row and toggles IMS detail to R.
  7. CBPAUP0J with short expiry threshold deletes old detail + zero-count summary segments.

Acceptance Criteria Template

Given CP00 receives a well-formed request from MQ,
When the card exists and available amount covers transaction amount,
Then CP00 returns response code 00,
And writes/updates IMS pending authorization summary,
And inserts IMS pending authorization detail with match status P.

Given CPVD displays an authorization detail,
When analyst presses PF5,
Then COPAUS2C persists AUTH_FRAUD in DB2,
And CPVD rewrites IMS detail and commits with CICS SYNCPOINT.

Performance Budgets

  • Target CP00 MQ round trip under 2 seconds including lookups and payload generation.
  • Target CPVS/CPVD screen response under 1 second for common account sizes.
  • Target CBPAUP0J completion within scheduled maintenance window.
