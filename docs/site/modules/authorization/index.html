📦 Authorization

Guide for building user stories around the MQ/IMS/DB2 authorization extension.

📋 Overview

Purpose: Showcase a multi-system credit card authorization pipeline that validates requests via MQ, persists decisions in IMS, and captures fraud metadata in DB2.

Business Context: This module proves how a workaround for PCI-style authorizations can live on the mainframe: MQ clients send requests, CICS/MQ programs (`CP00`) apply business rules, responses are queued, and auditors review stored IMS/DB2 data.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Authorization flow: As an MQ-integrated merchant simulator, I want to send authorization requests and receive approvals or declines so that my downstream POS system knows whether to accept the charge.
  • Fraud tracking: As an analyst, I want to flag a suspicious authorization and have `AUTHFRDS` capture the fraud indicator before the next reporting cycle.
  • Cleanup: As operations, I want to purge expired authorizations nightly so IMS storage stays performant.

⚡ Development Acceleration Factors

  • Message formats: Request/response copybooks in `app/app-authorization-ims-db2-mq` already define `DATE-REQUEST-MSG`, `ACCT-REQUEST-MSG`, etc., so new MQ listeners can reuse the layouts.
  • Shared DDL: The `AUTHFRDS` schema and index SQL scripts are ready to drop into DB2 and only need schema names replaced.
  • CICS resources: Provided `csd/` definitions for `CP00`, `CPVS`, `CPVD`, and DB2 plans simplify deployment; reuse them when introducing more MQ-triggered flows.

📊 Complexity Guidelines

  • Simple: Adjust the MQ request validation (e.g., allow additional `REQUEST-TYPE` codes) by editing the COBOL field processing in `COPAUA0C`.
  • Medium: Add a new authorization status screen in CICS, link it to `CPVD`, and update DB2 inserts accordingly.
  • Complex: Introduce a new IMS DB for additional fraud metadata, update the two-phase commit logic, and ensure MQ responses still correlate correctly.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs `COPAUA0C`, `COPAU00C`, `COPAU01C`, `COPAUS2C` under `app/app-authorization-ims-db2-mq/cbl`.
  • IMS DBD/PSB definitions in `app/app-authorization-ims-db2-mq/ims` for HIDAM storage of authorization records.
  • DB2 DDL and control cards (`AUTHFRDS`, indexes) plus `CBPAUP0C` for nightly purges.
  • MQ connection definitions and queue names (`CARDDEMO.REQUEST.QUEUE`, `CARDDEMO.RESPONSE.QUEUE`) controlled via `csd/` resources.

🔗 Dependencies

  • IBM MQ queue manager reachable from CICS.
  • IMS HIDAM database for `AUTHDS` storage.
  • DB2 subsystem with the `AUTHFRDS` table and indexes.
  • VSAM datasets for account/customer lookups (reused from core).

🌐 Public Interfaces

CP00  - Authorization MQ listener / COPAUA0C
CPVS  - Pending authorization summary / COPAU00C
CPVD  - Authorization detail/edit / COPAU01C
MQ: CARDDEMO.REQUEST.QUEUE / CARDDEMO.RESPONSE.QUEUE (request/response correlation via IDs)
Batch job: CBPAUP0J purges expired authorizations

✅ Acceptance Criteria Examples

  • An MQ authorization request with a valid account number inserts a row into both IMS HIDAM and DB2 `AUTHFRDS`, then sends an approval response with `AUTH_RESP_CODE='00'`.
  • Flagging an authorization as fraud on `CPVD` toggles `AUTH_FRAUD` in the DB2 table and marks the record in IMS for analysts.
  • `CBPAUP0J` removes authorizations older than the configured retention period without leaving orphaned MQ responses.
