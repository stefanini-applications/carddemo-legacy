📡 Account Extraction via MQ

Storybook for MQ request/response patterns that expose VSAM-backed account data.

📋 Overview

Purpose: Demonstrate MQ-based account data release patterns using `CDRD` (date inquiry) and `CDRA` (account detail) transactions.

Business Context: MQ responses return 300-byte payloads backed by `CVACT01Y` layouts and allow external systems to integrate without direct VSAM access.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Date Inquiry: As an integration partner, I want `CDRD` to return the system date via MQ so that clients can reconcile timezones before sending requests.
  • Account Details: As a partner service, I want `CDRA` to fetch VSAM-backed account details for an account number so that legacy data can be lifted into a modern dashboard.
  • Correlated Response: As a platform architect, I want correlation IDs returned with MQ responses so that requesters can safely match asynchronous replies to their requests.

⚡ Development Acceleration Factors

  • Message Layouts: Date and account request/response copybooks mimic `CVACT01Y` structure—reuse them for new MQ payloads.
  • MQ definitions: Queue names in `app/app-vsam-mq/README.md` describe `CARDDEMO.REQUEST.QUEUE` + `CARDDEMO.RESPONSE.QUEUE` setups.
  • Transaction Patterns: `CODATE01` and `COACCT01` show how to pack/unpack MQ payloads and call VSAM records from BMS screens.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new field to `ACCT-RESPONSE-MSG` and expose it in the `CDRA` detail screen without touching MQ headers.
  • Medium (3-5 pts): Introduce a new MQ listener that merges MQ data with VsAM data before pushing to a JMS queue.
  • Complex (5-8 pts): Build a service that reuses the MQ request/response pattern to backfill a third-party API while reusing `CODATE01` message framing.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs: `CODATE01`, `COACCT01` handling MQ requests and VSAM reads.
  • MQ queues: `CARDDEMO.REQUEST.QUEUE`, `CARDDEMO.RESPONSE.QUEUE`, correlation ID usage for `CDRD`/`CDRA`.
  • Copybooks: `ACCT-REQUEST-MSG`, `ACCT-RESPONSE-MSG`, `DATE-REQUEST/RESPONSE-MSG` define payload structures.

🔗 Dependencies

  • core-online: Uses the same VSAM structures as account screens (`CVACT01Y`).
  • scripts: MQ definitions referenced from `app/app-vsam-mq/README.md` and `csd` definitions.
  • authorization: Shares MQ expertise for correlation logic.

🌐 Public Interfaces

CDRD - Sends DATE request to MQ, receives date response via CARDDEMO.RESPONSE.QUEUE
CDRA - Sends account number request, receives 300-byte account payload referencing CVACT01Y structure
MQ correlation IDs - ensures asynchronous replies match original requests

🧪 Acceptance Criteria & Patterns

  • Payload fidelity: Account responses must be exactly 300 bytes and align with `ACCT-RESPONSE-MSG` layout.
  • Messaging: Each request uses correlation IDs; scripts or job logs should capture RC codes to detect MQ errors.
  • Performance: MQ request/response should finish < 5 seconds P95.
