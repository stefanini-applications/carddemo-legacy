📡 MQ Integration Module

User Story Development Guide — Asynchronous MQ Request/Response Patterns  Optional Module

📋 Overview

Purpose: Demonstrates asynchronous request/response communication patterns using IBM MQ. External systems can query system date or account details through MQ channels without direct VSAM access, showcasing loosely-coupled integration patterns.

Business Context: This module is a reference implementation for MQ integration patterns commonly used in mainframe-to-distributed system integration scenarios. The two transactions (CDRD for date, CDRA for account) demonstrate how external consumers can request data from the mainframe asynchronously — a key pattern for service enablement and API extraction in modernization projects.

Prerequisites: IBM MQ configured and accessible from CICS; queues CARDDEMO.REQUEST.QUEUE and CARDDEMO.RESPONSE.QUEUE defined.

CICS Transactions: CDRD (system date), CDRA (account details)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Date Inquiry: As an external system, I want to query the mainframe system date via MQ so that I can synchronize timestamps without a direct CICS connection.
  • Account Inquiry: As an external system, I want to retrieve account details via MQ so that I can build distributed features without direct VSAM access.
  • Integration Test: As an integration developer, I want to test the MQ request/response pattern so that I can validate the asynchronous communication channel is working correctly.
  • Error Handling: As an external system, I want to receive structured error responses via MQ so that I can handle failure cases gracefully.

⚡ Development Acceleration Factors

  • Complete reference implementation: CODATE01 and COACCT01 demonstrate full MQ get/put patterns in COBOL
  • MQ queue definitions: CRDDEMOM.csd provides CICS resource definitions for MQ transactions
  • Account data pattern: COACCT01 uses the same VSAM CARDXREF + ACCTDATA lookup as online account-management
  • No screen required: MQ transactions are headless (no BMS maps) — simpler to implement and test

📊 Complexity Guidelines

  • 1–2 pts System date inquiry via MQ (CDRD — get message, format date, put response)
  • 3–5 pts Account details via MQ (CDRA — get message, VSAM lookup, format response, put)
  • 5–8 pts New MQ-exposed service for customer/transaction data; error queue handling; message correlation ID tracking; MQ security

🔄 MQ Request/Response Pattern

-- Date Inquiry (CDRD / CODATE01)
1. External system → MQPUT to CARDDEMO.REQUEST.QUEUE
   Message: { "type": "DATE_INQUIRY", "correlId": "XXXXX" }
2. CICS CDRD transaction triggered → CODATE01
3. CODATE01 → MQGET from request queue
4. Format current system date
5. CODATE01 → MQPUT to CARDDEMO.RESPONSE.QUEUE
   Response: { "date": "YYYY-MM-DD", "correlId": "XXXXX" }

-- Account Inquiry (CDRA / COACCT01)
1. External system → MQPUT to CARDDEMO.REQUEST.QUEUE
   Message: { "type": "ACCT_INQUIRY", "cardNum": "XXXXXXXXXXXXXXXX" }
2. CICS CDRA transaction triggered → COACCT01
3. COACCT01 → MQGET from request queue
4. READ CARDXREF by card number → get ACCT-ID
5. READ ACCTDATA by ACCT-ID → get account details
6. COACCT01 → MQPUT to CARDDEMO.RESPONSE.QUEUE
   Response: { "acctId": "...", "balance": "...", "limit": "..." }

🔗 Dependencies

  • IBM MQ: MQ queues must be defined before transactions can run
  • account-management: COACCT01 uses same VSAM CARDXREF and ACCTDATA files
  • authorization module (optional): Both use MQ patterns; authorization builds on these fundamentals

✅ Acceptance Criteria Patterns

  • Given a date inquiry message on request queue, when CDRD processes it, then response with current date is placed on response queue with matching correlId
  • Given an account inquiry with valid card number, when CDRA processes it, then account balance and credit limit are returned on response queue
  • Given an account inquiry with invalid card number, when CDRA processes it, then error response is placed on response queue with error code
  • Given MQ queue is empty, when CDRD/CDRA triggered, then transaction completes without abend
  • Given MQ is unavailable, when transaction attempts MQGET, then error is logged and CICS transaction ends gracefully
  • Given response message, then correlId in response matches correlId from request message
