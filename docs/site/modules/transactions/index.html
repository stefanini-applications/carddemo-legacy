💸 Transactions Module

CardDemo — User Story Development Guide

📋 Overview

Purpose: Manages credit card transaction records — online list/view/add and batch posting (POSTTRAN) and reporting (TRANREPT).

Business Context: Transactions are the financial events that drive account balance changes, interest calculation, and statement generation. The daily transaction file (DALYTRAN) is the batch input; TRANFILE is the master record store with AIX on date for reporting.

CICS Transactions: CT00 (list), CT01 (view), CT02 (add)

Batch Jobs: POSTTRAN (CBTRN02C), TRANREPT (CBTRN03C)

🎯 User Story Development Guide

📝 US Templates

  • As a cardholder, I want to see a list of my recent transactions so that I can review my spending
  • As a cardholder, I want to view the full details of a specific transaction (merchant, amount, date) so that I can verify charges
  • As a cardholder, I want to add a new transaction online so that I can record a purchase immediately
  • As a batch operator, I want POSTTRAN to process DALYTRAN daily so that all cardholder purchases are reflected in account balances
  • As a batch operator, I want rejected transactions written to DALYREJS so that invalid records can be reviewed and corrected
  • As a financial analyst, I want transaction category balances in TCATBALF so that spending by type/category can be tracked for interest

⚡ Development Acceleration Factors

  • CVTRA01Y–CVTRA07Y: Comprehensive transaction data structure copybooks
  • XREFFILE lookup pattern: Card-to-account resolution established in CBTRN02C
  • TCATBALF accumulation pattern: Type/category balance update logic in CBTRN02C
  • AIX on TRANFILE: Date-range queries for reporting already supported

📊 Complexity Guidelines

  • 1–2 pts Transaction List (CT00), Transaction View (CT01) — read from TRANFILE
  • 3–5 pts Transaction Add (CT02) — validate card/account, write to TRANFILE, update ACCTFILE
  • 5–8 pts POSTTRAN batch with full validation, category balance updates, rejection handling

📊 Transaction Category Balance Key

TRANCAT-ACCT-ID   PIC 9(11)  -- Account ID
TRANCAT-TYPE-CD   PIC X(02)  -- 2-char transaction type code
TRANCAT-CD        PIC 9(04)  -- 4-digit category code
TRAN-CAT-BAL      PIC S9(09)V99  -- Accumulated balance

📊 POSTTRAN Processing Logic (CBTRN02C)

FOR EACH record in DALYTRAN (sequential):
  1. Lookup CARD-NUM in XREFFILE → get ACCT-ID
  2. Read ACCTFILE by ACCT-ID → check ACCT-ACTIVE-STATUS
  3. Validate credit limit not exceeded
  4. IF valid:
       WRITE record to TRANFILE (VSAM KSDS)
       UPDATE ACCTFILE balance (ACCT-CURR-BAL, cycle accumulators)
       UPDATE TCATBALF category balance
  5. IF invalid:
       WRITE record to DALYREJS sequential

✅ Acceptance Criteria Examples

  • Given account with transactions → CT00 displays transaction list in date order
  • Given transaction ID selected → CT01 shows all transaction fields
  • Given valid card/account/amount on CT02 → Transaction written to TRANFILE, ACCTFILE balance updated
  • Given POSTTRAN runs with N records in DALYTRAN → Valid records in TRANFILE, invalid in DALYREJS, counts reconcile to N
  • Given transaction for inactive account → Record rejected to DALYREJS with appropriate rejection code
  • Given POSTTRAN → TCATBALF balances updated for each transaction type/category per account
