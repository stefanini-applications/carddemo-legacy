🗄️ Transaction Type DB2

User Story Development Guide — CardDemo Mainframe Application

Optional Module

Overview

Optional module for maintaining transaction type and category reference data in DB2 tables (TRNTYPE, TRNTYCAT). CICS programs support list/update/delete (CTLI / COTRTLIC) and add/edit (CTTU / COTRTUPC). Batch program MNTTRDB2 (COBTUPDT) handles bulk maintenance. The TRANEXTR job extracts DB2 data to VSAM for use by other programs. Accessed from the Admin Menu.

⚠️ Prerequisites

  • Base CardDemo application installed
  • DB2 tables created via CREADB21 JCL job
  • Admin user access (USER-TYPE=A)

Writing Good Stories for This Module

  • Distinguish between the DB2 master tables (TRNTYPE) and the VSAM working copies (TRANTYPE).
  • Add/update stories must specify DB2 constraint handling (duplicate type code = constraint violation).
  • Sync stories should specify that TRANEXTR must run after any DB2 change to keep VSAM in sync.
  • The list screen (COTRTLIC) uses a DB2 cursor — pagination stories should note cursor management.
  • This module demonstrates all four DB2 operations: SELECT (cursor), INSERT, UPDATE, DELETE.

Complexity Grid

🟢 Simple (1–2 pts)

  • List transaction types via DB2 cursor (CTLI read-only)
  • Add a new display field to the transaction type list
  • View transaction type detail

🟡 Medium (3–5 pts)

  • Add/update transaction type with DB2 insert/update and validation
  • Delete transaction type with referential integrity check

🔴 Complex (5–8 pts)

  • Batch extract synchronization with conflict detection
  • Transaction type with hierarchical category relationships

Business Rules

  • Source of truth: DB2 TRNTYPE and TRNTYCAT are the master tables; VSAM TRANTYPE and TRANCATG are working copies for batch jobs.
  • Synchronization: TRANEXTR must run after any DB2 changes to keep VSAM working copies current; batch jobs reading VSAM may use stale data otherwise.
  • Admin access: All CICS transactions (CTTU, CTLI) require admin user type (USER-TYPE=A).
  • DB2 cursor: COTRTLIC uses a DB2 cursor for paginated list display; cursor must be explicitly closed on program exit.
  • DB2 operations: This module demonstrates all four DB2 DML operations — SELECT (cursor), INSERT, UPDATE, and DELETE.
  • Constraint handling: Duplicate type codes are prevented by DB2 primary key constraint; COTRTUPC must handle SQLCODE -803 gracefully.
