📄 Bill Payment Module

User Story Development Guide — Credit Card Bill Payment Processing

📋 Overview

Purpose: Process credit card bill payments — the cardholder enters a payment amount, which is applied immediately to reduce the account's current balance in the VSAM account file.

Business Context: Bill payment is a critical financial operation that directly modifies account balances. Unlike transactions (which are batch-posted), bill payments are applied in real time via CICS. The account must be retrieved via card number cross-reference, then the balance updated.

CICS Transaction: CB00 → COBIL00C

VSAM Datasets: AWS.M2.CARDDEMO.ACCTDATA.PS (balance update), AWS.M2.CARDDEMO.CARDXREF.PS (card-to-account lookup)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Payment Entry: As a cardholder, I want to pay my credit card bill online so that my account balance is reduced immediately.
  • Payment Confirmation: As a cardholder, I want to review the payment amount before confirming so that I can catch entry errors before they are processed.
  • Balance Display: As a cardholder, I want to see my current balance before and after payment so that I can confirm the payment was applied correctly.
  • Minimum Payment: As a cardholder, I want the system to display my minimum payment due so that I know the minimum amount I must pay.

⚡ Development Acceleration Factors

  • CVACT01Y copybook: Account record with ACCT-CURR-BAL field — direct REWRITE after update
  • CVACT03Y copybook: CARDXREF lookup allows account retrieval from card number
  • COBIL00.bms: Screen layout already defined for payment input
  • Pattern established: Same VSAM REWRITE pattern used in account-management CAUP

📊 Complexity Guidelines

  • 1–2 pts Enter amount, validate > 0, read account, subtract from balance, REWRITE
  • 3–5 pts Add minimum payment display, payment history tracking, multiple payment methods
  • 5–8 pts EFT/ACH integration via CUST-EFT-ACCOUNT-ID, payment scheduling, payment reversal

🔗 Payment Flow

1. Cardholder enters card number + payment amount on CB00 screen
2. COBIL00C reads CARDXREF by CARD-NUM → get XREF-ACCT-ID
3. Read ACCTDATA by ACCT-ID → get current ACCT-CURR-BAL
4. Validate: amount > 0, account is active
5. New balance = ACCT-CURR-BAL - payment amount
6. REWRITE account record with new balance
7. Display confirmation with new balance

🔗 Dependencies

  • account-management: Shares ACCTDATA VSAM file; same REWRITE pattern
  • card-management: Uses CARDXREF for card-to-account lookup
  • signon: Authentication required; user context from COMMAREA

✅ Acceptance Criteria Patterns

  • Given valid card and positive payment amount, when submitted, then ACCT-CURR-BAL is reduced by payment amount
  • Given payment amount = 0, when submitted, then system rejects with "amount must be greater than zero" error
  • Given payment amount > current balance, when submitted, then system either rejects (overpayment) or creates credit balance (per business rule)
  • Given invalid card number, when submitted, then system displays "card not found" error
  • Given inactive account, when payment submitted, then system displays account status error
  • Given successful payment, then confirmation screen shows: amount paid, previous balance, new balance
