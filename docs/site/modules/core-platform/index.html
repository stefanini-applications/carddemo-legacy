core-platform

Module role: Base CardDemo runtime for sign-on, user/admin menu routing, account/card maintenance, transaction posting, bill payments, and batch lifecycle orchestration.

Scope boundary: This page documents only the core module in app/cbl, app/bms, app/cpy*, app/csd, app/jcl, app/proc, and app/scheduler.

Source-backed update generated from COBOL, BMS, CSD, JCL, PROC, and scheduler artifacts in the current repository checkout.

Module Overview and Purpose

The core-platform module implements the baseline credit-card processing lifecycle used by all optional extensions. User authentication starts in CC00/COSGN00C, then routes to either CM00/COMEN01C for cardholder operations or CA00/COADM01C for admin operations.

Business operations are split between online CICS transactions and nightly/monthly batch chains. Online transactions update VSAM in near-real-time, while batch jobs reconcile, post, summarize, and report data with deterministic scheduling dependencies.

Primary business outcomes

  • Authenticate users against USRSEC.
  • Maintain account/customer/card master data through CICS maps.
  • Add and browse card transactions in TRANSACT.
  • Accept and post bill payments from online flow.
  • Run batch posting, interest calculation, statement generation, and reporting.

Runtime profile

  • Online: CICS + BMS + VSAM KSDS.
  • Batch: COBOL + SORT + IDCAMS + scheduler chains.
  • Contracts: copybooks in app/cpy and app/cpy-bms.
  • Operations: CICS file close/open around batch windows.
  • Scheduling: CA7 and Control-M definitions in app/scheduler.

Architectural Context

Execution model

  1. Sign-on: COSGN00C validates user credentials from USRSEC.
  2. Menu dispatch: COMEN01C evaluates selected option and XCTLs to target program.
  3. Business transaction: target program performs map receive, validation, VSAM read/update/write, and map send.
  4. Nightly/monthly processing: batch jobs run with explicit dependencies and data handoffs.

Batch API summary

  • POSTTRAN: Executes CBTRN02C with DD allocations for DALYTRAN, TRANFILE, XREFFILE, ACCTFILE, TCATBALF, and DALYREJS.
  • INTCALC: Executes CBACT04C with parameter date and DDs for DISCGRP, TCATBALF, XREFFILE, ACCTFILE, output TRANSACT.
  • COMBTRAN: SORT merge of backup + system transactions, then IDCAMS REPRO to TRANSACT KSDS.
  • CREASTMT: SORT and repro staging plus statement generation outputs (STATEMNT.PS, STATEMNT.HTML).

Business rules from source

  • Credential checks: COSGN00C requires non-blank user/password and exact password match from USRSEC.
  • Menu permission checks: COMEN01C validates numeric option, option range, and admin-only constraints.
  • Installation-aware options: Option pointing to COPAUS0C is checked using EXEC CICS INQUIRE PROGRAM.
  • Transaction add constraints: COTRN02C enforces required fields, numeric keys/amounts, valid dates via CSUTLDTC, and explicit Y/N confirmation.
  • Bill payment rule: COBIL00C disallows payment when current balance is zero or below and requires confirmation before posting.
  • Account update consistency: COACTUPC locks account/customer records for update, checks whether records changed since fetch, rewrites both, and rolls back on partial failure.
  • Batch posting validation: CBTRN02C rejects transactions for invalid card number, missing account, over-limit balance, and expired account date.
  • Interest processing: CBACT04C looks up disclosure group rates, falls back to DEFAULT group, writes generated interest transactions, then updates account balances.
  • Report submission: CORPT00C validates date components and uses CSUTLDTC before writing JCL lines to TDQ.

Operator runbook sequence (core baseline)

  1. Load and define VSAM datasets.
  2. Install CICS resources from CARDDEMO.CSD.
  3. Execute CLOSEFIL before heavy batch updates.
  4. Run processing chain: POSTTRAN → INTCALC → COMBTRAN → CREASTMT → TRANIDX.
  5. Execute OPENFIL to reopen online resources.

Development Guidelines and Patterns

Online CICS coding pattern used in module

  1. Initialize working storage and commarea context flags.
  2. RECEIVE MAP for entered fields.
  3. Validate mandatory, type, date, and confirmation fields.
  4. Perform VSAM lookups using primary key or AIX path.
  5. For updates, lock with UPDATE and then REWRITE.
  6. Set message area and cursor for user feedback.
  7. SEND MAP and RETURN TRANSID.

Batch COBOL pattern used in module

  1. Open all required files and validate file status.
  2. Read sequential inputs in a controlled loop.
  3. Validate record using xref/account/reference data.
  4. Split valid and reject paths with explicit reason codes.
  5. Update dependent balances and write outputs.
  6. Close files and set non-zero return code when needed.

Design constraints

  • No filesystem i18n catalogs; text is in BMS maps.
  • No REST/HTTP endpoints in core module; public API is transaction/job interface.
  • Message area on map is canonical UX for warnings/errors.
  • Cross-record consistency uses copybook-aligned fixed-length fields.
  • Optional modules must not break baseline menu and batch paths.

Usage Examples and Code Snippets

Example walkthrough: cardholder payment

  1. User signs in through CC00.
  2. User selects bill payment option from CM00.
  3. CB00 receives account ID and reads ACCTDAT with UPDATE.
  4. If balance > 0 and confirmation is Y, program generates payment transaction.
  5. Program writes to TRANSACT and rewrites account with reduced balance.
