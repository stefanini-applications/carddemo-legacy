⚙️ Batch Processing Module

User Story Development Guide — JCL Batch Jobs, Posting, Interest & Statements

📋 Overview

Purpose: Nightly and scheduled batch jobs that handle the high-volume financial processing: daily transaction posting, interest calculation, statement generation, data file maintenance, and utility operations.

Business Context: Batch processing is the financial backbone of CardDemo. All daily transactions (from POS systems, online entries) are accumulated in DALYTRAN and posted nightly. Interest is calculated based on category balances and disclosure group rates. Statements are produced for all active accounts. The entire processing chain is scheduled via CA7 or Control-M.

Key Programs: CBTRN01C, CBTRN02C (posting), CBACT04C (interest), CBSTM03A/B (statements), CBTRN03C (reports)

Scheduler Files: app/scheduler/CardDemo.ca7, app/scheduler/CardDemo.controlm

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Transaction Posting: As a system operator, I want to run the nightly transaction posting job (POSTTRAN) so that all daily transactions are applied to account balances.
  • Interest Calculation: As a financial system, I want to calculate and apply interest charges (INTCALC) so that accounts accurately reflect accrued interest each cycle.
  • Statement Generation: As a system, I want to produce monthly statements (CREASTMT) so that cardholders receive accurate summaries of their account activity.
  • Rejection Handling: As a system operator, I want rejected transactions written to DALYREJS so that invalid records are identifiable and reprocessable.
  • Data Refresh: As a system operator, I want to refresh VSAM master files (ACCTFILE, CARDFILE) so that the most current data is available for online processing.

⚡ Development Acceleration Factors

  • Scheduler definitions: CA7 and Control-M job definitions already exist in app/scheduler/
  • JCL procedures: BUILDBAT.prc, BUILDONL.prc provide compilation procedures
  • CSUTLDTC.cbl: Date utility program available for date arithmetic in batch
  • COBSWAIT.cbl: Timer utility for job scheduling gaps (WAITSTEP)
  • All copybooks shared: CVACT01Y, CVTRA01Y–07Y used across online and batch programs

📊 Complexity Guidelines

  • 1–2 pts Data refresh JCL (ACCTFILE, CARDFILE — IDCAMS REPRO), file open/close (OPENFIL, CLOSEFIL)
  • 3–5 pts Add new validation in transaction posting, modify interest rate logic, add new report format
  • 5–8 pts New batch pipeline (end-to-end), GDG integration, DB2 batch processing, IMS batch purge

✅ Acceptance Criteria Patterns

  • Given DALYTRAN contains valid transactions, when POSTTRAN runs, then all valid records are written to TRANSACT.VSAM.KSDS and account balances are updated
  • Given a transaction with invalid card number, when POSTTRAN runs, then record is written to DALYREJS with rejection reason code
  • Given ACCTDATA is current, when INTCALC runs, then interest is calculated for each TCATBALF entry using matching DISCGRP rate
  • Given all transactions posted, when CREASTMT runs, then a statement record is produced for each active account
  • Given batch completes, when OPENFIL runs, then all VSAM files are available for CICS online transactions
  • Given POSTTRAN failure mid-run, then accounts partially updated should be identifiable via restart/recovery JCL
