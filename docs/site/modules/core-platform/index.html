Guide for Developing User Stories

📋 Overview

Purpose: Deliver the 3270-based credit card management experience inside CICS, backed by VSAM datasets and batch jobs for posting, billing, and reporting.

Business Context: US modernization programs use this module as the base layer that handles daily cardholder interactions, builds trust in data, and feeds optional integrations.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Visibility: As a cardholder, I want to view my current balances and recent transactions (COACTVW → COTRN00C) so that I can reconcile purchases before paying a bill.
  • Billing and Reporting: As an operations analyst, I want POSTTRAN and INTCALC to finish before statement generation so that users see a consistent daily snapshot.
  • User Management: As an admin, I want to add/update a user (`CU01`/`CU02`) so that access to core transactions is properly provisioned.

⚡ Development Acceleration Factors

  • Shared Copybooks: Copybooks like `CVACT01Y` and `CVTRA06Y` standardize VSAM fields, reducing regression risk when multiple programs read the same data.
  • Batch orchestration script: `scripts/run_full_batch.sh` sequentially submits the refresh/posting job chain with documented sleeps—reuse it for acceptance/regression runs.
  • Mapset conventions: Screen flows follow `CO*` naming (COSGN* for signon, COACT* for account), so reuse BMS templates with consistent field anchors.

📊 Complexity Guidelines

  • Simple (1-2 pts): Update a single BMS field or label that maps directly to an existing copybook field.
  • Medium (3-5 pts): Add a new VSAM field propagated to COBOL, BMS, and batch jobs (e.g., extend `DALYTRAN` and adjust `POSTTRAN`).
  • Complex (5-8 pts): Introduce a new dataset plus integration with optional modules (e.g., new VSAM files consumed by authorizations or MQ).

🔧 Technical Foundation

🏗️ Key Components

  • Signon/Navigation: `COSGN00C`/`CM00` manage authentication and the main menu.
  • Account/Card processing: `COACTVW`, `COCRDUPC`, `COCRDSLC` leverage `CVACT01Y`/`CVCRD01Y` layouts.
  • Transactions: `COTRN00C`/`COTRN02C` write `DALYTRAN` records via `CVTRA06Y` and call batch posting (`POSTTRAN`).
  • Billing & Reports: `COBIL00C`, `CBSTM03A`, and `TRANBKP` maintain reporting data.

🔗 Dependencies

  • VSAM datasets (`AWS.M2.CARDDEMO.ACCTDATA.PS`, `CARDDATA`, `DALYTRAN.PS`, etc.).
  • JCL jobs (`POSTTRAN`, `CARDFILE`, `TRANIDX`) and `scripts/run_full_batch.sh` for automation.
  • Copybooks (`CVACT01Y`, `CVTRA06Y`, `CSMSG01Y`) and BMS definitions in `app/bms/`.
  • Admin menu options (CA00) enable optional module activation.

🌐 Public Interfaces

CC00 (Signon) - `COSGN00C`
CT00/CT02 - Transaction list/add (`COTRN00C`, `COTRN02C`)
CB00 - Bill payment (`COBIL00C`)
POSTTRAN / INTCALC / TRANIDX - Batch posting chain (JCL)
scripts/run_full_batch.sh - Submits CLOSEFIL → POSTTRAN → INTCALC → TRANIDX → OPENFIL

📌 Acceptance Criteria Examples

  • Role-based menus visible only after successful `CC00` signon; unauthorized PF keys disabled.
  • `DALYTRAN` updates must produce matching backup files (`TRANBKP`) and catalog updates (`TRANIDX`).
  • Batch automation script logs job names and waits indicated seconds to keep JES throughput stable.
