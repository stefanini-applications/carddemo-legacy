📦 Transaction Type Management (Db2)

Guide for stories that keep the transaction reference catalog governed by Db2 and synchronized with VSAM.

📋 Overview

Purpose: Provide an administrative experience for managing transaction types using Db2 tables, embedded static SQL, and batch jobs that synchronize VSAM references.

Business Context: This feature adds Db2 robustness for metadata, making the transaction catalogue easier to audit than flat VSAM files while still feeding the high-performance core pipeline via extraction jobs.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Admin CRUD: As an admin, I want CTTU to let me add a capture code with a description so Hudson reporting shows the new option immediately.
  • Batch sync: As an operator, I want TRANEXTR to refresh VSAM transaction-type files after a Db2 INSERT so live transactions see the current catalog.
  • Data hygiene: As a DBA, I want CTLI to provide backward/forward cursor patterns that help me detect stale rows before deleting them.

⚡ Development Acceleration Factors

  • Db2 `ddl/` scripts outline `TRANSACTION_TYPE` and `TRANSACTION_TYPE_CATEGORY` structures, which can be mirrored when adding columns.
  • `dcl/` declarations supply SQLCA and host variable libraries so new Db2 programs reuse the same error handling.
  • CAM: BMS templates for `COTRTUP/COTRTLI` expedite new admin screens with consistent row layouts.

📊 Story Complexity Guidelines

  • Simple (1-2 pts): Display a new column in CTLI's list view while keeping existing Db2 SQL unchanged.
  • Medium (3-5 pts): Extend TRANSACTION_TYPE with a new attribute, ensure CTTU writes it, and update TRANEXTR outputs.
  • Complex (5-8 pts): Introduce a supporting Db2 category table and cascade deletes through VSAM via an updated MNTTRDB2 job.

✅ Acceptance Criteria Examples

  • CTTU/CTLI commit Db2 transactions only when SQLCA indicates success and display BMS messages on rollback.
  • TRANEXTR produces VSAM files with the same number of rows as the Db2 table and sorts them before CICS consumption.
  • MNTTRDB2 job updates both `TRANSTIT` and derived VSAM metadata while respecting existing locks.

🔧 Technical Foundation

🏗️ Key Components

  • CICS programs `COTRTUPC`, `COTRTLIC` with embedded static SQL/SQLCA for Db2 CRUD operations.
  • `db2/ddl` definitions for `TRANSACTION_TYPE` (code, description) and the `TRANSACTION_TYPE_CATEGORY` join table.
  • `ctrans` copybooks for host variable declarations and `dcl/SQLCA` standard declarations.
  • Batch jobs `CREADB21`, `TRANEXTR`, `MNTTRDB2` that create tables, extract rows, and maintain VSAM reference files.

🔗 Dependencies

  • Db2 subsystem availability for the `Db2: Transaction Type Mgmt` feature.
  • CICS resource definitions in `app/app-transaction-type-db2/csd/` for transactions `CTTU` and `CTLI`.
  • VSAM extraction utilities that consume output files from TRANEXTR to keep the online catalog current.

🌐 Public Interfaces

CTTU  - Transaction Type add/edit (COTRTUPC)
CTLI  - Transaction Type list/update/delete (COTRTLIC)
CREADB21 - Db2 schema creation (batch)
TRANEXTR - Extract Db2 rows to VSAM (/DSNTIAUL)
MNTTRDB2 - Batch maintenance job (COBTUPDT)
