📦 Authorization IMS & Db2 & MQ Bridge

Guide for enabling pending authorization workflows, MQ triggers, and Db2 fraud analytics.

📋 Overview

Purpose: Extend CardDemo with MQ-driven authorization processing, IMS persistence for pending holds, and Db2 fraud tracking (AUTHFRDS table).

Business Context: Optional feature used in modernization demos where cloud clients push authorization requests to MQ, and the mainframe stores decisions across IMS and Db2 before releasing credit holds.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Fraud recognition: As a fraud analyst, I want CPVD to flag suspect authorizations and persist them in AUTHFRDS so that compliance can trigger downstream reviews.
  • MQ integration: As an MQ integrator, I want the `AWS.M2.CARDDEMO.PAUTH.REQUEST` format to remain stable so my simulator can replay authorization swipes.
  • Housekeeping: As an operator, I want CBPAUP0J to purge expired holds nightly so account balances stay accurate.

⚡ Development Acceleration Factors

  • IMS artifacts: `ims/` directory already stores DBDs and PSBs needed for the HIDAM schema (PSBPAUTB, PSBPAUTL).
  • Db2 schema templates: `db2/schema` docs include AUTHFRDS + index definitions so new fraud columns inherit the same DDL style.
  • MQ queue templates: Documented queue names and comma-separated payload fields allow quick binding of new request types.

📊 Story Complexity Guidelines

  • Simple (1–2 pts): Surface a new BMS message on CPVS and log its value to a new IMS detail segment.
  • Medium (3–5 pts): Add a Db2 column to AUTHFRDS and ensure CPVD writes it while respecting SQLCA error handling.
  • Complex (5–8 pts): Introduce a two-phase commit between IMS (for pending auth) and Db2 (fraud flags) plus MQ retries for the CBPAUP0J purge job.

✅ Acceptance Criteria Examples

  • CPVS/CPVD must read IMS segments, update statuses, and insert a row into AUTHFRDS with the fraud flag when PF5 is pressed.
  • MQ authorization responses honor correlation IDs and indicate the same `TRANSACTION-ID` as the incoming request.
  • CBPAUP0J release job completes within four minutes and resets the available credit stored in VSAM `CARDDATA`.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs such as `COPAU00C`, `COPAUS0C`, `COPAU01C` orchestrate MQ-triggered authorizations and the CPVS/CPVD screens.
  • IMS DBD/PSB definitions for HIDAM (`DBPAUTP0`, `DBPAUTX0`, `PSBPAUTB`, `PSBPAUTL`).
  • Db2 `AUTHFRDS` table plus index `XAUTHFRD` holding fraud outcomes.
  • MQ queue definitions `AWS.M2.CARDDEMO.PAUTH.REQUEST/REPLY` plus CICS MQCONN entries.

🔗 Dependencies

  • CICS resource definitions in `app/app-authorization-ims-db2-mq/csd/` (programs, mapsets, transactions).
  • Shared `COPY` statements referencing `app/cpy/` record layouts for accounts and authorizations.
  • Batch job `CBPAUP0J` within `app/app-authorization-ims-db2-mq/jcl/` to keep holds current.

🌐 Public Interfaces

MQ Queue   - AWS.M2.CARDDEMO.PAUTH.REQUEST / AWS.M2.CARDDEMO.PAUTH.REPLY
CICS Trans - CPVS (summary) / CPVD (details) / CP00 (authorization request handler)
Batch Job  - CBPAUP0J (daily purge and credit reset)
Db2 Table  - AUTHFRDS (fraud log) with SQLCA checks and index XAUTHFRD

Payloads use comma-separated lists; see the main system overview for the exact field order.
