🔐 Authentication Module

CardDemo — User Story Development Guide

📋 Overview

Purpose: Validates user credentials and establishes an authenticated session for all CardDemo users via the CICS signon screen.

Business Context: All CardDemo functionality is gated by this module. User type (Admin vs Regular) determines which menu and features are accessible after login.

Entry Transaction: CC00 — the application entry point for all users

🎯 User Story Development Guide

📝 US Templates

  • Signon: As a cardholder, I want to sign in with my User ID and password so that I can access my account securely
  • Role Routing: As an admin user, I want to be routed to the Admin Menu after signon so that I can manage users and system data
  • Invalid Login: As a security auditor, I want invalid login attempts to be rejected with an error message so that unauthorized access is prevented
  • Session State: As a developer, I want authenticated user data passed via COMMAREA so that downstream programs know who is logged in

⚡ Development Acceleration Factors

  • COCOM01Y copybook: Shared COMMAREA structure — reuse across all online programs
  • CSUSR01Y copybook: User record structure pre-defined — VSAM USRSEC access pattern established
  • BMS map COSGN00: Screen layout already defined — no UI design work required
  • DUSRSECJ JCL: User file initialization pattern available

📊 Complexity Guidelines

  • 1–2 pts Display signon screen, navigate to authenticated menu
  • 3–5 pts Credential validation logic, user-type routing, error handling
  • 5–8 pts Security integration (RACF + application-level auth), session management changes

🔗 Dependencies

  • VSAM USRSEC: Must be loaded (DUSRSECJ) before application can authenticate users
  • CICS region: Must have COSGN00C, COMEN01C, COADM01C defined in CSD (CARDDEMO.CSD)
  • RACF: System-level security layer; complements application-level auth

💾 Key COMMAREA Fields Set by Authentication

CDEMO-USER-ID      PIC X(08)  -- authenticated user ID
CDEMO-USER-TYPE    PIC X(01)  -- 'A' = Admin, 'U' = Regular User
CDEMO-FROM-TRANID  PIC X(04)  -- 'CC00' on entry
CDEMO-TO-TRANID    PIC X(04)  -- 'CM00' (user) or 'CA00' (admin)
CDEMO-PGM-CONTEXT  PIC 9(01)  -- 0 = first entry, 1 = re-entry

✅ Acceptance Criteria Examples

  • Given valid User ID (USER0001) and Password (PASSWORD) → User is authenticated, COMMAREA populated, routed to Main Menu (CM00)
  • Given valid Admin ID (ADMIN001) and Password → Admin authenticated, routed to Admin Menu (CA00)
  • Given incorrect password → Error message displayed on COSGN00 screen, no COMMAREA populated, no menu transition
  • Given blank User ID or Password → Error message displayed, transaction stays on signon screen
  • Given non-existent User ID → VSAM record-not-found handled gracefully, error message shown
