📦 Core

Guidelines for delivering the baseline credit-card lifecycle functionality.

📋 Overview

Purpose: Surface accounts, cards, transactions, bill payments, statements, and user management through CICS/BMS backed by VSAM.

Business Context: This is the experience regulators, analysts, and cardholders see in demos: a terminal-driven workflow from signon (CC00) through menu navigation (CM00) into detail screens. Batch jobs keep the VSAM datasets ready for next-day processing.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account view: As a cardholder, I want to see my account balance, credit limits, and cycle dates so that I can confirm there are no unexpected holds before paying.
  • Transactions: As an operations analyst, I want to list recent transactions (CT00) with search filters so that I can investigate disputes without leaving CICS.
  • User management: As an admin, I want to add a user through the CU01 screen so they can access the simulator quickly.

⚡ Development Acceleration Factors

  • Reusable copybooks: Records like `CUSTOMER-RECORD` and `ACCOUNT-RECORD` under `app/cpy/` keep fields consistent when new screens or batch jobs read VSAM files.
  • BMS map templates: Screens such as `COACTVW.bms` and `COTRN00.bms` use the same layout helper fields (e.g., `SEL000n` selection indicators) so copy-pasting for new listings takes minutes.
  • Batch shells: Existing JCL jobs can be cloned (e.g., copy `CREASTMT.jcl`) to add new dataset loads or reporting runs with minimal extra steps.

📊 Complexity Guidelines

  • Simple (1-2 pts): Tweak a BMS label, add a new field to an existing copybook, or adjust a Validation clause in COBOL.
  • Medium (3-5 pts): Extend a CICS screen that composes multiple copybooks, add a new list column, or insert a small batch step into the morning job chain.
  • Complex (5-8 pts): Add a new VSAM dataset, align its BMS representation, and update the nightly JCL flow plus any derived statements.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs under `app/cbl/` (e.g., `COSGN00C`, `COACTUPC`, `COTRN00C`, `CBSTM03A`) orchestrating screen interaction and VSAM access.
  • Copybooks (`app/cpy/`) defining customer, account, card, transaction, and statement records.
  • BMS map sets (`app/bms/`) that define input fields, validation keys, and listings (e.g., `COTRN00` for transaction lists).
  • JCL procedures under `app/jcl/` for initial dataset loads (`ACCTFILE`), batch posting (`POSTTRAN`), statements (`CREASTMT`), and reports.

🔗 Dependencies

  • CICS region with RACF user-ID support (ADMIN001, USER0001).
  • VSAM KSDS/ESDS datasets for `ACCTDATA`, `CARDDATA`, `DALYTRAN`, and `USRSEC`.
  • Assembler macros (MVSWAIT, etc.) from `app/asm/` used by job control.
  • Scripts such as `scripts/local_compile.sh` to compile and upload programs.

🌐 Public Interfaces

CC00  - Signon / COSGN00C
CM00  - Main menu / COMEN01C
CAVW  - Account View / COACTVWC
CAUP  - Account Update / COACTUPC
CT00  - Transaction List / COTRN00C
CT01  - Transaction View / COTRN01C
CT02  - Transaction Add / COTRN02C
CB00  - Bill Payment / COBIL00C
CU00-CU03 - User CRUD
CR00  - Transaction Reports / CORPT00C
Batch jobs: ACCTFILE, CARDFILE, TRANBKP, POSTTRAN, CREASTMT, TRANREPT, INTCALC

✅ Acceptance Criteria Examples

  • When a valid account number is entered on `CAVW`, the screen shows the expected credit limit, balances, and cycle stats within 2 s.
  • Searching `CT00` for a transaction ID returns the matching row plus full date/amount details and allows the user to press ENTER to open `CT01`.
  • `POSTTRAN` nightly job completes without errors and produces a refreshed `DALYTRAN` file used by `CREASTMT`.
