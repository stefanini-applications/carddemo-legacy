🃏 Card Management Module

User Story Development Guide — Credit Card List, View & Update

📋 Overview

Purpose: Manage credit cards associated with a cardholder's account: list all cards, view individual card details (number, CVV, expiry, status), and update card information.

Business Context: Cards are the physical instruments of account access. The card management module provides visibility into what cards exist on an account and enables lifecycle operations (issuance, activation, deactivation). The CARDXREF dataset is the critical cross-reference linking card number → customer → account.

CICS Transactions: CCLI (Card List), CCDL (Card View/Select), CCUP (Card Update)

VSAM Datasets: AWS.M2.CARDDEMO.CARDDATA.PS, AWS.M2.CARDDEMO.CARDXREF.PS

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Card List: As a cardholder, I want to see all cards linked to my account so that I have a complete overview of my card portfolio.
  • Card View: As a cardholder, I want to view my card details (number, expiry, embossed name) so that I can verify the card is valid for use.
  • Card Deactivation: As a cardholder, I want to deactivate a lost or stolen card so that unauthorized charges are prevented immediately.
  • Card Update: As a cardholder, I want to update the embossed name on my card so that it reflects a legal name change.
  • XREF Integrity: As a system, I want the card-to-account cross-reference to be maintained so that transaction routing is always accurate.

⚡ Development Acceleration Factors

  • CVACT02Y copybook: Pre-defined 150-byte card record — 16-digit card number as key
  • CVACT03Y copybook: Pre-defined 50-byte XREF record — links card → customer → account
  • Dual-key VSAM on CARDXREF: Supports lookup by card number OR account ID (AIX)
  • Existing BMS maps: COCRDLI.bms, COCRDSL.bms, COCRDUP.bms fully define all three screens

📊 Complexity Guidelines

  • 1–2 pts View card list for account, view single card detail
  • 3–5 pts Update card status (activate/deactivate), update embossed name; require XREF consistency check
  • 5–8 pts New card issuance (create CARDDATA + CARDXREF records), card replacement workflow, fraud block integration

🗄️ Key Data Structures

-- CARD-RECORD (CVACT02Y, 150 bytes)
01  CARD-RECORD.
    05  CARD-NUM            PIC X(16).  -- Primary key
    05  CARD-ACCT-ID        PIC 9(11).  -- Linked account
    05  CARD-CVV-CD         PIC 9(03).  -- CVV security code
    05  CARD-EMBOSSED-NAME  PIC X(50).  -- Name on card
    05  CARD-EXPIRAION-DATE PIC X(10).  -- YYYY-MM-DD
    05  CARD-ACTIVE-STATUS  PIC X(01).  -- 'Y'=active, 'N'=inactive

-- CARD-XREF-RECORD (CVACT03Y, 50 bytes)
01 CARD-XREF-RECORD.
    05  XREF-CARD-NUM  PIC X(16).  -- Card number (primary key)
    05  XREF-CUST-ID   PIC 9(09).  -- Customer ID
    05  XREF-ACCT-ID   PIC 9(11).  -- Account ID (AIX key)

🔗 Dependencies

  • account-management: CARD-ACCT-ID must reference a valid account in ACCTDATA
  • transaction-processing: TRAN-CARD-NUM resolved to account via CARDXREF during posting
  • signon: Account context required from COMMAREA
  • VSAM CARDDATA and CARDXREF: Both must be open in CICS (JCL: OPENFIL)

✅ Acceptance Criteria Patterns

  • Given a valid account ID, when accessing CCLI, then all cards linked via CARDXREF are listed
  • Given no cards on account, when accessing CCLI, then system displays "no cards found" message
  • Given card status = 'Y', when user deactivates via CCUP, then status is set to 'N' and VSAM record is rewritten
  • Given card status = 'N' (inactive), when user views card, then system displays deactivation indicator
  • Given expiry date update, when the date is in the past, then system rejects with validation error
  • Given a VSAM write error during update, then system displays error code and card is not changed
