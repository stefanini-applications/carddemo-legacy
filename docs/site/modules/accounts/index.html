💳 Accounts

User Story Development Guide — CardDemo Application

📋 Overview

Purpose: Allow users to view and update credit card account information including current balance, credit limits, account status, and expiration dates.

Business Context: Accounts are the core entity in CardDemo. Every card, transaction, and bill payment is linked to an account. Account data drives credit decisions, interest calculations, and statement generation.

COACTVWC COACTUPC VSAM: ACCTDAT VSAM: CUSTDAT

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account View: As a cardholder, I want to view my account details so that I can see my current balance, credit limit, and account status.
  • Account Update: As an admin, I want to update an account's credit limit so that I can respond to customer credit requests.
  • Account Status: As an admin, I want to activate or deactivate an account so that I can respond to fraud alerts or customer requests.
  • Available Credit: As a cardholder, I want to see my available credit so that I know how much more I can spend.

⚡ Development Acceleration Factors

  • CVACT01Y.cpy: Account record structure reused by both CICS and batch programs — no duplication needed.
  • COACTVWC: Account view pattern established — lookup by ACCT-ID with CARDXREF navigation.
  • Available Credit Calculation: ACCT-CREDIT-LIMIT - ACCT-CURR-BAL is the standard formula used throughout.

🔧 Technical Foundation

🏗️ Key Components

  • COACTVWC.cbl: Account view — reads ACCTDAT by ACCT-ID; also retrieves linked customer data from CUSTDAT.
  • COACTUPC.cbl: Account update — validates input fields; writes updated record back to ACCTDAT.
  • CVACT01Y.cpy: Account record structure (300 bytes, KSDS key = ACCT-ID 11 digits).
  • CVCUS01Y.cpy: Customer record structure (500 bytes, KSDS key = CUST-ID 9 digits).

🔗 Dependencies

  • ACCTDAT VSAM: Account master file — must be populated.
  • CUSTDAT VSAM: Customer master — linked via CUST-ID.
  • CARDXREF VSAM: Card-to-account cross-reference — enables lookup by card number.
  • Authentication module: User must be authenticated; user type may restrict update operations.

📊 Account Record Key Fields

ACCT-ID (9(11))           - Primary key
ACCT-ACTIVE-STATUS (X(1)) - 'Y'=Active, 'N'=Inactive
ACCT-CURR-BAL             - Current balance (S9(10)V99)
ACCT-CREDIT-LIMIT         - Maximum credit (S9(10)V99)
ACCT-CASH-CREDIT-LIMIT    - Cash advance limit (S9(10)V99)
ACCT-OPEN-DATE (X(10))    - YYYY-MM-DD
ACCT-EXPIRAION-DATE (X(10))
ACCT-GROUP-ID (X(10))     - Used for interest rate lookup (DISCGRP)

✅ Acceptance Criteria Patterns

Account View:
  • Given a valid account ID exists in ACCTDAT
  • When user navigates to account view
  • Then all account fields are displayed (ID, status, balance, limits, dates)
  • And linked customer name is shown
Account Not Found:
  • Given an account ID that does not exist
  • When user requests account view
  • Then error message "Account not found" is displayed
Account Update Validation:
  • Given user is on account update screen
  • When credit limit entered is less than current balance
  • Then validation error displayed; record NOT saved
