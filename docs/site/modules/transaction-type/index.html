🧭 Transaction Type Management

User story playbook for Db2-driven transaction type CRUD plus VSAM exports.

📋 Overview

Purpose: Provide administrators with Db2 CRUD (via `CTTU`/`CTLI`) plus nightly exports (`TRANEXTR`) that feed legacy processing.

Business Context: Db2 tables `TRANSACTION_TYPE` and `TRANSACTION_TYPE_CATEGORY` replace static VSAM files for metadata; exports keep legacy screens working while enabling relational tooling.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Admin CRUD: As an admin, I want `CTTU` to insert or edit transaction descriptions so that `TRANEXTR` exports include the latest metadata for `COACTVWC`.
  • Data Stewardship: As a data steward, I want `CTLI` to enforce `DELETE RESTRICT` constraints so that dependent transaction types cannot disappear unexpectedly.
  • Migration Safety: As a migration specialist, I want `TRANEXTR` to output VSAM-friendly files after `MNTTRDB2` runs so that COBOL jobs read consistent datasets.

⚡ Development Acceleration Factors

  • DDL & CTL: Reuse DDL and CTL scripts in `app/app-transaction-type-db2/ddl` and `ctl` when deploying new tables.
  • SQL Patterns: Follow existing embedded SQL + SQLCA handling (seen in `COTRTUPC` and `COTRTLIC`).
  • Export Jobs: `TRANEXTR`, `TRANIDX`, `MNTTRDB2` show how to convert Db2 data to VSAM copybooks.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new descriptive field to `TRANSACTION_TYPE` and surface it in `CTLI`.
  • Medium (3-5 pts): Create a new category table with a referential link; ensure `TRANEXTR` exports handle the structure.
  • Complex (5-8 pts): Introduce a new Db2 cursor-based report that powers a VSAM list page with `forward/backward` paging.

🔧 Technical Foundation

🏗️ Key Components

  • CICS screens: `CTTU` (add/edit) and `CTLI` (list/update/delete), with SQL statements embedded in `COTRTUPC`/`COTRTLIC`.
  • Db2 tables: `CARDDEMO.TRANSACTION_TYPE`, `TRANSACTION_TYPE_CATEGORY`, with SQLCA handling.
  • Batch: `TRANEXTR`, `MNTTRDB2`, `CREADB21`, `TRANEXTR` uses static SQL via `COBTUPDT`.

🔗 Dependencies

  • core-online: Reads exported metadata via `COACTVWC` and `COCRDUP`.
  • core-batch: Depends on `TRANEXTR` sequencing to rebuild indexes.
  • scripts: `remote_compile.sh` shows how to precompile Db2 programs.

🌐 Public Interfaces

CTTU / COTRTUPC - Insert or update transaction types with SQLCA-driven validation
CTLI / COTRTLIC - List, page, and delete transaction types using Db2 cursors
TRANEXTR - Export Db2 data into VSAM copybook layout for legacy flows
MNTTRDB2 - Batch maintenance via COBTUPDT to update Db2 tables
CREADB21 - Creates Db2 schema and tables

🧪 Acceptance Criteria & Patterns

  • Transactional Safety: Each embedded SQL must check `SQLCODE` in SQLCA and rollback/abort on violations.
  • Cursor Paging: `CTLI` uses forward/backward cursor keywords; ensure new screens reuse the same PF key layout for navigation.
  • Export Integrity: `TRANEXTR` outputs must match the layout expected by `COACTVWC` for downstream screens.
