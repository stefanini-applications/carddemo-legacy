🧾 Bill Payment

User Story Development Guide · CardDemo Mainframe Application

Core Module CB00 · COBIL00C

📋 Overview

The Bill Payment module provides a single CICS transaction for submitting credit card bill payments online. It reads the current account balance, accepts a payment amount from the cardholder, creates a debit transaction record, and atomically updates the account balance — all within a single CICS unit of work.

CICS Transaction

Transaction IDProgramFunction
CB00COBIL00CSubmit bill payment — validate amount, update account, create transaction record

Data Stores

ACCTDATA VSAM KSDS — account record updated to decrement ACCT-CURR-BAL and increment ACCT-CURR-CYC-CREDIT after a successful payment.

TRANSACT VSAM KSDS — a payment transaction record is written using the CVTRA05Y layout with a payment-type transaction code, creating an auditable record of the payment event.

Business Context

Bill payment is the primary mechanism for cardholders to reduce their outstanding balance. The payment flow is tightly coupled: the account balance read, validation, transaction record insert, and account balance update must all succeed or all roll back — the program uses CICS syncpoint logic to ensure atomicity across both VSAM files.

Business Rules

  • Payment amount must be greater than zero
  • Payment amount cannot exceed the current account balance (ACCT-CURR-BAL)
  • Account must be in active status
  • Both the account update and transaction record insert must commit atomically
  • ACCT-CURR-CYC-CREDIT is incremented by the payment amount to track cycle-to-date credits

Story Templates

Make Payment for Specific Amount
As a cardholder, I want to submit a payment against my credit card balance so that I can reduce my outstanding debt.

Program: COBIL00C (CB00) — reads ACCTDATA, validates amount, writes payment transaction to TRANSACT VSAM, updates ACCTDATA balance
View Current Balance Before Payment
As a cardholder, I want to see my current balance displayed on the payment screen so that I know the maximum amount I can pay.

COBIL00C displays ACCT-CURR-BAL from ACCTDATA VSAM on the COBIL00.bms payment screen before the user enters an amount
Receive Payment Confirmation
As a cardholder, I want to receive a confirmation message after submitting a payment so that I know the payment was accepted.

COBIL00C displays confirmation with new balance after successful VSAM updates; TRAN-ID of the payment record shown
Payment Appears in Transaction List
As a cardholder, I want my payment to appear in my transaction history immediately after submission so that I have a complete audit trail.

Payment creates CVTRA05Y record in TRANSACT VSAM — visible via CT00 (COTRN00C) immediately after commit

Acceleration Factors

  • COBIL00C complete implementation — already implements the full account read + transaction write + balance update pattern; new stories can extend this directly
  • COBIL00.bms screen map exists — payment screen layout ready; field additions are incremental BMS changes
  • CVACT01Y + CVTRA05Y reused — no new copybooks needed for standard payment stories
  • Atomic update pattern — CICS syncpoint handling in COBIL00C is a template for any story requiring multi-VSAM atomicity

Payment Processing Flow

CB00 (COBIL00C) Payment Flow:
1. Read ACCTDATA VSAM (keyed on ACCT-ID) → display ACCT-CURR-BAL
2. Accept PAYMENT-AMT from user input
3. Validate: PAYMENT-AMT > 0 AND PAYMENT-AMT <= ACCT-CURR-BAL
4. Read ACCTDATA for update (exclusive lock)
5. Build CVTRA05Y payment record with:
      TRAN-TYPE-CD = payment type code
      TRAN-AMT     = PAYMENT-AMT (negative debit)
      TRAN-ORIG-TS = current timestamp
6. Insert TRANSACT VSAM record
7. Decrement ACCT-CURR-BAL by PAYMENT-AMT
8. Increment ACCT-CURR-CYC-CREDIT by PAYMENT-AMT
9. Rewrite ACCTDATA VSAM record
10. CICS SYNCPOINT (commit both updates atomically)
11. Display confirmation with new balance and TRAN-ID

✅ Acceptance Criteria Patterns

Payment Validation

  • GIVEN a payment amount of zero or negative, WHEN CB00 validates the input, THEN the payment is rejected with an error message "Payment amount must be greater than zero"
  • GIVEN a payment amount exceeding the current balance, WHEN CB00 validates the input, THEN the payment is rejected with an error showing the maximum allowable amount
  • GIVEN an inactive account, WHEN CB00 reads the account, THEN the payment screen displays an error and no payment is processed

Payment Processing

  • GIVEN a valid payment amount, WHEN CB00 processes the payment, THEN ACCT-CURR-BAL is decremented by the payment amount and ACCT-CURR-CYC-CREDIT is incremented by the same amount
  • GIVEN a successful payment, WHEN CB00 completes, THEN a payment transaction record exists in TRANSACT VSAM with the correct amount and timestamp
  • GIVEN a VSAM I/O error during commit, WHEN CB00 detects the error, THEN the entire unit of work is rolled back and an error message is displayed

Confirmation Display

  • GIVEN a successful payment, WHEN the confirmation screen is displayed, THEN the new account balance and the payment transaction ID are shown clearly
  • GIVEN a completed payment, WHEN the cardholder navigates to CT00, THEN the payment appears in the transaction list with correct amount and type code
