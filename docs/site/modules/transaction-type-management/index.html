🗂️ Transaction Type Management Module

User Story Development Guide — DB2 Reference Data Maintenance  Optional Module

📋 Overview

Purpose: Admin maintenance of transaction type reference data in DB2. Demonstrates embedded static SQL, forward/backward cursor processing, and full CRUD operations against a relational database from a CICS environment.

Business Context: Transaction types (e.g., "PR" = Purchase, "CR" = Credit) are reference data used throughout the application to categorize and classify transactions. Maintaining this data in DB2 (rather than static VSAM files) demonstrates the relational integration patterns common in modernized mainframe applications. A batch extract (TRANEXTR) synchronizes DB2 data to VSAM for use in existing batch jobs.

Prerequisites: DB2 available; CREADB21 JCL executed to create and load tables.

CICS Transactions: CTTU (add/edit), CTLI (list/update/delete)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Add Type: As an admin, I want to add a new transaction type in DB2 so that new transaction categories can be processed by the system.
  • Update Type: As an admin, I want to update the description of a transaction type so that the reference data accurately reflects current business terms.
  • Delete Type: As an admin, I want to delete obsolete transaction types so that the reference table stays clean and relevant.
  • List Types: As an admin, I want to page through the transaction type list so that I can review all defined types.
  • VSAM Sync: As a system operator, I want to run TRANEXTR to extract DB2 types to VSAM so that batch processing uses the latest reference data.

⚡ Development Acceleration Factors

  • DB2 patterns established: COTRTUPC and COTRTLIC demonstrate host variables, cursors, SQLCA error handling — copy and adapt
  • Existing DDL: TRNTYPE.ddl and TRNTYCAT.ddl define the complete schema
  • Existing BMS maps: COTRTUP.bms and COTRTLI.bms provide screen definitions
  • DB2 SQLCA pattern: Standard SQLCODE/SQLSTATE error handling used throughout — established convention

📊 Complexity Guidelines

  • 1–2 pts List transaction types (DB2 forward cursor, display CTLI screen)
  • 3–5 pts Add/update transaction type (CTTU with host variable SQL INSERT/UPDATE + SQLCA handling)
  • 5–8 pts Delete with referential integrity checks, backward cursor navigation, add TRANCAT entries for new type, TRANEXTR sync

🗄️ DB2 Table Structure

-- TRNTYPE table (equivalent to CVTRA03Y / TRANTYPE.PS)
TRAN-TYPE      CHAR(2)    NOT NULL  -- Primary key (e.g. 'PR', 'CR')
TRAN-TYPE-DESC VARCHAR(50)          -- Description

-- TRNTYCAT table (equivalent to CVTRA04Y / TRANCATG.PS)
TRAN-TYPE-CD   CHAR(2)    NOT NULL  -- FK to TRNTYPE
TRAN-CAT-CD    INTEGER    NOT NULL  -- Category code
TRAN-CAT-DESC  VARCHAR(50)          -- Category description

🔗 DB2 Cursor Patterns Used

-- Forward cursor (COTRTLIC)
EXEC SQL DECLARE TRNCUR CURSOR FOR
    SELECT TRAN-TYPE, TRAN-TYPE-DESC FROM TRNTYPE
    ORDER BY TRAN-TYPE
END-EXEC

-- Backward cursor (COTRTLIC backward navigation)
EXEC SQL DECLARE TRNCURB CURSOR WITH HOLD FOR
    SELECT TRAN-TYPE, TRAN-TYPE-DESC FROM TRNTYPE
    ORDER BY TRAN-TYPE DESC
END-EXEC

🔗 Dependencies

  • DB2: Must be available from CICS; DB2 precompiler used during program compile (CICDBCMP.jcl)
  • transaction-processing: TRAN-TYPE-CD in transactions must exist in TRNTYPE/TRANTYPE reference
  • batch-processing: TRANEXTR extracts DB2 types to VSAM TRANTYPE.PS for use in POSTTRAN, INTCALC
  • user-management: Only admin users can access CTTU/CTLI (via CA00 admin menu)

✅ Acceptance Criteria Patterns

  • Given admin accesses CTLI, when navigating forward, then transaction types are listed in alphabetical order by type code
  • Given admin adds new type via CTTU with unique type code, then DB2 INSERT succeeds (SQLCODE = 0) and confirmation displayed
  • Given admin adds duplicate type code via CTTU, then DB2 rejects with SQLCODE -803 (unique constraint) and error displayed
  • Given admin deletes a type code via CTLI, then DB2 DELETE succeeds and type no longer appears in list
  • Given TRANEXTR runs after DB2 update, then VSAM TRANTYPE.PS contains all current DB2 transaction types
  • Given DB2 is unavailable, when CTTU/CTLI accessed, then SQLCA error displayed and no partial changes made
