👥 User Management

User Story Development Guide — CardDemo Mainframe Application

Overview

Admin-only module for managing the USRSEC VSAM security file. Provides four CICS programs: list users (CU00), add user (CU01), update user (CU02), and delete user (CU03). Access to all four transactions is controlled by a user-type check enforced in the Admin Menu (COADM01C).

User Story Development Guide

Story Templates

Writing Good Stories for This Module

  • All stories in this module should specify the admin-only access constraint as a precondition.
  • Add/update stories must validate USER-TYPE is exactly A or U.
  • Delete stories must clarify that deletion is immediate and permanent — no soft-delete or undo.
  • Password stories should note the modernization target: plain-text password storage is a security risk to address.

Complexity Grid

🟢 Simple (1–2 pts)

  • List all users (read-only CU00 view)
  • View user details (ID, name, type)
  • Add display of user count on the list screen

🟡 Medium (3–5 pts)

  • Add new user with USER-TYPE validation (A/U enforcement)
  • Update user password with length/format validation
  • Update user type with confirmation screen

🔴 Complex (5–8 pts)

  • Bulk user import from a sequential file
  • Password complexity enforcement (mixed case, numbers, special chars)
  • Audit log of all user changes written to a new VSAM or DB2 table

Acceptance Criteria

GivenWhenThen
User is logged in with USER-TYPE=A (admin) CU00 is entered All USRSEC records are listed with USER-ID, name, and USER-TYPE
A new user is added via CU01 with USER-TYPE=A The new user logs in via CC00 The user is routed to the Admin Menu (COADM01C)
A user is deleted via CU03 The deleted USER-ID attempts login via CC00 Access is denied with an authentication error
A non-admin user (USER-TYPE=U) is logged in The user attempts to enter CU01 directly Access is denied and an authorization error is displayed

Business Rules

  • Access control: All four User Management transactions are accessible only to users with USER-TYPE=A; enforced by COADM01C.
  • Valid types: USER-TYPE must be exactly A (Admin) or U (Regular user); any other value is invalid.
  • Immediate deletion: Deleting a user via CU03 immediately removes the record from USRSEC; there is no soft-delete or recovery mechanism.
  • Password storage: USER-PASSWORD is stored as plain text in USRSEC — this is a known modernization target (hash + salt required).
  • Record size: USRSEC is 80-byte fixed-block KSDS; keyed on USER-ID (first 8 bytes).
