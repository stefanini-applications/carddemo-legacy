📨 MQ Integration

User Story Development Guide — CardDemo Mainframe Application

Optional Module

Overview

Optional module demonstrating the MQ request/response pattern for mainframe modernization. CODATE01 (CDRD transaction) responds to MQ requests for the current system date/time. COACCT01 (CDRA transaction) responds to MQ requests for account details by reading ACCTFILE VSAM. External MQ-compatible clients send requests; CICS programs process and place responses on the MQ reply queue.

⚠️ Prerequisites

  • Base CardDemo application installed
  • MQ queue manager configured with request and reply queues
  • CICS MQ trigger definitions installed
  • MQ-compatible client application for testing

Technical Foundation

Programs

  • CODATE01 — Returns current system date and time in MQ reply; triggered by CDRD
  • COACCT01 — Returns account balance and status from ACCTFILE in MQ reply; triggered by CDRA

User Story Development Guide

Story Templates

Writing Good Stories for This Module

  • Always specify the MQ queue names (request queue, reply queue) and the correlation ID mechanism in technical notes.
  • Error response stories must specify what is returned in the reply when the account is not found or a system error occurs.
  • Modernization stories should note the asynchronous nature of the pattern — client must await the reply queue.
  • New MQ inquiry stories require a new CICS transaction code, a new trigger definition, and a new COBOL program.

Complexity Grid

🟢 Simple (1–2 pts)

  • Read and document the system date response format (CODATE01)
  • Read and document the account response format (COACCT01)
  • Add a new field to an existing MQ reply message

🟡 Medium (3–5 pts)

  • Add a new MQ inquiry type (e.g., card status by card number)
  • Add error response codes to COACCT01 for invalid account IDs
  • Add request logging to a VSAM audit file

🔴 Complex (5–8 pts)

  • Bidirectional MQ integration with full error handling and retry
  • Dead-letter queue processing with operator notification
  • MQ to REST API gateway: CICS → MQ → API Connect → cloud

Acceptance Criteria

GivenWhenThen
An MQ request message is placed on the CDRD request queue CODATE01 processes the triggered CDRD transaction The current system date and time are returned in the MQ reply queue
A valid account ID is included in an MQ request to the CDRA queue COACCT01 processes the triggered CDRA transaction Account balance and active status are returned in the MQ reply queue
An invalid (non-existent) account ID is in the MQ request to CDRA COACCT01 processes the CDRA transaction An error response with a not-found code is placed on the MQ reply queue
A new MQ inquiry (e.g., card status) is added An MQ request is sent to the new queue The new CICS transaction is triggered and the expected response is returned

Business Rules

  • Trigger mechanism: CICS MQ trigger automatically starts the appropriate CICS transaction (CDRD or CDRA) when a message arrives on the request queue.
  • Asynchronous pattern: The request/response is asynchronous — the external client places a message and must monitor the reply queue for the response.
  • Correlation: The MQ correlation ID is used to match replies to requests; clients must preserve and check the correlation ID.
  • Data access: COACCT01 reads ACCTFILE VSAM directly; no CICS service layer — account must exist in VSAM.
  • Authentication: No application-level authentication beyond MQ queue access controls (queue manager security).
  • Pattern purpose: Demonstrates the foundational asynchronous request/response pattern for mainframe modernization — a building block for microservices integration.
