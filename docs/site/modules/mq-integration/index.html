📨 MQ Integration

User Story Development Guide — CardDemo Application (Extension Module)

📋 Overview

Purpose: Demonstrate asynchronous integration patterns using IBM MQ. Provides request/response messaging for account data inquiry (CDRA) and system date inquiry (CDRD) from external systems.

Business Context: Enables external applications to query CardDemo data without direct VSAM access. Demonstrates asynchronous messaging, message correlation, and MQ-based service patterns common in enterprise mainframe environments.

COACCT01 CODATE01 TranID: CDRA / CDRD IBM MQ Request/Response

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Inquiry via MQ: As an external system, I want to request account details via MQ so that I can integrate with CardDemo without direct VSAM access.
  • Date Inquiry via MQ: As an external system, I want to query the system date via MQ so that I can synchronize date-sensitive external processes.
  • Message Correlation: As an integration developer, I want request/response messages to be correlated by REQUEST-ID so that responses are matched to the correct request.
  • Error Handling: As an external system, I want to receive a meaningful error response when account is not found so that I can handle exceptions gracefully.

🔧 Technical Foundation

🏗️ Key Components

  • CODATE01.cbl: Date inquiry via MQ. TranID: CDRD. Reads MQ request, gets system date, puts response to reply queue.
  • COACCT01.cbl: Account inquiry via MQ. TranID: CDRA. Reads MQ request with ACCOUNT-NUMBER, retrieves account from VSAM ACCTDAT, sends account data (300 bytes) in response.

🔗 Dependencies

  • IBM MQ: Required — queues CARDDEMO.REQUEST.QUEUE and CARDDEMO.RESPONSE.QUEUE.
  • ACCTDAT VSAM: Account data source for COACCT01 responses.
  • CICS-MQ Connection: MQCONN and MQQUEUE CICS resource definitions required.

📊 MQ Message Formats

-- Date Request:
01 DATE-REQUEST-MSG.
   05 REQUEST-TYPE   PIC X(4)  VALUE 'DATE'
   05 REQUEST-ID     PIC X(8)            -- Correlation ID

-- Date Response:
01 DATE-RESPONSE-MSG.
   05 RESPONSE-TYPE  PIC X(4)  VALUE 'DATE'
   05 RESPONSE-ID    PIC X(8)
   05 SYSTEM-DATE    PIC X(10)           -- YYYY-MM-DD

-- Account Request:
01 ACCT-REQUEST-MSG.
   05 REQUEST-TYPE   PIC X(4)  VALUE 'ACCT'
   05 REQUEST-ID     PIC X(8)
   05 ACCOUNT-NUMBER PIC X(11)           -- 11-digit account ID

-- Account Response:
01 ACCT-RESPONSE-MSG.
   05 RESPONSE-TYPE  PIC X(4)  VALUE 'ACCT'
   05 RESPONSE-ID    PIC X(8)
   05 ACCOUNT-DATA   PIC X(300)          -- Full account record

✅ Acceptance Criteria Patterns

Account Inquiry Success:
  • Given a valid account number exists in ACCTDAT
  • When ACCT request message sent to CARDDEMO.REQUEST.QUEUE
  • Then response with ACCOUNT-DATA (300 bytes) placed on CARDDEMO.RESPONSE.QUEUE with matching RESPONSE-ID
Account Not Found:
  • Given account number not in ACCTDAT
  • When ACCT request sent
  • Then response placed on reply queue with ACCOUNT-DATA containing error indicator
Date Inquiry:
  • Given DATE request sent to request queue
  • When CDRD processes the message
  • Then response with SYSTEM-DATE in YYYY-MM-DD format placed on reply queue with matching RESPONSE-ID
