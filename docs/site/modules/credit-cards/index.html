💳 Credit Cards

Card List, View & Update — CARDFILE VSAM Management

📋 Overview

Module ID: credit-cards

Purpose: Manages credit card records in CARDFILE VSAM. Provides list view of all cards for an account, detailed card view, and card update functionality. Cards link to accounts via CARD-ACCT-ID and to the XREFFILE cross-reference.

Business Context: A customer may have multiple cards linked to one account. Card status (active/inactive) controls whether a card can be used for transactions. Card number is the key linking mechanism between transactions and accounts.

🎯 User Story Templates

As a regular user, I want to see all credit cards associated with my account so that I can identify which cards are active.
As a regular user, I want to view the details of a specific card (expiry date, embossed name) so that I have my card information.
As a regular user, I want to deactivate a lost/stolen card so that unauthorized use is prevented.
As an admin, I want to view card records for any account so that I can assist customers with card issues.

📊 Story Complexity

Simple 1-2 pts Medium 3-5 pts Complex 5-8 pts

Simple (1-2 pts)

  • Display card list screen (CCLI/COCRDLIC)
  • Browse CARDFILE with STARTBR/READNEXT
  • Navigate card list pages (PF7/PF8)

Medium (3-5 pts)

  • View individual card details (CCDL)
  • Update card active status (CCUP)
  • Update card embossed name or expiry date

Complex (5-8 pts)

  • Add new card to account with XREFFILE update
  • Card reissue workflow with new card number
  • Bulk card status update via batch

✅ Acceptance Criteria Patterns

Card List (CCLI): Given ACCT-ID in COMMAREA, when screen loads, then all cards with matching CARD-ACCT-ID are listed showing card number (masked), status, and expiry date.
Card View (CCDL): Given user selects a card from list, when CCDL loads, then full card details are shown: number, account ID, CVV (masked), embossed name, expiry, status.
Card Update (CCUP): Given user modifies CARD-ACTIVE-STATUS and presses ENTER, then CARDFILE is rewritten and success message is displayed.
No Cards Found: Given account has no cards in CARDFILE, then appropriate message is displayed and user can navigate back.
PF3 Navigation: PF3 from any card screen returns to card list (CCLI), PF3 from CCLI returns to main menu (CM00).

🔧 Technical Foundation

ComponentTransactionFunction
COCRDLIC.cblCCLICard list — STARTBR/READNEXT on CARDFILE filtered by ACCT-ID
COCRDSLC.cblCCDLCard detail view — READ CARDFILE by CARD-NUM
COCRDUPC.cblCCUPCard update — READ UPDATE + REWRITE on CARDFILE
CARDFILE VSAM—Credit card master — key=CARD-NUM (16 chars), RECLN=150
XREFFILE VSAM—Card-Account-Customer cross-ref — key=CARD-NUM, AIX=ACCT-ID

Card Record Structure (CVACT02Y.cpy)

01  CARD-RECORD.                          (RECLN 150)
    05  CARD-NUM               PIC X(16)  -- 16-digit card number (key)
    05  CARD-ACCT-ID           PIC 9(11)  -- links to ACCOUNT-RECORD
    05  CARD-CVV-CD            PIC 9(03)  -- security code
    05  CARD-EMBOSSED-NAME     PIC X(50)  -- name on card
    05  CARD-EXPIRAION-DATE    PIC X(10)  -- YYYY-MM-DD (typo preserved)
    05  CARD-ACTIVE-STATUS     PIC X(01)  -- 'Y'=active, 'N'=inactive

🔗 Dependencies

  • accounts: CDEMO-ACCT-ID in COMMAREA used to filter card list
  • XREFFILE: Links CARD-NUM to ACCT-ID and CUST-ID
  • transactions: TRAN-CARD-NUM references CARD-NUM
  • authentication: Session must be established before accessing cards

⚡ Development Acceleration

  • STARTBR/READNEXT pattern: COCRDLIC.cbl — reuse for any browsing list screen
  • CVCRD01Y.cpy: Card data structure — reuse in any program touching CARDFILE
  • PF7/PF8 paging: Scroll pattern established in card list for similar list screens
  • XREFFILE AIX: Alternate index on ACCT-ID allows account-based card lookups
