💰 Transactions

Transaction List, View, Add & Batch Posting

Module ID: transactions

Purpose: Core financial transaction management. Online functions allow users to list, view, and add transactions. Batch programs post daily transactions (CBTRN02C), generate reports (CBTRN03C), and combine transaction files (SORT).

Business Context: Every financial activity (purchases, payments, fees) is recorded as a transaction. Transactions link to accounts via card number through XREFFILE. Transaction type code + category code classify each transaction for interest calculation and reporting.

🎯 User Story Templates

As a regular user, I want to view my transaction history so that I can track and verify my spending.
As a regular user, I want to view the details of a specific transaction so that I have full information about a charge.
As a regular user, I want to add a new transaction so that I can record a purchase manually.
As a batch system, I want to post daily transactions from DALYTRAN to TRANFILE so that account balances are updated nightly.
As a batch system, I want to reject invalid transactions to DALYREJS with a reason code so that bad records are tracked separately.

📊 Story Complexity

Simple 1-2 pts Medium 3-5 pts Complex 5-8 pts

Simple (1-2 pts)

  • Display transaction list (CT00)
  • View single transaction detail (CT01)
  • Navigate pages with PF7/PF8

Medium (3-5 pts)

  • Add new transaction with type/category (CT02)
  • Validate card number via XREFFILE lookup
  • CBTRN03C: generate transaction report

Complex (5-8 pts)

  • POSTTRAN (CBTRN02C): full daily transaction posting with balance update
  • COMBTRAN: SORT-based file merge with complex control cards
  • Transaction reversal / correction workflow

✅ Acceptance Criteria Patterns

Transaction List (CT00): Given CARD-NUM in COMMAREA, when screen loads, then transactions are listed in reverse chronological order showing: ID, date, amount, merchant name, type code.
Transaction View (CT01): Given user selects transaction, when CT01 loads, then all fields are displayed: ID, type, category, source, description, amount, card number, merchant details, timestamps.
Transaction Add (CT02): Given user fills required fields and presses ENTER, then new TRAN-ID is generated, record written to TRANFILE, and success message displayed.
Card Validation: Given card number entered in CT02, when ENTER pressed, then XREFFILE is read to validate card exists; if not found, error message displayed.
POSTTRAN Reject: Given transaction with invalid account or card, then record is written to DALYREJS with reason code and processing continues with next record.
Balance Update: Given valid transaction in POSTTRAN, then ACCT-CURR-BAL is updated and TRAN-CAT-BAL-RECORD is updated for interest calculation.

Transaction Record (CVTRA series)

01  TRAN-RECORD.                           (RECLN ~350)
    05  TRAN-ID               PIC X(16)   -- unique transaction ID (key)
    05  TRAN-TYPE-CD          PIC X(02)   -- transaction type (e.g. 'PU', 'PY')
    05  TRAN-CAT-CD           PIC 9(04)   -- category code
    05  TRAN-SOURCE           PIC X(10)   -- origination source
    05  TRAN-DESC             PIC X(100)  -- description
    05  TRAN-AMT              PIC S9(10)V99
    05  TRAN-CARD-NUM         PIC X(16)   -- links to CARD-RECORD
    05  TRAN-MERCHANT-ID      PIC 9(15)
    05  TRAN-MERCHANT-NAME    PIC X(50)
    05  TRAN-MERCHANT-CITY    PIC X(50)
    05  TRAN-MERCHANT-ZIP     PIC X(10)
    05  TRAN-ORIG-TS          PIC X(26)   -- origination timestamp
    05  TRAN-PROC-TS          PIC X(26)   -- processing timestamp

🔗 Dependencies

  • credit-cards: CARD-NUM from CARDFILE used as transaction key
  • accounts: ACCT-CURR-BAL updated during POSTTRAN
  • XREFFILE: Links card number to account ID
  • TCATBALF: Category balances updated for interest calculation
  • reports: CBTRN03C produces report output
  • transaction-type-db2 (optional): TRAN-TYPE-CD validated against DB2 table

⚡ Development Acceleration

  • XREFFILE AIX: Alternate index on ACCT-ID enables account-to-transaction lookups
  • CBTRN02C pattern: Sequential file + random VSAM update — template for any batch posting job
  • TRAN-ID generation: Established pattern in CT02 — reuse for any new transaction type
  • CVTRA01Y-07Y copybooks: Full set of transaction data structures ready to reuse
