📦 Authorization

Guide for Developing User Stories

📋 Overview

Purpose: Provide the mainframe simulation of credit card authorizations, fraud evaluation, IMS persistence, and DB2 analytics tied to MQ-based merchant requests.

Business Context: This extension models how modern systems interact with legacy middleware—MQ triggers, IMS storage, DB2 analytics, and COBOL screens for manual review.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Fraud Marking: As a fraud analyst, I want PF5 to flag suspicious authorizations in CPVD so the DB2 fraud table knows the investigation status.
  • MQ Integration: As an integration developer, I want `AWS.M2.CARDDEMO.PAUTH.REPLY` to contain APPROVED-AMT and AUTH-RESP-CODE so merchant systems can decide in real time.
  • Retention: As a platform owner, I want `CBPAUP0J` to purge 30-day-old authorizations so the IMS database stays lean.

⚡ Development Acceleration Factors

  • MQ Templates: Request/response formats defined in the README ensure new payloads follow the same comma-delimited layout.
  • DB2 Copybooks: Use `AUTHFRDS` structure when enriching fraud rows; SQL parameter names already align with DB2 table columns.
  • BMS Map Reuse: Authorization summary/detail screens share the `COPAU00`/`COPAU01` mapsets and the `COPAUS*` programs.

📊 Complexity Guidelines

  • Simple: Add another status field to the summary screen without changing persistence.
  • Medium: Adjust MQ payload parsing to capture a new merchant field and persist it in IMS.
  • Complex: Introduce a new MQ queue/formatter plus DB2 update plus batch purge adjustment.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL Programs: COPAUA0C (MQ-triggered request processor), COPAUS0C/COPAUS1C (summary/detail), COPAUS2C (fraud flag updates), CBPAUP0C (batch purge).
  • BMS Maps: Map sets COPAU00, COPAU01 carry authorization screens.
  • IMS/DB2 Artifacts: IMS DBD/PSBs under app/app-authorization-ims-db2-mq/ims and DB2 table AUTHFRDS + index XAUTHFRD.
  • MQ Queues: Input queue AWS.M2.CARDDEMO.PAUTH.REQUEST, reply queue AWS.M2.CARDDEMO.PAUTH.REPLY, correlation via request ID.

🔗 Dependencies

  • IMS HIDAM: Authorization history stored in nested segments and PSBs (PAUTSUM0, PAUTDTL1).
  • DB2 Schema: Fraud audit rows in AUTHFRDS; schema name must match the environment referenced in COPAUS2C.cbl.
  • MQ Client: Any MQ-capable client may hit the request queue using the documented payload.

🌐 Public Interfaces

AWS.M2.CARDDEMO.PAUTH.REQUEST → authorization request (AUTH-DATE, AUTH-TIME, CARD-NUM, ..., TRANSACTION-ID)
AWS.M2.CARDDEMO.PAUTH.REPLY → response (CARD-NUM, TRANSACTION-ID, AUTH-ID-CODE, AUTH-RESP-CODE/REASON, APPROVED-AMT)
CICS CP00/CPVS/CPVD → summary, detail, fraud marking screens
JCL CBPAUP0J → daily purge of expired authorizations

✅ Acceptance Criteria Examples

  • Given an MQ request arrives with recognizable card data, when COPAUA0C runs, then IMS stores the request and MQ replies with AUTH-RESP-CODE within 500 ms.
  • Given an analyst presses PF5 inside CPVD, when COPAUS2C runs, then DB2 AUTHFRDS column AUTH_FRAUD toggles to 'Y' and FRAUD_RPT_DATE equals today.
