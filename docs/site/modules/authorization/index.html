🔏 Authorization (IMS/DB2/MQ)

User Story Development Guide — CardDemo Mainframe Application

Optional Module

Overview

Optional module providing real-time credit card authorization via MQ message trigger. COPAUA0C processes MQ authorization requests, validates account and card, and stores results in IMS DB (DBPAUTP0). Users can view pending authorizations (CPVS) and details (CPVD), and can flag fraud stored in DB2. Batch CBPAUP0C purges expired authorizations and restores available credit.

⚠️ Prerequisites

  • Base CardDemo application installed
  • IMS DB setup: DBPAUTP0 / DBPAUTX0 (via DBDGEN/PSBGEN)
  • DB2 tables created: CREADB21 + LOADPADB JCL jobs
  • MQ queue configuration (request queue + reply queue)
  • CICS resource group CRDDEMO2 installed

Technical Foundation

Programs

  • COPAUA0C — MQ-triggered authorization processor: validates card/account, writes to IMS DB
  • COPAUS0C — Authorization summary screen (pending list)
  • COPAUS1C — Authorization detail and fraud flagging screen
  • COPAUS2C — Authorization detail subroutine
  • CBPAUP0C — Batch: purge expired authorizations from IMS DB, restore credit

BMS Maps

  • COPAU00 — Authorization summary map
  • COPAU01 — Authorization detail/fraud map

CICS Transactions

User Story Development Guide

Story Templates

PersonaUser StoryTags
Cardholder As a cardholder, I want to view pending authorizations on my account so that I can identify suspicious charges. Read-only
Fraud Analyst As a fraud analyst, I want to mark an authorization as fraudulent so that the risk team is alerted and a DB2 fraud record is created. Fraud
System Operator As a system operator, I want expired authorizations purged nightly so that available credit is restored to cardholders. Batch
Integration Developer As an integration developer, I want to send authorization requests via MQ so that real-time approval or decline is returned. Integration

Writing Good Stories for This Module

  • Specify the MQ queue names (request and reply) and the triggering mechanism in technical notes.
  • Fraud flag stories must mention both the CICS update (COPAUS1C) and the DB2 insert (AUTHFRDS).
  • Batch purge stories must note the two effects: IMS record deletion AND ACCTFILE credit restoration.
  • Two-phase commit (IMS + DB2) scenarios need explicit acceptance criteria for rollback behavior.

Complexity Grid

🟢 Simple (1–2 pts)

  • View pending authorization summary list (CPVS read-only)
  • Add a new display field to the authorization summary screen
  • View authorization detail without fraud action (CPVD read-only)

🟡 Medium (3–5 pts)

  • View authorization detail (CPVD)
  • Mark authorization as fraud (DB2 AUTHFRDS insert via COPAUS1C)
  • Add new field to MQ authorization request/reply format

🔴 Complex (5–8 pts)

  • Full MQ authorization processor implementation (COPAUA0C)
  • IMS DB schema change (new segment or field)
  • Two-phase commit IMS+DB2 enhancement for fraud workflow

Acceptance Criteria

GivenWhenThen
A pending authorization record exists in IMS DB (DBPAUTP0) CPVS is entered The authorization appears in the summary list
An authorization is flagged as fraud via CPVD The fraud flag is submitted A record is inserted into DB2 table AUTHFRDS
Authorizations older than the expiry threshold exist in IMS DB CBPAUP0J runs nightly Records are deleted from IMS DB and available credit is restored in ACCTFILE
An MQ authorization request with valid card and sufficient credit CP00 (COPAUA0C) processes the message An approved response is sent to the MQ reply queue
An MQ authorization request with CARD-ACTIVE-STATUS=N CP00 (COPAUA0C) processes the message A declined response with reason code is sent to the MQ reply queue

Business Rules

  • Authorization decision: Approved if available credit (ACCT-CREDIT-LIMIT − ACCT-CURR-BAL) ≥ requested amount AND CARD-ACTIVE-STATUS=Y.
  • Two-phase commit: Fraud flagging performs a two-phase commit across IMS DB (authorization update) and DB2 (AUTHFRDS insert).
  • Batch purge: CBPAUP0C deletes expired authorization records from IMS DB and adjusts ACCT-CURR-BAL in ACCTFILE when no matching posted transaction exists.
  • MQ trigger: CICS MQ trigger starts the CP00 transaction on message arrival in the request queue.
  • Resource group: Requires CICS resource group CRDDEMO2 in addition to the base CRDDEMO1.
