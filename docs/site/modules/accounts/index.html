🏦 accounts

CardDemo Mainframe Application · Core Module

Core Module CICS Online Batch JCL VSAM KSDS

Overview

The accounts module is the central entity of the CardDemo application. It manages the lifecycle of credit card accounts: online viewing and updating of account details through CICS 3270 screens, and nightly batch processing for interest calculation and data maintenance. Every other module (transactions, bill-payment, credit-cards, authorization) depends on or references the account record defined here.

The module consists of two CICS online programs (COACTVWC for view, COACTUPC for update), four batch COBOL programs (CBACT01C–CBACT04C), two BMS screen definitions, and the shared copybook CVACT01Y that defines the 300-byte account record layout used throughout the application.

Account data is stored in ACCTDATA VSAM KSDS (AWS.M2.CARDDEMO.ACCTDATA.VSAM.KSDS) with ACCT-ID (11-digit numeric) as the primary key. Every VSAM read or write to this file is governed by the CVACT01Y copybook layout.

Navigation Context

Both transactions are accessible from the Main Menu (CM00 / COMEN01C) after a successful sign-on. COMMAREA state (COCOM01Y) carries the current account ID and navigation history across XCTL calls. The update screen (CAUP) provides PF-key navigation to the credit card list (CCLI / COCRDLIC) and card update (CCUP / COCRDUPC).

Architectural Context

The accounts module follows the standard CardDemo CICS Online Transaction Pattern:

  1. Transaction ID (CAVW or CAUP) maps to a COBOL program via the CICS CSD.
  2. Program initialises from COMMAREA (COCOM01Y) and a program-specific COMMAREA extension (WS-THIS-PROGCOMMAREA).
  3. BMS RECEIVE MAP collects 3270 keyboard input; BMS SEND MAP paints the screen.
  4. Business logic reads/writes VSAM via EXEC CICS READ/REWRITE.
  5. Navigation is achieved by EXEC CICS XCTL with an updated COMMAREA; state is preserved across calls.
  6. EXEC CICS RETURN TRANSID(LIT-THISTRANID) suspends the task and waits for the next user input.

Business Rules

Account Identification

  • Account ID (ACCT-ID) is an 11-digit numeric value; it must be non-zero and must exist in ACCTDATA VSAM.
  • Lookup starts via CARDXREF AIX path (CXACAIX): the user supplies an account ID or a card number; the cross-reference resolves the account.
  • If account is not found in ACCTDATA, the program sets DID-NOT-FIND-ACCT-IN-ACCTDAT and displays an error message — no account data is shown.

Account Status

  • ACCT-ACTIVE-STATUS stored as 'A' (Active) or 'I' (Inactive).
  • The update screen presents this as a Y/N flag; COACTUPC maps 'Y'→'A' and 'N'→'I' during validation (FLG-ACCT-STATUS-ISVALID).
  • Inactive accounts cannot have new financial transactions applied; this is enforced in COTRN02C (transaction add) and COPAUA0C (authorization).

Credit Limit Validation

  • All monetary fields (ACCT-CURR-BAL, ACCT-CREDIT-LIMIT, ACCT-CASH-CREDIT-LIMIT, ACCT-CURR-CYC-CREDIT, ACCT-CURR-CYC-DEBIT) are validated as signed 10-digit numbers with 2 decimal places via 1250-EDIT-SIGNED-9V2.
  • A blank monetary field is treated as no-change (preserves current value).
  • Transaction processing in other modules checks available credit: ACCT-CREDIT-LIMIT − ACCT-CURR-BAL must be ≥ transaction amount.

Customer Data Co-Update

  • COACTUPC updates both the ACCTDATA record and the linked CUSTDATA record in a single PF5 operation.
  • Both files are locked with EXEC CICS READ UPDATE before either is rewritten, minimising the window for concurrent-update conflicts.
  • If the account record has been changed by another user since the screen was last loaded (stale-data check via 1205-COMPARE-OLD-NEW), the update is rejected with DATA-WAS-CHANGED-BEFORE-UPDATE error message.
  • If either lock fails, the error COULD-NOT-LOCK-ACCT-FOR-UPDATE or COULD-NOT-LOCK-CUST-FOR-UPDATE is displayed and no writes occur.

Interest Calculation (CBACT04C)

  • CBACT04C reads TCATBALF sequentially, grouped by account. For each account group it calls 1050-UPDATE-ACCOUNT.
  • For each transaction category (TRAN-TYPE-CD + TRAN-CAT-CD), it looks up the interest rate from DISCGRP using the account's ACCT-GROUP-ID.
  • Monthly interest is computed as: WS-MONTHLY-INT = balance × rate / 1200 (annual rate ÷ 12 months).
  • A new interest transaction record is written to the TRANSACT sequential output file for each interest charge.
  • The program receives a date parameter from JCL PARM (PARM-DATE PIC X(10)) used as the transaction processing date.
  • If an account has zero balance in a category, no interest transaction is generated for that category.

Development Guidelines

CICS Online Program Patterns

All CICS programs in the accounts module follow the CardDemo standard pattern. New programs should replicate this structure:

PROCEDURE DIVISION.
*-- 1. Install ABEND handler
    EXEC CICS HANDLE ABEND LABEL(ABEND-ROUTINE) END-EXEC.
*-- 2. Retrieve COMMAREA (navigation state + program state)
    EXEC CICS ADDRESS COMMAREA(WS-COMMAREA) END-EXEC.
    MOVE DFHCOMMAREA(1:LENGTH OF CARDDEMO-COMMAREA) TO CARDDEMO-COMMAREA.
    MOVE DFHCOMMAREA(LENGTH OF CARDDEMO-COMMAREA + 1: ...) TO WS-THIS-PROGCOMMAREA.
*-- 3. Store PF key (COPY CSSTRPFY)
    PERFORM YYYY-STORE-PFKEY.
*-- 4. Evaluate AID key / program state
    EVALUATE TRUE
      WHEN CCARD-AID-PFK03  -- Exit
           PERFORM EXIT-ROUTINE
      WHEN CDEMO-PGM-ENTER  -- First time: send empty/initial screen
           PERFORM 1000-SEND-MAP
      WHEN CDEMO-PGM-REENTER  -- Return from user input
           PERFORM 2000-PROCESS-INPUTS
           IF INPUT-OK PERFORM 9000-READ-ACCT
           PERFORM 1000-SEND-MAP
    END-EVALUATE.
*-- 5. Return with TRANSID to stay in this transaction
    EXEC CICS RETURN
         TRANSID(LIT-THISTRANID)
         COMMAREA(WS-COMMAREA)
         LENGTH(LENGTH OF WS-COMMAREA)
    END-EXEC.

VSAM Read-for-Update Pattern (COACTUPC)

*-- Lock record before update (prevents concurrent modification)
    EXEC CICS READ UPDATE
         DATASET (LIT-ACCTFILENAME)
         INTO    (ACCOUNT-RECORD)
         RIDFLD  (WS-ACCT-ID)
         RESP    (WS-RESP-CD)
    END-EXEC.
    IF WS-RESP-CD = DFHRESP(NORMAL)
       -- Apply changes to ACCOUNT-RECORD fields
       EXEC CICS REWRITE
            DATASET (LIT-ACCTFILENAME)
            FROM    (ACCOUNT-RECORD)
            RESP    (WS-RESP-CD)
       END-EXEC
    ELSE
       SET COULD-NOT-LOCK-ACCT-FOR-UPDATE TO TRUE
    END-IF.

Adding a New Field to the Account Record

⚠ FILLER Reduction Pattern: The 178-byte FILLER in CVACT01Y is specifically reserved for adding new fields. To add a field:
  1. Replace bytes from the FILLER in CVACT01Y.cpy (e.g., 05 ACCT-NEW-FIELD PIC X(10).) and reduce FILLER accordingly to maintain 300-byte total.
  2. Add the new field to the COACTUP BMS map (COACTUP.bms) if it should be editable online.
  3. Update COACTUP.CPY symbolic map copybook to match the BMS map change.
  4. Update validation logic in COACTUPC sections 1200-EDIT-MAP-INPUTS and 3200-SETUP-SCREEN-VARS.
  5. Update display logic in COACTVWC sections for the view screen if needed.
  6. All batch programs (CBACT01C–04C) that COPY CVACT01Y will recompile automatically; check their output record layouts.

Error Message Conventions

Both COACTVWC and COACTUPC use a standard error message working storage area:

05  WS-RETURN-MSG               PIC X(75).
  88  WS-RETURN-MSG-OFF          VALUE SPACES.

* Error conditions are set via 88-level condition names, e.g.:
  88  DID-NOT-FIND-ACCT-IN-ACCTDAT  VALUE 'Account not found in Account Data file'.
  88  COULD-NOT-LOCK-ACCT-FOR-UPDATE VALUE 'Could not lock account record for update'.
  88  DATA-WAS-CHANGED-BEFORE-UPDATE VALUE 'Record changed by some one else. Please review'.

Acceptance Criteria Patterns

Account View (CAVW / COACTVWC)

  • GIVEN a signed-in user with a valid 11-digit account ID, WHEN CAVW is invoked and Enter is pressed, THEN the account balance, credit limit, dates, and linked customer name are displayed from ACCTDATA VSAM and CUSTDATA VSAM.
  • GIVEN an account ID that does not exist in ACCTDATA, WHEN CAVW is invoked, THEN the error message "Account not found in Account Data file" is displayed and no account fields are populated.
  • GIVEN a valid account ID with no CARDXREF entry, WHEN CAVW performs 9200-GETCARDXREF-BYACCT, THEN the program continues to display available account data (CARDXREF is supplementary).
  • GIVEN a user presses PF3, WHEN COACTVWC evaluates the AID key, THEN the program XCTLs back to the calling program or Main Menu (CM00).

Account Update (CAUP / COACTUPC)

  • GIVEN a valid account with modified credit limit, WHEN PF5 is pressed and 9600-WRITE-PROCESSING executes, THEN both ACCTDATA and CUSTDATA VSAM records are updated atomically and a success message is displayed.
  • GIVEN an invalid SSN format (e.g., area code 000), WHEN PF5 is pressed, THEN 1265-EDIT-US-SSN sets INPUT-ERROR and the SSN field is highlighted with an error indicator.
  • GIVEN the account record was changed by another process after the screen was loaded, WHEN PF5 is pressed and 1205-COMPARE-OLD-NEW detects a discrepancy, THEN the update is rejected with "Record changed by some one else. Please review" message.
  • GIVEN a VSAM record lock fails (another task holds the update lock), WHEN 9600-WRITE-PROCESSING executes, THEN "Could not lock account record for update" error is displayed and no data is written.
  • GIVEN a user presses PF12, WHEN COACTUPC evaluates the AID key, THEN the current account data is re-read from VSAM and the screen is refreshed with original values (edits discarded).

Batch Interest Calculation (INTCALC / CBACT04C)

  • GIVEN accounts with non-zero balances in TCATBALF, WHEN CBACT04C runs, THEN monthly interest is calculated correctly for each account-group-ID rate and written as a transaction to TRANSACT(+1).
  • GIVEN an account with ACCT-GROUP-ID not found in DISCGRP, WHEN CBACT04C attempts the rate lookup, THEN the account is skipped and an error display is produced (no abend).
  • GIVEN a complete batch run with valid input files, WHEN CBACT04C completes, THEN the job step returns condition code 0 and WS-RECORD-COUNT matches the number of records processed.
  • GIVEN an account with zero balance in all transaction categories, WHEN CBACT04C processes that account, THEN no interest transaction is generated and ACCT-CURR-BAL is unchanged.

Batch Account Read/Update Utilities

  • GIVEN CBACT01C is executed, WHEN it completes, THEN OUTFILE contains one flat record per ACCTFILE entry and record counts match.
  • GIVEN CBACT02C is provided an ACCT-ID that exists in ACCTFILE, WHEN it runs, THEN the keyed read returns the correct account record and condition code is 0.
  • GIVEN CBACT03C is provided updated account data, WHEN it runs, THEN the VSAM record is rewritten with new values and file status is '00'.

Initialisation — ACCTFILE.jcl

To reinitialise the account VSAM file (e.g., for a fresh demo environment):

//STEP05 EXEC PGM=IDCAMS        -- Delete existing VSAM cluster
//SYSIN DD *
   DELETE AWS.M2.CARDDEMO.ACCTDATA.VSAM.KSDS CLUSTER
   IF MAXCC LE 08 THEN SET MAXCC = 0

//STEP10 EXEC PGM=IDCAMS        -- Define new VSAM cluster
//SYSIN DD *
   DEFINE CLUSTER (NAME(AWS.M2.CARDDEMO.ACCTDATA.VSAM.KSDS)
          CYLINDERS(1 5) KEYS(11 0) RECORDSIZE(300 300)
          SHAREOPTIONS(2 3) ERASE INDEXED)

//STEP15 EXEC PGM=IDCAMS        -- Load initial data from flat file
//ACCTDATA DD DISP=SHR,DSN=AWS.M2.CARDDEMO.ACCTDATA.PS
//ACCTVSAM DD DISP=SHR,DSN=AWS.M2.CARDDEMO.ACCTDATA.VSAM.KSDS
//SYSIN DD *
   REPRO INFILE(ACCTDATA) OUTFILE(ACCTVSAM)

Modernisation Notes

⚠ Tech Debt and Modernisation Considerations:
  • No audit trail: Account updates do not log who changed what or when. A modernised implementation should capture operator ID, timestamp, old value, and new value for every field change.
  • Plain-text storage: No encryption of sensitive fields (e.g., SSN stored in CUSTDATA, CVV in CARDDATA). PCI-DSS and data-at-rest standards require encryption for production use.
  • No optimistic locking token: Stale-data detection in COACTUPC relies on field-by-field comparison of old vs. new values (1205-COMPARE-OLD-NEW). A modernised approach would use a record version number or timestamp.
  • Hardcoded date in INTCALC PARM: The JCL PARM date must be manually updated for each run. Automation via symbolic JCL variables or a date-generation step is recommended.
  • FILLER typo in source: ACCT-EXPIRAION-DATE has a typo (missing 't'). Correction requires recompilation of all programs that COPY CVACT01Y.
  • Interest rate granularity: DISCGRP stores rates at transaction-category level. Modern implementations typically support promotional rates, teaser rates, and dynamic pricing — requiring schema extension.
