Core Module CR00 · CBTRN03C · CBSTM03A

📋 Overview

The Reports module bridges online CICS interaction with batch report generation. A user submits a report request online (CR00), which is submitted to the internal reader (INTRDR) as a batch job. Batch programs then generate transaction reports and account statements stored in GDG (Generation Data Group) datasets for archival and printing.

Business Context

Reports cover a date range specified interactively on the CR00 screen. The internal reader (INTRDR) mechanism allows CICS to submit batch jobs dynamically without operator intervention — CORPT00C writes the JCL stream to INTRDR, which MVS picks up and schedules immediately.

Account statements are stored in GDG (Generation Data Group) datasets. GDG base datasets are defined by DEFGDGB and DEFGDGD JCL members — each statement run increments the generation number, preserving a rolling history of statement generations. Outputs are available for printing or downstream archival systems.

🎯 User Story Development Guide

Story Templates

Request Transaction Report for Date Range
As a manager, I want to submit a request for a transaction report covering a specific date range so that I can review activity for any given period.

Program: CORPT00C (CR00) — accepts START-DATE and END-DATE, builds JCL with DATE-FROM/DATE-TO parameters, submits to INTRDR; batch TRANREPT job is scheduled automatically
View Account Statement
As a cardholder, I want to retrieve my most recent account statement so that I can review my billing cycle activity.

CREASTMT (CBSTM03A) generates the statement and writes it to GDG (+1 generation); statement contains account summary, transaction detail, payment due
Admin: Generate Monthly Statements for All Accounts
As a billing administrator, I want to trigger monthly statement generation for all active accounts so that statements are produced at end of cycle.

CREASTMT JCL job runs CBSTM03A sequentially across all ACCTDATA records; each account gets a GDG generation; run typically follows EOD batch pipeline
Access Statement History from GDG
As a customer service representative, I want to access a prior month's statement so that I can answer a customer inquiry about past billing.

GDG relative generation numbers (0, -1, -2 …) allow access to previous statement generations; DEFGDGB defines the maximum number of generations retained

Acceleration Factors

  • CORPT00C internal reader submission pattern — complete INTRDR write pattern reusable for any online-to-batch job submission story
  • CBTRN03C date-filter pattern — sequential TRANSACT VSAM scan with date range comparison is directly reusable for any date-filtered report story
  • CBSTM03A GDG write pattern — GDG (+1) dataset definition and write pattern is the template for any archival output story
  • CBSTM03B support routines — formatting and calculation routines callable from new statement sections

Internal Reader Submission Pattern

CORPT00C (CR00) INTRDR Submission:
1. Accept START-DATE and END-DATE from user
2. Validate date range (START <= END, valid format)
3. Open CICS INTRDR file
4. Write JCL header records (//TRANREPT JOB ...)
5. Write EXEC PGM=CBTRN03C step
6. Write DD statements with DATE-FROM=&START, DATE-TO=&END
7. Close INTRDR (triggers MVS job scheduler)
8. Display confirmation with submitted job name on screen

✅ Acceptance Criteria Patterns

Online Report Request (CR00)

  • GIVEN a valid date range, WHEN CR00 submits the report request, THEN JCL is written to INTRDR and a job ID is displayed as confirmation
  • GIVEN an invalid date range (start after end), WHEN CR00 validates, THEN an error message is displayed and no job is submitted
  • GIVEN an invalid date format, WHEN CR00 validates, THEN a format-error message is displayed

Transaction Report (CBTRN03C / TRANREPT)

  • GIVEN a TRANREPT job with a valid date range, WHEN the job completes, THEN return code is 0 and the report contains all transactions within the specified date range
  • GIVEN a date range with no transactions, WHEN CBTRN03C completes, THEN the report contains a "no transactions found" message and RC=0

Statement Generation (CBSTM03A / CREASTMT)

  • GIVEN a CREASTMT run, WHEN CBSTM03A completes, THEN the GDG generation number is incremented by 1 and the new statement is accessible as GDG(0)
  • GIVEN a completed statement run, WHEN the previous generation is accessed (GDG(-1)), THEN the prior statement is still available and unmodified
  • GIVEN a CREASTMT run, WHEN it completes with RC=0, THEN every active account has a corresponding statement generation
