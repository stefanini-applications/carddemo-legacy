📦 Transaction Type

Guide for stories that touch the DB2-backed transaction type administration flow.

📋 Overview

Purpose: Administer transaction type reference data from DB2 while keeping VSAM-based consumers in sync via nightly exports.

Business Context: This module demonstrates how relational metadata (CRUD, cursors, transaction management) integrates with the legacy VSAM core to validate transaction codes for posting and reporting.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • CRUD maintenance: As a transaction-type admin, I want to add a new `TR_TYPE` and its category so the transaction list displays the correct meaning.
  • Batch sync: As an operations engineer, I want `TRANEXTR` to produce VSAM-ready files after I finish editing DB2 rows so the core system picks up the latest codes.
  • Error handling: As a developer, I want the `CTLI` screen to surface SQLCA errors when DB2 rejects an update so I can reproduce the issue in testing.

⚡ Development Acceleration Factors

  • DB2 declarations: `dcl/` files already outline the host variables used by `COTRTUPC`/`COTRTLIC`, making it easy to extend or reuse new tables.
  • Cursor demos: Forward/backward cursor examples exist in `CTLI` and can be copied to support additional listing screens.
  • JCL templates: `CREADB21.jcl`, `TRANEXTR.jcl`, and `MNTTRDB2.jcl` show how to sequence DB2 object creation, extraction, and maintenance; reuse them for new tables.

📊 Complexity Guidelines

  • Simple: Add a new column to the `TRANSACTION_TYPE` DB2 table and expose it on the `CTTU` map without altering the VSAM extract.
  • Medium: Introduce a new category screen using the existing cursor pattern (copy `CTLI`), and ensure `TRANEXTR` still outputs the updated payload.
  • Complex: Change the extraction format to deliver JSON to another system while also feeding VSAM; this requires updates to `TRANEXTR`, copybooks, and JCL.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL transactions `COTRTUPC` (add/edit) and `COTRTLIC` (list/update/remove) plus batch `COBTUPDT`.
  • BMS map definitions for CTLI/CTTU providing list columns and edit fields, including `TRC`/`TR` data entry slots.
  • DB2 DDL for `TRANSACTION_TYPE` and `TRANSACTION_TYPE_CATEGORY` (with referential integrity) plus SQL scripts under `ddl/`.
  • JCL jobs `CREADB21`, `TRANEXTR`, and `MNTTRDB2` to create schemas, export to VSAM, and perform maintenance.

🔗 Dependencies

  • DB2 subsystem with appropriate plans (`DB201PLN`), transactions, and storage group definitions.
  • VSAM files that consume the exported transaction type data (`TRANTYPE.PS`, `TRANCATG.PS`).
  • Admin menu wiring so `CA00` exposes options 5/6 when this module is installed.

🌐 Public Interfaces

CTTU  - Transaction Type add/edit / COTRTUPC
CTLI  - Transaction Type list/update/delete / COTRTLIC
Batch: CREADB21 (DB2 objects), TRANEXTR (VSAM export), MNTTRDB2 (batch updates)

✅ Acceptance Criteria Examples

  • When a new `TR_TYPE` row is created via `CTTU`, the DB2 `TRANSACTION_TYPE` table stores it and `TRANEXTR` refreshes the VSAM file before the next batch cycle.
  • `CTLI` uses the DB2 cursor pattern to show forward/backward navigation and flags SQLCA errors in the UI when `DELETE RESTRICT` fires.
  • `MNTTRDB2` keeps VSAM customers in sync even if DB2 updates fail mid-run; rollback leaves VSAM data unchanged.
