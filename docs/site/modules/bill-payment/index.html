1. Overview & Purpose

The bill-payment module enables cardholders to submit full balance payments against their credit card account online through a CICS 3270 terminal session. It is a single-screen, confirmation-driven workflow that reads the current account balance, prompts for user confirmation, and — upon confirmed approval — atomically writes a payment transaction record to the TRANSACT VSAM file and decrements the account balance in the ACCTDAT VSAM file.

The module covers the complete online bill payment lifecycle:

  • Account lookup via 11-digit account ID
  • Display of current outstanding balance
  • Explicit Y/N confirmation prompt to prevent accidental payments
  • Card cross-reference lookup to associate payment with the primary card number
  • Sequential transaction ID generation (increment-from-max strategy)
  • Atomic write of payment transaction record (type code 02)
  • Account balance update via CICS REWRITE
  • Success confirmation with generated Transaction ID
Note: Bill payment always pays the full outstanding balance. Partial payment is not supported in the base CardDemo application. The payment amount is taken directly from ACCT-CURR-BAL at the time of confirmation.

2. Architectural Context

The bill-payment module follows the standard CardDemo CICS Online Transaction Processing pattern: a 4-character transaction ID (CB00) maps to a single COBOL program (COBIL00C) that owns one BMS mapset (COBIL00) defining one 3270 screen (COBIL0A). State between screen interactions is passed entirely via the CICS COMMAREA (defined by COCOM01Y.cpy).

Position in the Application

Bill payment is accessible from the Main Menu (COMEN01C / CM00) and requires the user to be authenticated. It interacts with three VSAM files:

  • ACCTDAT — reads and updates the account balance record
  • CXACAIX — alternate index on CARDXREF (keyed by account ID) to retrieve the primary card number
  • TRANSACT — browses for the highest existing transaction ID, then writes the new payment record

COMMAREA Navigation

COBIL00C uses the shared CARDDEMO-COMMAREA structure (copybook COCOM01Y) to receive navigation context (calling program, transaction ID) and to return control to the appropriate screen on PF3 (Back). The program re-enters itself via EXEC CICS RETURN TRANSID(CB00).

6. Data Models

Account Record (CVACT01Y) — ACCTDAT VSAM KSDS, 300 bytes

Key: ACCT-ID (11-digit numeric). The bill payment module reads this record in UPDATE mode, displays the current balance, and rewrites it after decrementing the balance.

01  ACCOUNT-RECORD.
    05  ACCT-ID                  PIC 9(11).     *> Primary key
    05  ACCT-ACTIVE-STATUS       PIC X(01).     *> A=Active, I=Inactive
    05  ACCT-CURR-BAL            PIC S9(10)V99. *> Current balance (READ + decremented)
    05  ACCT-CREDIT-LIMIT        PIC S9(10)V99. *> Credit limit
    05  ACCT-CASH-CREDIT-LIMIT   PIC S9(10)V99. *> Cash credit limit
    05  ACCT-OPEN-DATE           PIC X(10).     *> YYYY-MM-DD
    05  ACCT-EXPIRAION-DATE      PIC X(10).
    05  ACCT-REISSUE-DATE        PIC X(10).
    05  ACCT-CURR-CYC-CREDIT     PIC S9(10)V99. *> Cycle credits
    05  ACCT-CURR-CYC-DEBIT      PIC S9(10)V99. *> Cycle debits
    05  ACCT-ADDR-ZIP            PIC X(10).
    05  ACCT-GROUP-ID            PIC X(10).
    05  FILLER                   PIC X(178).
Fields used by bill-payment: ACCT-ID (key), ACCT-CURR-BAL (read for display and as payment amount; decremented on payment).

Card Cross-Reference Record (CVACT03Y) — CARDXREF VSAM, 50 bytes

Accessed via the CXACAIX alternate index (keyed by XREF-ACCT-ID) to retrieve the card number associated with the paying account. The card number is embedded in the payment transaction record as TRAN-CARD-NUM. 

01  CARD-XREF-RECORD.
    05  XREF-CARD-NUM            PIC X(16). *> 16-digit card number
    05  XREF-CUST-ID             PIC 9(09). *> Customer ID
    05  XREF-ACCT-ID             PIC 9(11). *> Account ID (AIX key)
    05  FILLER                   PIC X(14).

Transaction Record (CVTRA05Y) — TRANSACT VSAM KSDS, 350 bytes

The bill payment module browses this file to generate a new unique transaction ID, then writes a new payment record. The payment is classified as transaction type 02 with category code 2. 

01  TRAN-RECORD.
    05  TRAN-ID              PIC X(16).       *> Generated: max+1
    05  TRAN-TYPE-CD         PIC X(02).       *> '02' (payment)
    05  TRAN-CAT-CD          PIC 9(04).       *> 2
    05  TRAN-SOURCE          PIC X(10).       *> 'POS TERM'
    05  TRAN-DESC            PIC X(100).      *> 'BILL PAYMENT - ONLINE'
    05  TRAN-AMT             PIC S9(09)V99.   *> = ACCT-CURR-BAL at time of payment
    05  TRAN-MERCHANT-ID     PIC 9(09).       *> 999999999 (internal)
    05  TRAN-MERCHANT-NAME   PIC X(50).       *> 'BILL PAYMENT'
    05  TRAN-MERCHANT-CITY   PIC X(50).       *> 'N/A'
    05  TRAN-MERCHANT-ZIP    PIC X(10).       *> 'N/A'
    05  TRAN-CARD-NUM        PIC X(16).       *> From CXACAIX lookup
    05  TRAN-ORIG-TS         PIC X(26).       *> CICS ASKTIME timestamp
    05  TRAN-PROC-TS         PIC X(26).       *> Same as TRAN-ORIG-TS
    05  FILLER               PIC X(20).
Transaction Type Code: 02 is the hardcoded type code for bill payments. The merchant ID 999999999 and merchant name 'BILL PAYMENT' are internal sentinel values used to identify payment transactions vs. external purchase transactions.

7. Business Rules

Validation Rules

RuleConditionBehavior
Account ID Required ACTIDINI = SPACES OR LOW-VALUES Error: "Acct ID can NOT be empty..." — cursor positioned on ACTIDIN
Account Must Exist CICS READ returns NOTFND Error: "Account ID NOT found..." — no payment processed
Balance Must Be Positive ACCT-CURR-BAL <= ZEROS Error: "You have nothing to pay..." — no payment processed
Confirm Field Must Be Y or N Any value other than Y, y, N, n, SPACES Error: "Invalid value. Valid values are (Y/N)..." — cursor on CONFIRM
Payment Requires Explicit Y Confirmation CONFIRMI = 'N' or 'n' Screen cleared, message: "Confirm to make a bill payment..."
Duplicate Transaction ID CICS WRITE returns DUPKEY/DUPREC Error: "Tran ID already exist..." — payment not recorded

Payment Processing Rules

  • Payment amount equals ACCT-CURR-BAL — always full balance payment
  • Transaction ID is generated as: max(existing TRAN-ID) + 1 (browse from HIGH-VALUES with READPREV)
  • If the TRANSACT file is empty, transaction ID starts at 1 (READPREV returns ENDFILE → TRAN-ID set to ZEROS → ADD 1)
  • Account balance update: ACCT-CURR-BAL = ACCT-CURR-BAL - TRAN-AMT (results in zero after full payment)
  • Transaction type code is hardcoded to '02'
  • Transaction category code is hardcoded to 2
  • Source system is hardcoded to 'POS TERM'
  • Description is hardcoded to 'BILL PAYMENT - ONLINE'
  • Merchant ID is hardcoded to 999999999 (internal sentinel)
  • Both TRAN-ORIG-TS and TRAN-PROC-TS are set to the current CICS timestamp at time of payment

8. Processing Flow

First Interaction — Account Lookup

1
User navigates to CB00 from Main Menu. COBIL00C is invoked with COMMAREA. EIBCALEN > 0 and CDEMO-PGM-CONTEXT = 0 (first entry).
2
Program sets CDEMO-PGM-REENTER flag and sends the blank bill payment screen.
3
User types an 11-digit account ID into ACTIDIN and presses ENTER.
4
PROCESS-ENTER-KEY validates that ACTIDIN is not blank, then calls READ-ACCTDAT-FILE (READ UPDATE on ACCTDAT).
5
On successful read: ACCT-CURR-BAL is moved to WS-CURR-BAL and displayed in the CURBAL field. The confirm field prompt is shown.
6
If balance ≤ 0, error message is shown and processing stops.

Second Interaction — Payment Confirmation

7
User types Y in the confirmation field and presses ENTER.
8
READ-CXACAIX-FILE: Read CXACAIX alternate index by XREF-ACCT-ID to obtain XREF-CARD-NUM.
9
STARTBR-TRANSACT-FILE: Browse TRANSACT from HIGH-VALUES to position at the logical end.
10
READPREV-TRANSACT-FILE: Read backward one record to get the highest existing TRAN-ID. If ENDFILE (empty file), TRAN-ID is set to ZEROS.
11
ENDBR-TRANSACT-FILE: Release browse cursor.
12
New TRAN-ID = numeric value of last TRAN-ID + 1. TRAN-RECORD is INITIALIZED and populated with payment details (type 02, amount = current balance, card num, timestamps, etc.).
13
WRITE-TRANSACT-FILE: EXEC CICS WRITE to insert the new payment record.
14
On successful write: ACCT-CURR-BAL is decremented by TRAN-AMT, then UPDATE-ACCTDAT-FILE performs EXEC CICS REWRITE on ACCTDAT.
15
Success screen displayed: green message "Payment successful. Your Transaction ID is XXXXXXXXXXXXXXXX."
Concurrency Note: The ACCTDAT READ UPDATE acquires an implicit lock that is held until the REWRITE or until a subsequent CICS command releases it. In a high-volume environment, concurrent payments on the same account could result in contention. The base application does not implement explicit serialization beyond the CICS record-level lock.

10. Error Handling

All CICS commands in COBIL00C use the RESP / RESP2 pattern to capture the CICS response code rather than relying on CICS HANDLE CONDITION. Errors set WS-ERR-FLG = 'Y' and populate WS-MESSAGE before calling SEND-BILLPAY-SCREEN to display the error.

OperationCICS ConditionResponse
READ ACCTDAT DFHRESP(NORMAL) Continue normally
READ ACCTDAT DFHRESP(NOTFND) "Account ID NOT found..." error message
READ ACCTDAT Other DISPLAY RESP/REAS codes; "Unable to lookup Account..." error
REWRITE ACCTDAT DFHRESP(NORMAL) Continue normally
REWRITE ACCTDAT DFHRESP(NOTFND) "Account ID NOT found..." error message
REWRITE ACCTDAT Other DISPLAY RESP/REAS codes; "Unable to Update Account..." error
READ CXACAIX DFHRESP(NOTFND) "Account ID NOT found..." (no card cross-ref)
READ CXACAIX Other "Unable to lookup XREF AIX file..." error
STARTBR TRANSACT DFHRESP(NOTFND) "Transaction ID NOT found..." error
READPREV TRANSACT DFHRESP(ENDFILE) TRAN-ID set to ZEROS (file empty — start from 1)
WRITE TRANSACT DFHRESP(NORMAL) Success message with new Transaction ID
WRITE TRANSACT DFHRESP(DUPKEY/DUPREC) "Tran ID already exist..." error
WRITE TRANSACT Other "Unable to Add Bill pay Transaction..." error
Gap in Error Recovery: When the REWRITE ACCTDAT fails after a successful WRITE TRANSACT, the transaction record has already been committed but the account balance has not been updated. The base application does not implement compensating transactions or two-phase commit for this scenario. In a production modernization, this would require either a single-resource atomic update or a saga/compensating pattern.

12. Development Guidelines & Patterns

CICS Programming Patterns Used

  • RESP/RESP2 error handling: All EXEC CICS commands use RESP(WS-RESP-CD) RESP2(WS-REAS-CD) and EVALUATE for condition checking. Do not use CICS HANDLE CONDITION in new code.
  • COMMAREA state: Program state between screen interactions (re-entry flag, account ID, etc.) is maintained solely in the CICS COMMAREA. Do not use CICS temporary storage or global variables for state.
  • BMS ERASE: The SEND MAP command uses ERASE to clear the 3270 buffer on each send, ensuring a clean screen.
  • UPDATE mode READ: The ACCTDAT file is read with UPDATE to hold the record lock until the subsequent REWRITE. Always pair READ UPDATE with REWRITE.
  • Browse pattern for sequential ID: STARTBR from HIGH-VALUES + READPREV gives the highest key — a simple sequential ID generation strategy without a separate counter file.
  • EIBCALEN guard: The program checks EIBCALEN = 0 to handle the case where it is invoked without a COMMAREA (e.g., direct transaction entry) — it redirects to signon.

Extending Bill Payment

  • To support partial payment amounts: Add a new input field to the BMS map for amount entry, validate amount > 0 and amount ≤ ACCT-CURR-BAL, and use the entered amount instead of ACCT-CURR-BAL for TRAN-AMT.
  • To add payment history display: After successful payment, XCTL to COTRN00C (CT00) with the account ID to show recent transactions.
  • To add EFT account display: Use CVCUS01Y and the customer file (CUSTDAT) to show CUST-EFT-ACCOUNT-ID on the payment screen.
  • To improve transaction ID generation: Replace the browse-from-max approach with a dedicated counter in a VSAM RRDS or DB2 sequence to avoid potential race conditions under concurrency.

Testing Checklist

  • ✅ Test with valid account ID and positive balance → balance displayed correctly
  • ✅ Test confirmation with Y → payment recorded, balance updated to zero
  • ✅ Test confirmation with N → screen cleared, no payment
  • ✅ Test with account ID not in ACCTDAT → "Account ID NOT found" error
  • ✅ Test with account with zero or negative balance → "You have nothing to pay" error
  • ✅ Test PF3 → returns to calling program
  • ✅ Test PF4 → screen cleared
  • ✅ Test with blank account ID → "Acct ID can NOT be empty" error
  • ✅ Test with invalid confirm value (e.g., 'Z') → validation error
  • ✅ Verify transaction record written with type='02', correct amount, and card number from CXACAIX
  • ✅ Verify ACCTDAT ACCT-CURR-BAL decremented to zero after payment

13. User Stories

Core User Stories

IDStoryComplexity
BP-01 As a cardholder, I want to view my current outstanding balance on the bill payment screen so I know how much I owe before confirming payment. Simple (1–2)
BP-02 As a cardholder, I want to pay my full outstanding balance by entering my account ID and confirming with Y so my balance is reduced to zero. Medium (3–5)
BP-03 As a cardholder, I want to receive a transaction ID after successful payment so I have a reference for the payment record. Simple (1–2)
BP-04 As a cardholder, I want to be protected from accidental payments by an explicit Y/N confirmation prompt so I don't pay by mistake. Simple (1–2)
BP-05 As a cardholder, I want to see an informative error message when my account is not found or has no balance so I understand why payment was not processed. Simple (1–2)

Acceptance Criteria Template (BP-02)

GIVEN I am signed in as a regular user (type U)
  AND I navigate to transaction CB00 (Bill Payment)
  AND my account ACCTDAT record exists with ACCT-CURR-BAL > 0

WHEN I enter a valid 11-digit account ID in the ACTIDIN field
  AND I press ENTER
THEN the screen displays my current balance in the CURBAL field
  AND a confirmation prompt appears

WHEN I enter 'Y' in the CONFIRM field
  AND I press ENTER
THEN a TRAN-RECORD with TRAN-TYPE-CD='02' is written to TRANSACT VSAM
  AND ACCTDAT.ACCT-CURR-BAL is decremented by the payment amount
  AND a success message displays the new Transaction ID
  AND the success message is displayed in GREEN on the ERRMSG field

14. Code Examples

Payment Transaction Record Population (from COBIL00C)

      *--- After confirming payment (CONF-PAY-YES = TRUE) ---*

           PERFORM READ-CXACAIX-FILE          *> Get card number
           MOVE HIGH-VALUES TO TRAN-ID
           PERFORM STARTBR-TRANSACT-FILE      *> Position at end
           PERFORM READPREV-TRANSACT-FILE     *> Get highest TRAN-ID
           PERFORM ENDBR-TRANSACT-FILE        *> Release cursor

           MOVE TRAN-ID     TO WS-TRAN-ID-NUM *> Convert to numeric
           ADD 1 TO WS-TRAN-ID-NUM            *> Increment
           INITIALIZE TRAN-RECORD
           MOVE WS-TRAN-ID-NUM       TO TRAN-ID
           MOVE '02'                 TO TRAN-TYPE-CD
           MOVE 2                    TO TRAN-CAT-CD
           MOVE 'POS TERM'           TO TRAN-SOURCE
           MOVE 'BILL PAYMENT - ONLINE' TO TRAN-DESC
           MOVE ACCT-CURR-BAL        TO TRAN-AMT
           MOVE XREF-CARD-NUM        TO TRAN-CARD-NUM
           MOVE 999999999            TO TRAN-MERCHANT-ID
           MOVE 'BILL PAYMENT'       TO TRAN-MERCHANT-NAME

ACCTDAT READ UPDATE Pattern

       READ-ACCTDAT-FILE.

           EXEC CICS READ
                DATASET   (WS-ACCTDAT-FILE)
                INTO      (ACCOUNT-RECORD)
                LENGTH    (LENGTH OF ACCOUNT-RECORD)
                RIDFLD    (ACCT-ID)
                KEYLENGTH (LENGTH OF ACCT-ID)
                UPDATE                        *> Acquires record lock
                RESP      (WS-RESP-CD)
                RESP2     (WS-REAS-CD)
           END-EXEC

           EVALUATE WS-RESP-CD
               WHEN DFHRESP(NORMAL)
                   CONTINUE
               WHEN DFHRESP(NOTFND)
                   MOVE 'Y'     TO WS-ERR-FLG
                   MOVE 'Account ID NOT found...' TO WS-MESSAGE
                   MOVE -1       TO ACTIDINL OF COBIL0AI
                   PERFORM SEND-BILLPAY-SCREEN
               WHEN OTHER
                   DISPLAY 'RESP:' WS-RESP-CD 'REAS:' WS-REAS-CD
                   MOVE 'Y'     TO WS-ERR-FLG
                   MOVE 'Unable to lookup Account...' TO WS-MESSAGE
                   PERFORM SEND-BILLPAY-SCREEN
           END-EVALUATE.

Timestamp Generation Pattern

       GET-CURRENT-TIMESTAMP.

           EXEC CICS ASKTIME
             ABSTIME(WS-ABS-TIME)
           END-EXEC

           EXEC CICS FORMATTIME
             ABSTIME(WS-ABS-TIME)
             YYYYMMDD(WS-CUR-DATE-X10)
             DATESEP('-')
             TIME(WS-CUR-TIME-X08)
             TIMESEP(':')
           END-EXEC

           INITIALIZE WS-TIMESTAMP
           MOVE WS-CUR-DATE-X10 TO WS-TIMESTAMP(01:10)  *> YYYY-MM-DD
           MOVE WS-CUR-TIME-X08 TO WS-TIMESTAMP(12:08)  *> HH:MM:SS
           MOVE ZEROS           TO WS-TIMESTAMP-TM-MS6.
