core-platform

Base CardDemo module for online CICS card/account flows and nightly batch orchestration. This page documents the module from repository sources in app/cbl, app/bms, app/cpy, app/jcl, app/csd, and app/scheduler.

COBOL + CICS BMS 3270 VSAM KSDS/AIX JCL Batch CA7 / Control-M Assembler Utilities

Module Overview and Purpose

core-platform provides the mandatory runtime for CardDemo. Optional modules (authorization, DB2 transaction-type maintenance, MQ extractions) depend on this module’s CICS sign-on, menu routing, data contracts, and batch lifecycle.

Business objective: support complete cardholder and admin workflows over VSAM records with stable, auditable online and overnight processing.

Primary User Flows

  • Sign-on and role routing via CC00 and COSGN00C.
  • User main menu via CM00 and COMEN01C.
  • Account view/update via CAVW/CAUP.
  • Card list/view/update via CCLI/CCDL/CCUP.
  • Transaction list/view/add via CT00/CT01/CT02.
  • Bill payment via CB00.
  • Report submission via CR00.

Primary Data Domains

  • User security records (USRSEC).
  • Account master records (ACCTDAT).
  • Card records (CARDDAT and AIX path CARDAIX).
  • Card/account cross-reference (CCXREF, CXACAIX).
  • Transaction master (TRANSACT + AIX via TRANIDX job).

Architectural Context

Runtime pattern is synchronous terminal interaction in CICS + asynchronous nightly batch. CICS handles validation and point updates. Batch transforms daily feeds, applies balances, computes interest, combines system-generated records, and rebuilds indexes.

Online Layer

  • BMS mapsets define field-level contracts and PF-key prompts.
  • COBOL programs parse map input and call EXEC CICS READ/WRITE/REWRITE/XCTL.
  • CARDDEMO-COMMAREA propagates identity and navigation context.

Batch Layer

  • JCL jobs execute business batch COBOL and IDCAMS/SORT utilities.
  • Open/close file control around CICS via SDSF command jobs.
  • Scheduler definitions model daily/weekly/monthly dependencies.

Control Contracts

  • CICS resource definitions in app/csd/CARDDEMO.CSD.
  • CA7 chains in app/scheduler/CardDemo.ca7.
  • Control-M folders and conditions in app/scheduler/CardDemo.controlm.

Business Rules Extracted From Code

  • Authentication rule: COSGN00C requires non-empty user ID/password and exact password match in USRSEC.
  • Role routing rule: SEC-USR-TYPE='A' routes to admin menu; non-admin routes to user menu.
  • Menu validation rule: menu options must be numeric and within configured copybook count.
  • Cross-reference rule: account/card operations rely on CXACAIX or CCXREF lookups.
  • Transaction add rule: account or card must be supplied; numeric format enforced for key fields.
  • Date format rule: CT02 requires YYYY-MM-DD and validates through CSUTLDTC.
  • Amount format rule: CT02 amount must follow signed -99999999.99 style.
  • Bill payment rule: payment requires positive current balance and explicit confirmation Y.
  • Bill posting rule: on payment confirmation, a type/category transaction is written and account balance is rewritten.
  • Pagination rule: CT00 uses browse cursor semantics with PF7/PF8 boundaries.
  • Error display rule: online errors set map cursor fields (e.g., -1 length/cursor controls).
  • Batch consistency rule: close CICS files before batch mutations and reopen after completion.
  • Index rule: TRANIDX rebuilds AIX to preserve alternate access performance.

Operational Environment Assumptions

  • HLQ (example AWS.M2) must be aligned in CSD and JCL members.
  • Batch job libraries must expose AWS.M2.CARDDEMO.LOADLIB.
  • CICS region command target (e.g., CICSAWSA) must match SDSF JCL command streams.
  • Scheduler calendars and conditions must be loaded before production scheduling.
  • FTP JES submission scripts assume reachable tunnel endpoint on localhost:2121.

Development Guidelines and Patterns

CICS Program Pattern

  1. Initialize error flags/message and receive COMMAREA.
  2. For first-entry vs re-entry, branch on CDEMO-PGM-REENTER.
  3. Receive BMS map and interpret AID key (DFHENTER, PF keys).
  4. Run field validations; set cursor with -1 where needed.
  5. Execute VSAM operations with explicit response handling.
  6. Send map with message area and return with COMMAREA.

Validation Conventions

  • Use explicit numeric checks before moving into numeric storage.
  • Use EVALUATE TRUE branches for ordered required-field checks.
  • Emit user-focused message strings in map error area.
  • Prefer centralized field-edit paragraphs (e.g., 1210-EDIT-ACCOUNT).
  • Reject invalid PF key combinations with immediate redisplay.

Data Integrity Conventions

  • Cross-entity operations validate through XREF before touching master record.
  • Payment/transaction flows allocate next transaction ID from highest key + 1.
  • Batch pipeline should close CICS files before destructive IDCAMS/REPRO steps.
  • Rebuild alternate indexes after bulk transaction replacement jobs.
  • Avoid partial updates: write transaction + account rewrite as a logical unit in flow.

UI Pattern Conventions

  • Titles and timestamps are consistently populated by header paragraphs.
  • PF key hints are embedded in map bottom row constants.
  • Input fields use underline/highlight to indicate editable zones.
  • Messages use shared map area field ERRMSG.
  • Menus use copybook-driven option arrays to reduce hardcoded branches.

Batch Pattern Conventions

  • Use explicit utility step ordering (IDCAMS delete/define then repro/build).
  • Create GDG versions for backups and intermediate state snapshots.
  • Use scheduler dependencies rather than manual timing where possible.
  • Retain a controllable wait step (WAITSTEP) for file-state stabilization windows.

Appendix: Quick Checks

  • Verify CICS file states before and after batch windows (CLOSEFIL/OPENFIL).
  • Verify transaction list pagination boundaries and message text on top/bottom page limits.
  • Verify CT02 date validation rejects malformed and invalid calendar dates.
  • Verify CB00 blocks payment when current balance is zero or negative.
  • Verify TRANIDX run after COMBTRAN when transaction master is rebuilt.
  • Verify scheduler out conditions are reset as designed in Control-M folders.
