📦 core-online

Guide for Developing User Stories on the CICS 3270 front-end.

📋 Overview

Purpose: Provide the live CICS screens that let Regular Users and Admins manage accounts, credit cards, transactions, payments, and user records.

Business Context: These screens sit at the center of the CardDemo experience, surfacing VSAM-backed data while respecting job-driven refreshes and RACF security policies.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Visibility: As a cardholder, I want to see my account balance and APR on CAVW so I can know what I owe before the statement posts.
  • Admin Control: As an Admin, I want to add a new user via CU01 so that help desk staff gain access quickly.
  • Transaction Audit: As an Ops lead, I want to add a transaction in CT02 and verify its status in CT00 so that we can reproduce customer requests.

⚡ Development Acceleration Factors

  • Shared copybooks: CVACT01Y, CVTRA06Y define record layouts reused by every screen.
  • BMS templates: Map sets like COMEN01 and COACTVW can be cloned to build new menus with existing PF key lines.
  • VSAM files: The datasets listed in app/catlg/LISTCAT.txt already expose the fields your screens need.

📊 Complexity Guidelines

  • Simple (1-2 pts): Update text or add message lines on an existing map (e.g., modify the PF1 help line on CC00).
  • Medium (3-5 pts): Create a new screen that combines existing copybooks and VSAM fields but reuses the standard PF key layout.
  • Complex (5-8 pts): Add a new list/detail screen that links into optional modules (e.g., show MQ authorization details with DB2 joins) while handling stateful navigation.

✅ Acceptance Criteria Examples

  • Every new screen must display the expected VSAM field values within 2 seconds and show PF hints on line 24.
  • Navigation between menus must use existing PF keys (PF3 to exit, PF5 for detail, PF7/PF8 for paging).
  • All inputs pass `88`-level validation before issuing an `EXEC CICS SEND MAPSET`.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs under app/cbl/ (e.g., COACTVW, COCRDUP, COTRN00).
  • BMS map sets under app/bms/ (COSGN00, COMEN01, COACTVW, etc.).
  • Copybooks in app/cpy/ and app/cpy-bms/ ensure consistent layouts across screens and batch jobs.
  • CICS resource definitions in app/csd/ that wire transactions to programs and VSAM files.

🔗 Dependencies

  • Depends on VSAM datasets refreshed by core-batch.
  • Can surface data from optional modules (authorization-ims-db2-mq, transaction-type-db2, vsam-mq-extraction).
  • RACF entries must exist for each new transaction id.

🌐 Public Interfaces

CC00 (Signon), CM00 (Main Menu), CAVW/CAUP (Account view/update), CCLI/CCDL/CCUP (Card list/details), CT00/CT01/CT02 (Transaction list/detail/add), CR00 (Reports), CB00 (Bill payments), CA00 (Admin menu), CU00-03 (User management)
