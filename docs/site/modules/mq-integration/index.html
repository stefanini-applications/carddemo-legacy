📨 MQ Integration

User Story Development Guide · CardDemo Mainframe Application

Optional Module CDRD · CDRA · IBM MQ · VSAM

📋 Overview

The MQ Integration module is an optional extension demonstrating asynchronous request/response patterns between distributed systems and the CardDemo mainframe. It provides two CICS programs that expose mainframe data via IBM MQ, enabling external cloud or distributed applications to query mainframe data without direct CICS coupling.

CICS Programs

Transaction IDProgramFunction
CDRDCODATE01Date query — receives MQ request for system date, returns current date as response
CDRACOACCT01Account query — receives MQ request with account number, returns account data from VSAM

MQ Queue Configuration

Queue NameDirectionDescription
CARDDEMO.REQUEST.QUEUEInboundReceives requests from external systems
CARDDEMO.RESPONSE.QUEUEOutboundSends responses back to requesting systems

CICS resources: MQCONN(MQ01), MQQUEUE(CARDREQ) for request queue, MQQUEUE(CARDRES) for response queue.

Business Context

This module enables distributed and cloud systems to query mainframe data via MQ without requiring direct CICS connectivity or knowledge of VSAM internals. CDRD demonstrates the simplest possible MQ request/response pattern (date query). CDRA extends that pattern with VSAM data extraction — making it a building block for exposing any mainframe entity as a service. Both programs demonstrate proper message correlation using REQUEST-ID to match responses to requests.

Story Templates

External System Queries System Date via MQ
As an external system, I want to query the mainframe system date via MQ so that I can synchronize time-sensitive operations.

CODATE01 (CDRD): MQGET from CARDREQ queue; check REQUEST-TYPE='DATE'; get ASKTIME/FORMATTIME; MQPUT response with REQUEST-ID correlation and SYSTEM-DATE to CARDRES queue
External System Retrieves Account Balance via MQ
As an external banking system, I want to retrieve account data for a given account number via MQ so that I can display balance information in a mobile app without direct VSAM access.

COACCT01 (CDRA): MQGET request with ACCOUNT-NUMBER; VSAM keyed read on ACCTDATA using ACCOUNT-NUMBER; MQPUT response with REQUEST-ID + ACCOUNT-DATA (full 300-byte CVACT01Y record)
Modernization Architect Exposes VSAM Data as MQ Service
As a modernization architect, I want to expose credit card account data as a queued service so that new cloud microservices can consume mainframe data without tight coupling.

COACCT01 is the reference implementation — MQ request/response + VSAM read + message correlation; extend this pattern for cards (CVACT02Y), transactions (CVTRA05Y)
Integration Tester Verifies MQ Request/Response Correlation
As an integration tester, I want to verify that MQ responses are correctly correlated to their requests so that I can confirm the service is reliable under concurrent load.

Send multiple requests with unique REQUEST-IDs; verify each response contains the matching REQUEST-ID; test with CODATE01 (simple) before testing COACCT01 (complex)

Integration Patterns

  • Request/response over async MQ — sender puts to REQUEST queue; receiver (CICS) processes and puts to RESPONSE queue; sender polls RESPONSE queue for matching REQUEST-ID
  • Message correlation by REQUEST-ID — 8-char REQUEST-ID is echoed in every response; external systems use this to match responses to pending requests
  • VSAM account data extraction — COACCT01 extracts full 300-byte CVACT01Y record into MQ message payload; consumer maps fields from the record layout
  • MQ timeout handling — MQGET with WAIT option; timeout returns MQRC_NO_MSG_AVAILABLE; CICS transaction ends cleanly

Acceleration Factors

  • CODATE01 simple MQPUT/MQGET pattern — simplest possible MQ request/response; ideal starting point for new MQ service stories
  • COACCT01 VSAM + MQ pattern — adds VSAM keyed read before MQPUT response; template for any "expose VSAM entity via MQ" story
  • Message correlation pattern — REQUEST-ID echo in response is reusable across all new MQ service programs
  • CICS MQCONN resource definitions — MQ01, CARDREQ, CARDRES already defined; new queues follow same DEFINE QLOCAL pattern

COACCT01 Processing Flow

COACCT01 (CDRA) Account Query Flow:
1. EXEC CICS MQGET QUEUE(CARDREQ) INTO(REQUEST-MSG) WAIT(5000)
2. Check MQRC — if NO_MSG_AVAILABLE, end transaction normally
3. Extract REQUEST-ID and ACCOUNT-NUMBER from REQUEST-MSG
4. Validate REQUEST-TYPE = 'ACCT'
5. READ ACCTDATA VSAM keyed on ACCOUNT-NUMBER
6. If NOT FOUND: build error response, MQPUT to CARDRES, end
7. If FOUND: move ACCOUNT-RECORD (300 bytes) to ACCOUNT-DATA
8. Build response: REQUEST-ID + ACCOUNT-DATA
9. EXEC CICS MQPUT QUEUE(CARDRES) FROM(RESPONSE-MSG)
10. EXEC CICS RETURN

MQ Queue Setup (DEFINE QLOCAL)

* MQ queue definitions (run in MQ admin):
DEFINE QLOCAL('CARDDEMO.REQUEST.QUEUE')  +
    DEFPSIST(YES) MAXDEPTH(5000)

DEFINE QLOCAL('CARDDEMO.RESPONSE.QUEUE') +
    DEFPSIST(NO) MAXDEPTH(5000)

* CICS resource definitions:
MQCONN(MQ01)   MQMANAGER(QM01) STATUS(ENABLED)
MQQUEUE(CARDREQ) QUEUE('CARDDEMO.REQUEST.QUEUE')
MQQUEUE(CARDRES) QUEUE('CARDDEMO.RESPONSE.QUEUE')

✅ Acceptance Criteria Patterns

Date Query (CDRD / CODATE01)

  • GIVEN a date request on the MQ queue with REQUEST-TYPE='DATE', WHEN CODATE01 processes it, THEN a response is placed on CARDDEMO.RESPONSE.QUEUE with the current system date in YYYY-MM-DD format within the configured timeout
  • GIVEN a date request with REQUEST-ID='REQ00001', WHEN CODATE01 sends the response, THEN the response message contains REQUEST-ID='REQ00001' for correlation
  • GIVEN no messages on the request queue, WHEN CODATE01 executes, THEN the transaction ends normally with no error after the MQWAIT timeout expires

Account Query (CDRA / COACCT01)

  • GIVEN a valid account number in the MQ request, WHEN COACCT01 processes it, THEN the response contains the 300-byte account data from ACCTDATA VSAM with matching REQUEST-ID
  • GIVEN an account number not found in ACCTDATA VSAM, WHEN COACCT01 processes the request, THEN an error response is placed on the response queue indicating the account was not found
  • GIVEN multiple concurrent requests with different REQUEST-IDs, WHEN COACCT01 processes and responds, THEN each response message contains the correct REQUEST-ID matching its originating request

Error Handling

  • GIVEN an MQ timeout (no message within wait interval), WHEN COACCT01 or CODATE01 detects the timeout, THEN the CICS transaction ends cleanly with no abend and no orphaned resources
  • GIVEN an MQPUT failure on the response queue, WHEN the program handles the MQRC, THEN the error is logged to CICS CSSL and the transaction rolls back cleanly
