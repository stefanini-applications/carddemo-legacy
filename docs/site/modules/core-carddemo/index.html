📦 Core CardDemo

Guide for developing user stories that keep the foundational COBOL/CICS application stable and extensible.

📋 Overview

Purpose: Deliver credit card management flows (signon, account/card/transaction CRUD, billing, statements) that anchor all other optional services.

Business Context: This module is the heart of every modernization demo—the datasets and programs in `app/cbl`, `app/jcl`, `app/bms`, and `app/data` must remain consistent so downstream MQ and DB2 stories can rely on accurate account/transaction data.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Journeys: As a cardholder, I want to review my transactions via `CT01` so that I can confirm my activity before paying the bill.
  • Batch Operations: As an operator, I want the `POSTTRAN` job to finish with no VSAM errors so that subsequent statements (`CREASTMT`) reflect the latest data.

⚡ Development Acceleration Factors

  • Shared copybooks: `app/cpy` and `app/cpy-bms` copybooks define record layouts reused by forms like `COACTUPC` and lists such as `COUSR00`. Update once, apply multiple screens.
  • BMS map set reuse: Screens follow the same 3270 pattern of PF keys (PF1 help, PF3 exit, PF5 search) defined in `app/bms`. Lean on `COSGN00`, `COACTVW`, and `COCRDS` layouts when adding new fields.

📊 Complexity Guidelines

  • Simple (1-2 pts): Text changes on `CCLI` or adding validation labels through existing maps.
  • Medium (3-5 pts): Insert new fields into `COACTVWC` or `CT02` with PIC validation and dataset persistence.
  • Complex (5-8 pts): Introduce new batch job or dataset (e.g., new `TRANEXTR` variant) that touches VSAM, JCL, and execution order.

🔧 Technical Foundation

🏗️ Key Components

  • Programs: `COSGN00C`, `COMEN01C`, `COACTVWC`, `CT00`/`CT01`/`CT02`, `CBSTM03A/B` for statement generation.
  • BMS Maps: Mapsets such as `COSGN00`, `COCRDSL`, `COACTUP`, `COUSR00` rendered via PF keys.
  • Copybooks: `CMSTACT0`, `CVACT02Y`, `CVTRA06Y`, `CSUTLDTC` define shared structures for accounts, cards, and transactions.

🔗 Dependencies

  • CICS transaction server, RACF user database, and VSAM KSDS files located in `app/data`.
  • Batch orchestration with JCL under `app/jcl` and scheduler job decks (`app/scheduler`).
  • Optional MQ/DB2/IMS modules depend on this foundation; keep dataset layouts stable.

🌐 Public Interfaces


CICS Transactions:
  CC00 - Signon (COSGN00C)
  CM00 - Main menu (COMEN01C)
  CAUP/CCUP - Account/card updates
  CT00-CT02 - List/add transactions
  CR00 - Transaction reporting
  CB00 - Bill payment
  CU00-CU03 - User maintenance
Batch jobs:
  POSTTRAN, TRANEXTR, CREASTMT, TRANIDX, OPENFIL, CLOSEFIL, COMBTRAN, DUSRSECJ

✅ Acceptance Criteria Examples

  • Updates on `COACTVWC` persist to VSAM and show the new value when `CAVW` refreshes.
  • `POSTTRAN` must finish without abends so `CREASTMT` can start with the latest transaction file.
  • CU01 user creation requires RACF credentials and updates `app/data/USRSEC` dataset.
