💳 Account Management Module

User Story Development Guide — Account View & Update

📋 Overview

Purpose: Allows authenticated users to view and update their credit card account details including current balance, credit limits, dates, and billing zip code.

Business Context: Central to the cardholder experience. Account data is the source of truth for balance management, credit availability, and billing. Read-only view (CAVW) allows any authenticated user; update (CAUP) allows modification of specific fields.

CICS Transactions: CAVW (Account View), CAUP (Account Update)

VSAM Dataset: AWS.M2.CARDDEMO.ACCTDATA.PS (KSDS, 300-byte records, keyed by ACCT-ID)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account View: As a cardholder, I want to view my current account balance and credit limit so that I can plan my spending decisions.
  • Account Update: As a cardholder, I want to update my billing zip code so that my billing address information stays accurate.
  • Credit Info: As a cardholder, I want to view my cash credit limit separately from my purchase credit limit so that I know my cash advance availability.
  • Statement Dates: As a cardholder, I want to see my account open date and expiration date so that I can track my account history.

⚡ Development Acceleration Factors

  • CVACT01Y copybook: Pre-defined 300-byte account record — all fields readily accessible
  • Keyed VSAM access: Direct lookup by ACCT-ID (11 digits) — O(1) retrieval
  • Existing screen layouts: COACTUP.bms and COACTVW.bms provide complete screen definitions
  • Customer data linkage: CVCUS01Y provides customer name/address when cross-referenced with account

📊 Complexity Guidelines

  • 1–2 pts View account details (CAVW — read VSAM, display on screen)
  • 3–5 pts Update account fields with validation (zip, limits); retrieve linked customer info
  • 5–8 pts Multi-account aggregation, credit availability calculation across accounts, audit trail to DB2

🗄️ Account Record Key Fields

01  ACCOUNT-RECORD.
    05  ACCT-ID              PIC 9(11).     -- Primary key (11 digits)
    05  ACCT-ACTIVE-STATUS   PIC X(01).     -- 'Y' = active
    05  ACCT-CURR-BAL        PIC S9(10)V99. -- Current balance
    05  ACCT-CREDIT-LIMIT    PIC S9(10)V99. -- Credit limit
    05  ACCT-CASH-CREDIT-LIMIT PIC S9(10)V99. -- Cash credit limit
    05  ACCT-OPEN-DATE       PIC X(10).     -- YYYY-MM-DD
    05  ACCT-EXPIRAION-DATE  PIC X(10).     -- YYYY-MM-DD
    05  ACCT-REISSUE-DATE    PIC X(10).     -- YYYY-MM-DD
    05  ACCT-CURR-CYC-CREDIT PIC S9(10)V99. -- Cycle credits
    05  ACCT-CURR-CYC-DEBIT  PIC S9(10)V99. -- Cycle debits
    05  ACCT-ADDR-ZIP        PIC X(10).     -- Billing zip (updatable)
    05  ACCT-GROUP-ID        PIC X(10).     -- Disclosure group ID

🔗 Dependencies

  • signon module: Must be authenticated; ACCT-ID passed via COMMAREA
  • card-management module: Cards reference accounts via CARD-ACCT-ID
  • batch-processing module: POSTTRAN and INTCALC update account balances
  • VSAM ACCTDATA file: Must be open in CICS (JCL: OPENFIL)

✅ Acceptance Criteria Patterns

  • Given a valid account ID, when user accesses CAVW, then all account fields are displayed correctly formatted
  • Given account status = 'N' (inactive), when user attempts update (CAUP), then system displays appropriate status message
  • Given new credit limit < current balance, when user submits update, then system rejects with validation error
  • Given new cash limit > credit limit, when user submits update, then system rejects with validation error
  • Given valid zip code update, when submitted, then VSAM record is rewritten and confirmation message displayed
  • Given VSAM I/O error during update, then system displays file status error code and rolls back change

🔢 Business Rules Summary

  • Account active status must be Y to allow updates
  • Credit limit ≥ current balance at all times
  • Cash credit limit ≤ credit limit
  • All monetary fields use COMP-3 (packed decimal) format internally
  • Date fields stored as YYYY-MM-DD (10 chars)
  • ACCT-GROUP-ID links to disclosure groups for interest rate determination (used in batch)
