🏦 Accounts

User Story Development Guide · CardDemo Mainframe Application

Core Module CAVW · CAUP · CBACT01-04C

📋 Overview

The Accounts module provides online viewing and updating of credit card account information, backed by nightly batch processing for interest calculations and account maintenance. It is the central entity of the CardDemo application — all other modules reference account records.

Data Store

Account records are stored in the ACCTDATA VSAM KSDS (AWS.M2.CARDDEMO.ACCTDATA.PS). Each record is 300 bytes keyed on ACCT-ID (11 digits), defined in copybook CVACT01Y.

Business Context

Accounts are the central entity in CardDemo. They are linked to:

  • Credit Cards — via CARDXREF VSAM (CVACT03Y), enabling lookup by account or card number
  • Customers — via CUSTDATA VSAM (customer demographic data)
  • Transactions — TRANSACT VSAM KSDS references ACCT-ID on every transaction record

Interest is calculated nightly by CBACT04C. The program reads each account's ACCT-GROUP-ID, looks up the applicable interest rate in the DISCGRP file, and updates balances in TCATBALF.

🎯 User Story Development Guide

Story Templates

View Account Balance
As a cardholder, I want to view my current account balance and credit limit so that I can manage my spending.

Key fields: ACCT-CURR-BAL, ACCT-CREDIT-LIMIT, ACCT-CURR-CYC-DEBIT, ACCT-CURR-CYC-CREDIT
Update Account ZIP / Credit Limit
As a customer service representative, I want to update a cardholder's billing ZIP code and credit limit so that account information stays current.

Program: COACTUPC (CAUP) — reads current record, applies changes, rewrites to ACCTDATA VSAM
Batch Interest Calculation
As a batch operations manager, I want the nightly CBACT04C job to calculate and apply interest to all outstanding balances so that accounts reflect accurate charges each morning.

Input: ACCTDATA + DISCGRP; Output: updated ACCTDATA + TCATBALF entries
Account Status Management
As a risk officer, I want to set an account status to inactive so that no further transactions are authorized against it.

Field: ACCT-ACTIVE-STATUS PIC X(1) — controls transaction eligibility checks in COTRN02C

Acceleration Factors

  • CVACT01Y copybook — fully defined 300-byte account record; no data modeling required
  • COACTVWC read pattern — keyed VSAM read with ACCT-ID is directly reusable for any account display story
  • COACTUPC update pattern — read-for-update → modify → rewrite pattern is reusable for any account field update story
  • CBACT01-03C — provide sequential read, keyed read, and update patterns for batch account stories

✅ Acceptance Criteria Patterns

Account View (CAVW)

  • GIVEN an authenticated user with a valid account ID, WHEN CAVW is invoked, THEN account balance and credit limit are displayed correctly from ACCTDATA VSAM
  • GIVEN an invalid account ID, WHEN CAVW is invoked, THEN an appropriate error message is displayed and no account data is shown
  • GIVEN an unauthenticated session, WHEN CAVW is invoked, THEN the user is redirected to the sign-on screen

Account Update (CAUP)

  • GIVEN a valid update request, WHEN CAUP processes the change, THEN the updated fields are written back to ACCTDATA VSAM and a confirmation message is displayed
  • GIVEN a credit limit reduction below the current balance, WHEN CAUP processes the change, THEN the update is rejected with a validation error message

Batch Interest Calculation (CBACT04C)

  • GIVEN accounts with outstanding balances, WHEN CBACT04C runs, THEN interest is calculated using the correct rate from DISCGRP for each account's group ID
  • GIVEN an account with zero balance, WHEN CBACT04C runs, THEN no interest charge is applied
  • GIVEN a batch run, WHEN CBACT04C completes, THEN return code is 0 and all updated records are committed to ACCTDATA VSAM
