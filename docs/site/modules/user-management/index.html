👥 User Management Module

CardDemo — User Story Development Guide

📋 Overview

Purpose: Admin-only CICS transactions for listing, adding, updating, and deleting application users in the USRSEC VSAM file.

Business Context: User management controls who can access CardDemo and in what capacity (Admin vs Regular User). All four CRUD operations are available to admin users via the Admin Menu.

Entry: Admin Menu (CA00) → CU00–CU03

Access Restriction: Requires CDEMO-USER-TYPE = 'A' (Admin) in COMMAREA

🎯 User Story Development Guide

📝 US Templates

  • As an admin, I want to list all users so that I can audit who has access to the system
  • As an admin, I want to add a new user with an ID, password, and type so that new employees can access CardDemo
  • As an admin, I want to update a user's password or user type so that access permissions reflect current roles
  • As an admin, I want to delete a user so that departed employees lose access immediately
  • As a security manager, I want non-admin users to be blocked from the Admin Menu so that administrative functions are protected

⚡ Development Acceleration Factors

  • CSUSR01Y copybook: User record structure pre-defined
  • USRSEC VSAM: Access pattern established in COSGN00C (authentication)
  • BMS maps: COUSR00–COUSR03 all pre-defined; CRUD pattern consistent
  • DUSRSECJ JCL: Initial user file load pattern available as reference

📊 Complexity Guidelines

  • 1–2 pts List Users (CU00) — browse USRSEC
  • 3–5 pts Add/Update/Delete user (CU01–CU03) — VSAM write operations with validation
  • 5–8 pts User management with RACF integration; password policy enforcement; audit logging

📊 User Record Fields

User ID             PIC X(08)  -- Primary key (e.g., USER0001, ADMIN001)
User Password       PIC X(08)  -- Stored password
User Type           PIC X(01)  -- 'A' = Admin, 'U' = Regular User
(Additional fields per CSUSR01Y copybook)

✅ Acceptance Criteria Examples

  • Given regular user attempts to access CA00 → Access denied, redirected to main menu with error
  • Given admin user on CU00 → All users displayed with User ID and user type
  • Given valid new user data on CU01 → User record written to USRSEC, confirmation displayed
  • Given duplicate User ID on CU01 → Error message shown, no write to USRSEC
  • Given valid update on CU02 → USRSEC record updated, new password effective on next login
  • Given valid user ID on CU03 → User deleted from USRSEC, cannot log in after deletion
  • Given non-existent user ID on CU02/CU03 → Error message, no VSAM update
