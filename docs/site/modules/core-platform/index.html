core-platform

Base CardDemo module for sign-on, account and card maintenance, transaction browsing and posting, bill payments, report submission, and nightly batch orchestration over VSAM data sets.

CICS COBOL BMS VSAM KSDS/AIX JCL CA7 Control-M RACF-aligned security model

Module Overview And Purpose

The core-platform module is the mandatory runtime baseline for CardDemo. It provides interactive CICS screens for customer operations, VSAM-backed persistence contracts, and the scheduled batch jobs that maintain statements, balances, and transaction lifecycle files.

This module is also the dependency anchor for optional extensions: authorization, transaction-type-management, and mq-account-extractions. Even when those extensions are disabled, core-platform must remain fully functional for sign-on and day-to-day card/account activities.

Runtime baseline from code: app/csd/CARDDEMO.CSD binds online transactions (CC00, CM00, CAUP, CCUP, CT00, CT01, CT02, CB00, CR00) to specific COBOL programs and VSAM file resources.

Architectural Context

Online Path

  • User enters CICS transaction code (for example CC00).
  • Mapped screen exchanges through BMS mapsets (for example COSGN00).
  • Program reads/writes VSAM resources via EXEC CICS READ/WRITE/REWRITE.
  • Program returns or transfers control with COMMAREA.

Batch Path

  • JCL invokes batch COBOL programs (for example CBTRN02C for POSTTRAN).
  • Files are closed and opened around jobs (CLOSEFIL/OPENFIL pattern).
  • Jobs are chained through CA7 and Control-M conditions.
  • Output includes updated VSAM files, rejects, and generated statements.

Data Layer

  • ACCTDAT, CARDDAT, CUSTDAT, TRANSACT, USRSEC.
  • CCXREF and CXACAIX maintain card-account linkage paths.
  • Copybooks in app/cpy define hard record contracts.
  • Batch files mirror online structures for staging and rejects.

Security Context

  • Sign-on reads VSAM security file (USRSEC).
  • User type is propagated in COMMAREA (A admin, U user).
  • Menu option execution enforces user type and installed program checks.
  • CICS resources configured with user key and quasi-reentrant concurrency.

Dependencies

Internal Module Dependencies

  • transaction-type-management: weekly/periodic type/category refresh jobs affect transaction reference quality consumed by CT02.
  • authorization: optional menu item COPAUS0C appears in menu options and is runtime-checked through EXEC CICS INQUIRE PROGRAM.
  • mq-account-extractions: not required for core UI paths but shares VSAM entities.

External Runtime Dependencies

  • CICS region with CSD resources loaded from app/csd/CARDDEMO.CSD.
  • VSAM clusters and AIX paths: ACCTDATA, CARDDATA, CARDXREF, TRANSACT, USRSEC.
  • Scheduler platform (CA7 and/or Control-M) for nightly and periodic flows.
  • Batch utilities: IDCAMS, DFSORT, and internal reader TD queue JOBS.
  • Assembler/runtime utilities indirectly used by jobs and procedures in repository setup.

Dependency Notes

If file definitions are present in CSD but physical data sets are not provisioned, online flows will fail with read/write errors and batch chains will stop early.

Business Rules (Observed In Source)

  • Sign-on: User ID and password are mandatory, converted to upper case, and checked against USRSEC.
  • Menu execution: Option must be numeric, within configured count, and compatible with user type; uninstalled optional programs are reported to user.
  • Transaction add: Either account id or card number is required; both must be numeric where provided.
  • Transaction add: type/category/source/description/amount/orig date/proc date/merchant id/name/city/zip are all mandatory before write.
  • Transaction add: confirmation flag accepts only Y/N; Y triggers write and account update, blank prompts confirmation.
  • Bill payment: account id required; current balance must be positive to proceed with payment transaction creation.
  • Account update: extensive validation flags for status, dates, numeric ranges, and zip/state consistency before rewrite.
  • Card update: account/card filters and expiry validations gate rewrite behavior.
  • Transaction list: paging uses STARTBR with READNEXT/READPREV; selection moves to detail flow.
  • Batch posting: invalid daily transactions are moved to reject dataset with failure reason trail.
  • Interest calculation: uses disclosure-group lookup by account group and transaction category to compute fees/interest.
  • Statement generation: outputs both plain text and HTML statements from transaction snapshots.

Configuration And Environment Variables

Dataset And Resource Naming Pattern

The sample environment uses high-level qualifier AWS.M2.CARDDEMO. CSD and JCL definitions assume standard names for VSAM clusters and sequential files.

CICS Resource Configuration Highlights

  • Files such as ACCTDAT, CARDDAT, CCXREF, TRANSACT, and USRSEC are enabled for read/update/add/delete with locking update model.
  • Mapsets are registered with group CARDDEMO and normal usage mode.
  • Programs run in user key, CICS API, quasi-reentrant style.
  • Internal reader queue is exposed as extra-partition TD queue JOBS.

Scheduler Configuration Highlights

  • CA7 chain example: CLOSEFIL -> POSTTRAN -> WAITSTEP -> OPENFIL.
  • Control-M daily folder mirrors close/backup/wait/open orchestration.
  • Monthly folder chains close -> INTCALC -> COMBTRAN -> wait -> open.
  • Weekly folders support disclosure and transaction-type refresh dependencies.

Development Guidelines And Patterns

Coding Patterns

  • Keep COMMAREA compatibility stable; use COCOM01Y as single contract source.
  • When adding map fields, update BMS source and generated copybook consumers together.
  • Follow program re-entry pattern using CDEMO-PGM-CONTEXT and EIBCALEN checks.
  • Always provide map redisplay messages for invalid input (do not abend for user-data errors).
  • For browse-driven screens, preserve STARTBR/READNEXT/READPREV paging behavior.

Validation Patterns

  • Mandatory input check first.
  • Format/numeric/date checks second.
  • Data existence checks (VSAM read) third.
  • Only then perform write/rewrite and balance updates.
  • Use cursor placement (-1 map length field) to guide user correction.

Batch Patterns

  • Close online files before jobs that restructure/replace data.
  • Keep reject output explicit and traceable (DALYREJS trailers).
  • Use GDG outputs for historical traceability and recovery.
  • Re-open files for online path after job chain completion.

Verification Checklist

  • Confirm CC00 routes admin to CA00 and standard user to CM00.
  • Confirm CM00 blocks admin-only menu entries for non-admin users.
  • Confirm CT02 rejects missing mandatory fields with cursor focus on failing field.
  • Confirm CT00 paging uses both forward and backward browse without duplicate rows.
  • Confirm CB00 denies payment when account balance is zero/non-positive.
  • Confirm CR00 writes internal-reader job records and displays submission message.
  • Confirm POSTTRAN produces reject file records for invalid input rows.
  • Confirm INTCALC updates account values and emits generated transactions.
  • Confirm CREASTMT outputs both plain text and HTML statements.
  • Confirm scheduler chain closes files before processing and opens files after completion.

User Story Mapping Aids

Cardholder Story Seeds

  • As a cardholder, I want CT02 to validate merchant fields before save so transaction quality is preserved.
  • As a cardholder, I want CB00 to show my current balance before confirmation so I avoid accidental payment.
  • As a cardholder, I want CT00 to page transactions reliably so I can browse long histories.

Operations Story Seeds

  • As operations, I want POSTTRAN rejects to contain reason trailers so bad records can be remediated quickly.
  • As operations, I want CREASTMT HTML output generated nightly so downstream channels can consume statements.
  • As operations, I want scheduler dependency conditions visible so reruns can be managed safely.
