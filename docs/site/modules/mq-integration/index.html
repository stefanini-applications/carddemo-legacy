📡 MQ Integration Module

CardDemo — User Story Development Guide

📋 Overview

Purpose: Demonstrates MQ request/response integration patterns by exposing system date inquiry and account detail inquiry via MQ channels from CICS.

Business Context: This module showcases asynchronous messaging patterns for distributed application integration — a common scenario in mainframe modernization where CICS transactions need to serve as service providers to external systems via MQ.

CICS Transactions: CDRD (system date via MQ), CDRA (account details via MQ)

🎯 User Story Development Guide

📝 US Templates

  • As a distributed application, I want to query the system date via MQ transaction CDRD so that my system can synchronize with the mainframe clock
  • As a distributed application, I want to inquire account details via MQ transaction CDRA so that I can retrieve CardDemo account data without direct VSAM access
  • As a system integrator, I want to understand the MQ request/response pattern in CODATE01/COACCT01 so that I can build additional MQ-enabled services

⚡ Development Acceleration Factors

  • MQ request/response pattern: COACCT01 and CODATE01 are reference implementations for MQ CICS integration
  • VSAM account access pattern: COACCT01 reuses ACCTFILE access established in accounts module
  • CRDDEMOM CSD: MQ-enabled CICS definitions provided

📊 Complexity Guidelines

  • 3–5 pts System Date via MQ (CDRD) — CICS date retrieval, MQ put/get, response format
  • 3–5 pts Account Details via MQ (CDRA) — ACCTFILE read, MQ request/response, data formatting
  • 5–8 pts New MQ-enabled service based on this pattern (additional VSAM data, error handling, async patterns)

🔄 MQ Request/Response Pattern

Requester (external system):
  1. PUT request message to MQ request queue
  2. Include reply-to queue name in message descriptor

CICS Program (CODATE01 / COACCT01):
  3. GET message from MQ request queue (or triggered by MQ)
  4. Process request (get system date / read ACCTFILE)
  5. Format response data
  6. PUT response to reply-to queue specified in request

Requester:
  7. GET response from reply-to queue
  8. Process returned data

✅ Acceptance Criteria Examples

  • Given MQ request message on CDRD queue → CODATE01 returns current system date in response message
  • Given MQ request with valid account ID on CDRA queue → COACCT01 returns account details (balance, limits, status) in response
  • Given MQ request with non-existent account ID → COACCT01 returns appropriate error code in response message
  • Given MQ queue unavailable → CICS transactions handle MQ error gracefully without abend
