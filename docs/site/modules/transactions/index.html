📋 Transactions Module

CICS Online Batch JCL VSAM KSDS

The Transactions module is the financial core of CardDemo. It allows cardholders and operators to list, view, and add credit card transactions online via 3270 terminals (CICS), while a batch JCL pipeline handles end-of-day posting, reporting, and data migration for the same transaction data store.

Overview & Purpose

The Transactions module manages the full lifecycle of credit card financial transactions within CardDemo. It covers:

  • Online inquiry and data entry — cardholders and operators use 3270 CICS screens to list recent transactions (CT00), view a specific transaction (CT01), and add new transactions manually (CT02).
  • Batch posting — the POSTTRAN JCL job runs CBTRN02C nightly to read the daily transaction flat file (DALYTRAN), validate each record, and write accepted transactions to the permanent TRANSACT VSAM KSDS while updating account balance totals.
  • Reporting — CBTRN03C generates a dated transaction detail report (TRANREPT job), triggered either from the CICS CR00 screen or scheduled JCL.
  • Data migration — CBEXPORT and CBIMPORT provide bulk export/import of transaction data for branch migration scenarios.

Architectural Context

Transactions sit at the centre of the CardDemo architecture. Every spending event, fee, and payment ultimately creates a record in the TRANSACT VSAM KSDS. Upstream modules (accounts, credit-cards, bill-payment) read and write this file; downstream modules (reports, batch-processing) consume it.

Data Models

TRANSACT VSAM Record — CVTRA05Y (350 bytes)

Primary key: TRAN-ID (16 chars). Alternate Index (AIX) on TRAN-CARD-NUM allows lookup by card number.

01  TRAN-RECORD.
    05  TRAN-ID              PIC X(16).        *> Transaction identifier (primary key)
    05  TRAN-TYPE-CD         PIC X(02).        *> 2-char type code → refs TRANTYPE VSAM
    05  TRAN-CAT-CD          PIC 9(04).        *> 4-digit category → refs TRANCATG VSAM
    05  TRAN-SOURCE          PIC X(10).        *> Source system identifier
    05  TRAN-DESC            PIC X(100).       *> Free-form description
    05  TRAN-AMT             PIC S9(09)V99.    *> Signed amount (debit positive)
    05  TRAN-MERCHANT-ID     PIC 9(09).        *> 9-digit merchant identifier
    05  TRAN-MERCHANT-NAME   PIC X(50).        *> Merchant name
    05  TRAN-MERCHANT-CITY   PIC X(50).        *> Merchant city
    05  TRAN-MERCHANT-ZIP    PIC X(10).        *> Merchant ZIP code
    05  TRAN-CARD-NUM        PIC X(16).        *> Card used (AIX key)
    05  TRAN-ORIG-TS         PIC X(26).        *> Original transaction timestamp
    05  TRAN-PROC-TS         PIC X(26).        *> Processing (posting) timestamp
    05  FILLER               PIC X(20).        *> Reserved (total = 350 bytes)

DALYTRAN Daily Transaction File — CVTRA06Y (350 bytes)

Same structure as CVTRA05Y but field names prefixed with DALYTRAN-. Sequential (PS) flat file — input to CBTRN02C.

01  DALYTRAN-RECORD.
    05  DALYTRAN-ID          PIC X(16).
    05  DALYTRAN-TYPE-CD     PIC X(02).
    05  DALYTRAN-CAT-CD      PIC 9(04).
    05  DALYTRAN-SOURCE      PIC X(10).
    05  DALYTRAN-DESC        PIC X(100).
    05  DALYTRAN-AMT         PIC S9(09)V99.
    05  DALYTRAN-MERCHANT-ID PIC 9(09).
    05  DALYTRAN-MERCHANT-NAME   PIC X(50).
    05  DALYTRAN-MERCHANT-CITY   PIC X(50).
    05  DALYTRAN-MERCHANT-ZIP    PIC X(10).
    05  DALYTRAN-CARD-NUM    PIC X(16).
    05  DALYTRAN-ORIG-TS     PIC X(26).
    05  DALYTRAN-PROC-TS     PIC X(26).
    05  FILLER               PIC X(20).

Transaction Category Balance — CVTRA01Y (50 bytes)

Running balance by account × type × category. Updated by CBTRN02C on each posted transaction.

01  TRAN-CAT-BAL-RECORD.
    05  TRAN-CAT-KEY.
        10  TRANCAT-ACCT-ID    PIC 9(11).   *> Account ID
        10  TRANCAT-TYPE-CD    PIC X(02).   *> Transaction type code
        10  TRANCAT-CD         PIC 9(04).   *> Category code
    05  TRAN-CAT-BAL           PIC S9(09)V99. *> Running balance
    05  FILLER                 PIC X(22).

Transaction Type Reference — CVTRA03Y (60 bytes)

01  TRAN-TYPE-RECORD.
    05  TRAN-TYPE       PIC X(02).      *> 2-char type code (primary key)
    05  TRAN-TYPE-DESC  PIC X(50).      *> Human-readable description
    05  FILLER          PIC X(08).

Transaction Category Reference — CVTRA04Y (60 bytes)

01  TRAN-CAT-RECORD.
    05  TRAN-CAT-KEY.
        10  TRAN-TYPE-CD       PIC X(02).   *> Type code
        10  TRAN-CAT-CD        PIC 9(04).   *> Category code
    05  TRAN-CAT-TYPE-DESC     PIC X(50).   *> Description
    05  FILLER                 PIC X(04).

Disclosure Group (Interest Rate) Record — CVTRA02Y (50 bytes)

Links an account group ID + transaction type + category to an annual interest rate. Used by CBACT04C interest calculation (accounts module).

01  DIS-GROUP-RECORD.
    05  DIS-GROUP-KEY.
        10  DIS-ACCT-GROUP-ID  PIC X(10).   *> Account group (from ACCT-GROUP-ID)
        10  DIS-TRAN-TYPE-CD   PIC X(02).   *> Transaction type
        10  DIS-TRAN-CAT-CD    PIC 9(04).   *> Category
    05  DIS-INT-RATE           PIC S9(04)V99. *> Annual interest rate %
    05  FILLER                 PIC X(28).

Business Rules

Transaction Identification

  • Transaction ID (TRAN-ID) is a 16-character unique key — no duplicates allowed in TRANSACT KSDS.
  • COTRN02C generates a new TRAN-ID at add-time (numeric sequence padded to 16 chars).
  • Batch-posted records use the DALYTRAN-ID from the daily file as the TRAN-ID.

Amount Validation

  • Transaction amount (TRAN-AMT) must be non-zero.
  • Debit transactions carry a positive amount; credit/payment transactions carry a negative or zero-flagged amount in the type code.
  • Amount is stored as PIC S9(09)V99 — two implied decimal places, sign stored.

Card & Account Validation (CBTRN02C batch)

  • Validation step 1500-VALIDATE-TRAN first looks up DALYTRAN-CARD-NUM in XREFFILE (reason code 100 = invalid card).
  • If the card is found, it resolves XREF-ACCT-ID and reads the account record (reason code 101 = account not found).
  • Failed validations write the full 350-byte record to the DALYREJS reject file.
  • Job return code is set to 4 if any rejects occurred; otherwise 0.

Card & Account Validation (COTRN02C online)

  • Card number entered on CT02 screen is validated against CCXREF VSAM (ACCESS MODE RANDOM).
  • Account ID is resolved via CXACAIX alternate index on the cross-reference file.
  • Transaction date is validated using the CSUTLDTC date utility (format YYYY-MM-DD).

Transaction Type Code

  • TRAN-TYPE-CD is a 2-character code that must exist in the TRANTYPE VSAM (or DB2 TRANSACTION_TYPE table if the transaction-type-db2 module is deployed).
  • The CBTRN03C report program requires a valid type code to produce the description column.

Category Balance Tracking

  • CBTRN02C updates TRAN-CAT-BAL-RECORD in TCATBALF for each accepted transaction.
  • Key = account-ID + type-code + category-code; balance is incremented by the transaction amount.
  • These category balances are consumed by CBACT04C (interest calculation) in the accounts / batch-processing module.

Timestamps

  • TRAN-ORIG-TS (26 chars) records when the transaction originated at the merchant or entry point.
  • TRAN-PROC-TS (26 chars) records when the transaction was processed/posted by CardDemo.
  • CBTRN03C filters the report by TRAN-PROC-TS date range read from the DATEPARM file.

Batch Processing Pipeline

Transactions participates in the following JCL jobs within the end-of-day pipeline:

Development Patterns

CICS Browse Pattern (CT00 Paging)

COTRN00C uses CICS file control browse commands to page through the TRANSACT KSDS:

* Forward page — start browse at first transaction ID of next page
    EXEC CICS STARTBR FILE('TRANSACT')
              RIDFLD(WS-TRAN-ID-NEXT)
              RESP(WS-RESP-CD)
    END-EXEC

    PERFORM UNTIL WS-IDX >= 11 OR TRANSACT-EOF OR ERR-FLG-ON
        EXEC CICS READNEXT FILE('TRANSACT')
                  INTO(TRAN-RECORD)
                  RIDFLD(WS-TRAN-ID-CURRENT)
                  RESP(WS-RESP-CD)
        END-EXEC
        PERFORM POPULATE-TRAN-DATA
    END-PERFORM

    EXEC CICS ENDBR FILE('TRANSACT') END-EXEC

* Backward page — use READPREV to populate screen rows in reverse order

COMMAREA State Passing

Navigation state between CT00, CT01, and CT02 is passed via the DFHCOMMAREA, structured by COCOM01Y:

COPY COCOM01Y.
   05 CDEMO-CT00-INFO.
      10 CDEMO-CT00-TRNID-FIRST     PIC X(16). *> First tran-id on current page
      10 CDEMO-CT00-TRNID-LAST      PIC X(16). *> Last tran-id on current page
      10 CDEMO-CT00-PAGE-NUM        PIC 9(08). *> Current page number
      10 CDEMO-CT00-NEXT-PAGE-FLG   PIC X(01). *> 'Y' = more pages forward
      10 CDEMO-CT00-TRN-SEL-FLG     PIC X(01). *> Selection flag
      10 CDEMO-CT00-TRN-SELECTED    PIC X(16). *> Selected TRAN-ID → CT01

Batch Validation Pattern (CBTRN02C)

PERFORM UNTIL END-OF-FILE = 'Y'
    PERFORM 1000-DALYTRAN-GET-NEXT
    IF END-OF-FILE = 'N'
        ADD 1 TO WS-TRANSACTION-COUNT
        MOVE 0 TO WS-VALIDATION-FAIL-REASON
        PERFORM 1500-VALIDATE-TRAN        *> Card lookup then account lookup
        IF WS-VALIDATION-FAIL-REASON = 0
            PERFORM 2000-POST-TRANSACTION  *> Write TRANSACT + update TCATBAL
        ELSE
            ADD 1 TO WS-REJECT-COUNT
            PERFORM 2500-WRITE-REJECT-REC  *> Write to DALYREJS
        END-IF
    END-IF
END-PERFORM

*> Display counts at end
DISPLAY 'TRANSACTIONS PROCESSED :' WS-TRANSACTION-COUNT
DISPLAY 'TRANSACTIONS REJECTED  :' WS-REJECT-COUNT
IF WS-REJECT-COUNT > 0
    MOVE 4 TO RETURN-CODE  *> RC=4 signals partial failure
END-IF

Date Utility Call Pattern (COTRN02C)

* CSUTLDTC validates and converts dates; called from COTRN02C
01 CSUTLDTC-PARM.
   05 CSUTLDTC-DATE           PIC X(10).  *> Input date string
   05 CSUTLDTC-DATE-FORMAT    PIC X(10).  *> Format e.g. 'YYYY-MM-DD'
   05 CSUTLDTC-RESULT.
      10 CSUTLDTC-RESULT-SEV-CD    PIC X(04).
      10 FILLER                    PIC X(11).
      10 CSUTLDTC-RESULT-MSG-NUM   PIC X(04).
      10 CSUTLDTC-RESULT-MSG       PIC X(61).

MOVE WS-DATE-INPUT  TO CSUTLDTC-DATE
MOVE 'YYYY-MM-DD'   TO CSUTLDTC-DATE-FORMAT

User Story Examples

Online Stories

As a cardholder, I want to list my recent transactions (CT00) so I can monitor my spending.

Acceptance Criteria:
  • GIVEN I am signed in and on the Main Menu
  • WHEN I navigate to CT00
  • THEN the system displays up to 10 transactions per page, ordered by TRAN-ID
  • AND I can page forward with PF8 and backward with PF7
  • AND each row shows transaction ID, date, truncated description, and formatted amount

As a cardholder, I want to view the full details of a transaction (CT01) so I can verify a charge.

Acceptance Criteria:
  • GIVEN I am on the CT00 list screen
  • WHEN I select a transaction row and press Enter
  • THEN CT01 displays all fields including merchant name, city, ZIP, type, category, source, timestamps, and amount
  • AND PF3/PF12 returns me to the CT00 list

As an operator, I want to add a transaction manually (CT02) so a charge not present in the daily file can be recorded.

Acceptance Criteria:
  • GIVEN I am on CT02
  • WHEN I enter a valid card number, type code, category, amount, and origination date and press Enter
  • THEN COTRN02C validates the card via CCXREF, resolves the account, validates the date, generates a TRAN-ID
  • AND writes a new record to TRANSACT VSAM
  • AND displays a success confirmation message
  • AND an invalid card number or malformed date displays an error message without writing to VSAM

Batch Stories

As a batch operator, I want POSTTRAN to post daily transactions overnight so account balances reflect all card activity by morning.

Acceptance Criteria:
  • GIVEN DALYTRAN PS file is present and TRANSACT VSAM is open
  • WHEN POSTTRAN (CBTRN02C) is submitted
  • THEN each valid transaction is written to TRANSACT VSAM and TCATBAL is updated
  • AND invalid transactions (unknown card or account) are written to DALYREJS
  • AND job completes RC=0 (no rejects) or RC=4 (partial rejects)
  • AND counts are displayed: TRANSACTIONS PROCESSED / TRANSACTIONS REJECTED

As a cardholder, I want to export my transaction history so I can analyse spending in external tools.

Acceptance Criteria:
  • GIVEN CBEXPORT is submitted with all source VSAM files available
  • WHEN the job completes
  • THEN a multi-record export flat file is created containing all transaction, account, customer, card, and cross-reference records
  • AND export statistics are printed to SYSOUT

Acceptance Criteria Patterns

Online CT0x Programs

  • Must read current record from TRANSACT VSAM before any display or update
  • Must validate all required input fields (no spaces in card number, non-zero amount)
  • Must display error messages on the screen for invalid input without writing to VSAM
  • Must use CICS HANDLE CONDITION or RESP code checking on all EXEC CICS statements
  • Must return to the previous screen on PF3/PF12 without data loss in COMMAREA
  • Must set CDEMO-CT00-NEXT-PAGE-FLG correctly so the paging indicators are accurate

CBTRN02C Batch

  • Must complete with return code 0 for a clean run (zero rejects)
  • Must complete with return code 4 if any records were rejected (not an abend — partial success)
  • Must display TRANSACTIONS PROCESSED and TRANSACTIONS REJECTED counts
  • Must handle VSAM record-not-found conditions gracefully (reject and continue, not abend)
  • Must update TCATBAL for each accepted transaction — missing update causes interest calculation errors

Readiness Considerations

⚠ AIX Rebuild Required Post-POSTTRAN: The card-number Alternate Index on TRANSACT must be rebuilt (TRANIDX job) after POSTTRAN completes. Skipping TRANIDX will cause CT00 card-filtered lookups and any program using the AIX to return stale data.
⚠ DALYTRAN File Preparation: CBTRN02C opens DALYTRAN as ORGANIZATION SEQUENTIAL. The file must exist and be correctly populated before POSTTRAN is submitted. If DALYTRAN is empty, CBTRN02C will complete with RC=0 and zero records processed (not an error condition).
Tech Debt — No Duplicate Transaction Check: CBTRN02C does not check whether a DALYTRAN-ID already exists in the TRANSACT KSDS before writing. A duplicate TRAN-ID will cause a VSAM key-already-exists error (status '22') and abend the job. Ensure the DALYTRAN file contains unique IDs.
Performance: TRANSACT VSAM is a KSDS. Random access by key (TRAN-ID) is <50ms. Sequential browse (CT00 paging) performance depends on file size — the AIX on TRAN-CARD-NUM is critical for card-filtered paging. CBTRN02C processes thousands of records per batch run using sequential DALYTRAN reads and random TRANSACT writes.
