💰 Transactions

User Story Development Guide — CardDemo Application

📋 Overview

Purpose: Allow users to browse transaction history, view individual transaction details, and add new transactions online. Batch programs post daily transactions to the TRANSACT file and update account balances.

Business Context: Transactions are the primary financial events in CardDemo. They record purchases, credits, payments, fees, and interest charges. Transaction data drives statements, reports, and account balances.

COTRN00C COTRN01C COTRN02C VSAM: TRANSACT CBTRN01C

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Transaction List: As a cardholder, I want to browse my transaction history so that I can verify charges and identify suspicious activity.
  • Transaction Detail: As a cardholder, I want to view the details of a specific transaction so that I can see merchant info, amount, and timestamps.
  • Transaction Add: As an admin, I want to manually add a transaction so that I can process credits, fee waivers, or adjustment entries.
  • Transaction Search: As a cardholder, I want to filter transactions by date range so that I can review a specific billing period.

⚡ Development Acceleration Factors

  • CVTRA05Y.cpy: Complete 350-byte transaction record structure shared across all CICS and batch programs.
  • Pageable List (COTRN00C): Forward/backward navigation pattern established for transaction browsing.
  • TRAN-ID Generation: 16-character transaction ID pattern used for all new transactions.

🔧 Technical Foundation

🏗️ Key Components

  • COTRN00C.cbl: Transaction list — pageable display of transactions for the selected card/account.
  • COTRN01C.cbl: Transaction view — detailed view of a single transaction record.
  • COTRN02C.cbl: Transaction add — create new transaction record in TRANSACT file.
  • CBTRN01C.cbl: Batch daily posting — reads DALYTRAN sequential file, validates against CARDXREF/ACCTDAT, posts to TRANSACT.
  • CBTRN02C.cbl: Alternate batch posting with reject file (DALYREJS).
  • CVTRA05Y.cpy: Transaction record layout (350 bytes, key = TRAN-ID 16 chars).

🔗 Dependencies

  • TRANSACT VSAM: Transaction master file (KSDS, key = TRAN-ID).
  • CARDXREF VSAM: Used in batch to resolve CARD-NUM → ACCT-ID.
  • ACCTDAT VSAM: Account balance updated during batch transaction posting.
  • TCATBALF VSAM: Transaction category balance — updated by batch posting for interest calculation.
  • DALYTRAN: Sequential input file for daily batch posting.

📊 Transaction Record Key Fields

TRAN-ID (X(16))            - Transaction identifier (primary key)
TRAN-TYPE-CD (X(2))        - Transaction type (e.g., 'SA'=sale, 'CR'=credit)
TRAN-CAT-CD (9(4))         - Category code (links to TCATBALF)
TRAN-SOURCE (X(10))        - Source system (e.g., 'ONLINE', 'BATCH')
TRAN-AMT (S9(9)V99)        - Amount (positive=debit, negative=credit)
TRAN-CARD-NUM (X(16))      - Card number
TRAN-MERCHANT-NAME (X(50)) - Merchant name
TRAN-ORIG-TS (X(26))       - Original timestamp
TRAN-PROC-TS (X(26))       - Processing timestamp

✅ Acceptance Criteria Patterns

Transaction List:
  • Given a card with transaction history
  • When user navigates to transaction list
  • Then transactions displayed with ID, type, amount, merchant, and date
  • And user can page forward and backward through the list
Transaction Add:
  • Given all required fields entered (card number, type, amount, merchant)
  • When admin submits transaction add
  • Then transaction written to TRANSACT file and confirmation shown
Batch Posting:
  • Given DALYTRAN file contains daily transactions
  • When CBTRN01C runs
  • Then all valid transactions posted to TRANSACT; account balances updated; invalid records written to DALYREJS
