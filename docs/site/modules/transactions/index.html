Core Module CT00 · CT01 · CT02 · CBTRN

📋 Overview

The Transactions module handles the full lifecycle of financial transactions: online listing with paging, individual record viewing, online entry of new transactions, and batch posting of daily transaction files. It is the core financial processing engine of CardDemo.

Data Store

TRANSACT VSAM KSDS — 350-byte transaction records keyed on TRAN-ID (16 chars), defined in copybook CVTRA05Y. An Alternate Index (AIX) enables card-number-based lookup, supporting the CT00 filtered list view.

Business Context

Daily transactions arrive in the DALYTRAN sequential file. The batch job POSTTRAN (CBTRN02C) reads this file and posts each record to TRANSACT VSAM KSDS, updating the account balance atomically. TRAN-TYPE-CD references the TRANTYPE file; TRAN-CAT-CD references the TRANCATG file — both are validated on add (CT02) and used for categorized reporting (CBTRN03C).

🎯 User Story Development Guide

Story Templates

List Transactions for Card/Account
As a cardholder, I want to see a paginated list of transactions for my card so that I can review my spending history.

Program: COTRN00C (CT00) — uses TRANSACT AIX by card number; supports PF7/PF8 for backward/forward paging
View Transaction Detail
As a cardholder, I want to view the full detail of a specific transaction including merchant and amount so that I can verify or dispute a charge.

Program: COTRN01C (CT01) — keyed read on TRANSACT VSAM using TRAN-ID
Add New Transaction
As a teller, I want to add a new transaction for a card so that manual adjustments and corrections can be entered online.

Program: COTRN02C (CT02) — validates CARD-ACTIVE-STATUS, TRAN-TYPE-CD (TRANTYPE), TRAN-CAT-CD (TRANCATG), then inserts to TRANSACT VSAM
Batch Post Daily Transactions
As a batch operations manager, I want POSTTRAN (CBTRN02C) to post all daily transactions to the TRANSACT VSAM and update account balances so that end-of-day processing completes accurately.

Input: DALYTRAN sequential file; Output: TRANSACT VSAM KSDS + updated ACCTDATA balances; RC=0 expected
Export Transactions
As a data analyst, I want to export all transactions within a date range to a flat file so that I can perform offline analysis.

Program: CBEXPORT — sequential scan of TRANSACT VSAM with date filter; output to export flat file

Acceleration Factors

  • CVTRA05Y fully defined — complete 350-byte transaction record; no data modeling required for standard stories
  • CT00 paging pattern — forward/backward paging using TRANSACT AIX is directly reusable for any paginated list story
  • CBTRN02C post logic — sequential-to-KSDS post with balance update is the template for any new batch posting rule
  • CBTRN03C date-range filter — sequential scan with date comparison is reusable for any report with date range filtering

✅ Acceptance Criteria Patterns

Transaction List (CT00)

  • GIVEN an authenticated user with a valid card number, WHEN CT00 is invoked, THEN transactions are listed in reverse chronological order using the TRANSACT AIX
  • GIVEN more transactions than fit on one screen, WHEN the user presses PF8, THEN the next page of transactions is displayed; PF7 returns to previous page

Add Transaction (CT02)

  • GIVEN a valid transaction entry, WHEN CT02 processes it, THEN the record is inserted into TRANSACT VSAM with a unique TRAN-ID and a confirmation is displayed
  • GIVEN an inactive card, WHEN CT02 validates the card, THEN the transaction is rejected with an "inactive card" error message
  • GIVEN an invalid TRAN-TYPE-CD, WHEN CT02 validates the type, THEN the transaction is rejected with a validation error
  • GIVEN a duplicate TRAN-ID, WHEN CT02 attempts the insert, THEN the insert is rejected and an appropriate error is displayed

Batch Post (CBTRN02C / POSTTRAN)

  • GIVEN a DALYTRAN file with valid transaction records, WHEN POSTTRAN runs, THEN all records are posted to TRANSACT VSAM with RC=0
  • GIVEN posted transactions, WHEN POSTTRAN completes, THEN corresponding account balances in ACCTDATA VSAM reflect the posted amounts atomically
  • GIVEN a record with an invalid type code, WHEN POSTTRAN processes it, THEN the record is written to the error report and processing continues
