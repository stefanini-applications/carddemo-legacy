📦 Transaction Type Management

Guide for Developing User Stories

📋 Overview

Purpose: Offer administrators a DB2-driven cockpit for transaction-type metadata and keep those updates synchronized with VSAM-based transaction processing.

Business Context: Keeping transaction codes accurate enables modernization stories such as categorization-based rewards or analytics dashboards.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Metadata Editing: As an admin, I want to add a new transaction type via CTTU so that marketing campaigns can target a new category.
  • Batch Sync: As a release engineer, I want TRANEXTR to drop VSAM-compatible files right after DB2 commits so production screens always show the current categories.
  • Cursor Navigation: As a back office user, I want forward/back cursor buttons in CTLI so that I can page through long lists without reloading.

⚡ Development Acceleration Factors

  • DB2 Precompiler Setup: Jobs such as CREADB21 predefine tables; use the same sysin definitions for new tables.
  • SQLCA Template: Existing COBOL programs demonstrate error handling and COMMIT/ROLLBACK sequences.
  • Copybook Reuse: Use `app/app-transaction-type-db2/cpy/` copybooks for host variables when adding services.

📊 Complexity Guidelines

  • Simple: Adjust displayed text in CTLI without touching DB2.
  • Medium: Add a derived column to `TRANSACTION_TYPE` and update `CTTU`/`CTLI` to handle it.
  • Complex: Add new DB2 tables with referential integrity plus corresponding nightly extraction and validation.

🔧 Technical Foundation

🏗️ Key Components

  • CICS Transactions: CTLI (list/update/delete) and CTTU (add/edit) backed by programs COTRTLIC and COTRTUPC.
  • Batch Components: CREADB21 (build DB2 tables), TRANEXTR (extract DB2 rows into VSAM files), MNTTRDB2 (batch maintenance via COBTUPDT).
  • DB2 Tables: TRANSACTION_TYPE and TRANSACTION_TYPE_CATEGORY with `DELETE RESTRICT` constraints.
  • Copybooks & DDL: `app/app-transaction-type-db2/dcl/`, `ddl/`, and `cpy/` contain the declaration, definition, and map structures.

🔗 Dependencies

  • DB2 Subsystem: Requires a running DB2 subsystem accessible from CICS.
  • Base Application: `TRANEXTR` writes VSAM payloads consumed by `COTRN00C` and the transaction list screen.
  • CICS Resource Definitions: Provided in app/app-transaction-type-db2/csd/ to register the new transactions.

🌐 Public Interfaces

CICS CTLI → COTRTLIC (list/update/delete with forward/back cursors)
CICS CTTU → COTRTUPC (add/edit using static SQL)
JCL CREABD21 → create DB2 schema objects
JCL TRANEXTR → export DB2 rows and generate VSAM-friendly data for the base module

✅ Acceptance Criteria Examples

  • Given a transaction type marked for deletion in CTLI, when the system attempts to delete it, then DB2 enforces the DELETE RESTRICT constraint so no orphaned categories exist.
  • Given a DB2 update in CTTU, when TRANEXTR runs, then the extracted VSAM file contains the latest description and category values for downstream screens.
