Overview and Purpose

The credit-cards module enables listing, viewing, and updating credit card records associated with customer accounts in the CardDemo application. Credit cards are the primary payment instrument in the system, and this module manages the full lifecycle of online card management via three CICS transactions on IBM 3270 terminals.

Cards are stored in the CARDDATA VSAM file and cross-referenced to accounts and customers via the CARDXREF VSAM file. Multiple cards can be linked to a single account, and the module supports both admin-level browsing (all cards) and user-level browsing (cards filtered by account).

Module ID: credit-cards
Access Path: CC00 (Sign-on) → CM00 (Main Menu) → CCLI (Card List)
Layer: CICS Online Transaction Processing (3270 screen-based)

Architectural Context

The credit-cards module follows the standard CardDemo CICS architectural pattern: each transaction maps to a COBOL program, a BMS mapset (3270 screen definition), and a generated copybook. Navigation state is passed between programs via the CARDDEMO-COMMAREA structure (defined in COCOM01Y.cpy), and per-program state is maintained in a program-local commarea appended after the system commarea.

3270 Screen Flow

Users navigate from the main menu to the card list screen (CCLI), where they can page through card records filtered by account or card number. Selecting a card navigates to the detail screen (CCDL). From there, a further PF key invokes the update screen (CCUP) where card attributes can be modified and saved.

All programs pass state via COMMAREA (COCOM01Y), which tracks current program, next program, account ID, card number, customer ID, and navigation flags.

COBOL Business Logic Programs

Data Models

Card Record — CVACT02Y (RECLN=150)

Primary key: CARD-NUM (16 characters). Stored in VSAM KSDS dataset AWS.M2.CARDDEMO.CARDDATA.PS.

01  CARD-RECORD.
    05  CARD-NUM              PIC X(16).    *> 16-digit card number (VSAM primary key)
    05  CARD-ACCT-ID          PIC 9(11).    *> Linked account ID (foreign key to ACCTDATA)
    05  CARD-CVV-CD           PIC 9(03).    *> CVV security code (3 digits)
    05  CARD-EMBOSSED-NAME    PIC X(50).    *> Cardholder name as embossed on card (alpha only)
    05  CARD-EXPIRAION-DATE   PIC X(10).    *> Expiration date in format YYYY-MM-DD
    05  CARD-ACTIVE-STATUS    PIC X(01).    *> Y=Active, N=Inactive
    05  FILLER                PIC X(59).    *> Padding to 150 bytes
Note on field naming: CARD-EXPIRAION-DATE is intentionally spelled with the original application typo ("EXPIRAION" not "EXPIRATION") to match the existing VSAM layout. Do not correct this in new code without updating all references.

Interaction Flows

Card List Flow (COCRDLIC / CCLI)

1

User enters CCLI transaction from main menu (CM00). EIBCALEN=0 on first call → initialize COMMAREA, set page to first.

2

Program checks user type (CDEMO-USRTYP-ADMIN). Admin users may browse all cards; regular users see only cards matching the account in COMMAREA.

3

CICS STARTBR on CARDDATA VSAM at the current RID (card number). Reads up to 7 records forward using READNEXT, applying account/card filter via 9500-FILTER-RECORDS.

4

Screen is populated with rows: select column, account number, card number, status. ENDFILE response sets "no next page" flag.

5

User enters S (select for view) or U (select for update) in row selector field. Program validates selection, stores card/account in COMMAREA, XCTLs to COCRDSLC (view) or COCRDUPC (update).

6

PF7 → backward page (9100-READ-BACKWARDS using READPREV). PF8 → forward page. PF3 → return to main menu via XCTL to COMEN01C.

Card Detail View Flow (COCRDSLC / CCDL)

1

Invoked via XCTL from COCRDLIC with COMMAREA containing account ID and card number. Also accessible by typing CCDL transaction directly.

2

Program performs CICS READ on CARDDATA VSAM by card number key. On NOTFND, displays error. On NORMAL, populates screen with all card fields.

3

Account ID and card number fields are protected (DFHBMPRF) if coming from the list screen. Editable if entered directly.

4

PF5 navigates to update screen (COCRDUPC). PF3 returns to card list (COCRDLIC). ENTER re-queries the card.

Card Update Flow (COCRDUPC / CCUP)

1

Initial entry: program reads current CARDDATA record into CCUP-OLD-DETAILS for change-detection snapshot. Screen is populated with current values.

2

User modifies fields: embossed name (alpha only), expiry month (01–12), expiry year (1950–2099), expiry day, active status (Y/N).

3

On ENTER: program performs 1200-EDIT-MAP-INPUTS, validating each field. Errors are highlighted in red (DFHRED) with cursor positioned at first invalid field.

4

If all edits pass, state changes to CCUP-CHANGES-OK-NOT-CONFIRMED. User must press ENTER again to confirm.

5

On confirmation: 9200-WRITE-PROCESSING reads CARDDATA with UPDATE option to lock record. Calls 9300-CHECK-CHANGE-IN-REC to detect concurrent modification. If unchanged, performs CICS REWRITE to commit the update.

6

If concurrent modification detected, the screen is refreshed with the most recent values from disk. PF3 cancels and returns to card list.

Business Rules

Card List Display Rules

  • Admin users (COMMAREA CDEMO-USRTYP-ADMIN) see all cards in CARDDATA regardless of account.
  • Regular users see only cards where CARD-ACCT-ID matches the account ID in their COMMAREA session.
  • Account ID filter must be an 11-digit numeric value; card number filter must be a 16-digit numeric value.
  • Page size is 7 rows per screen. PF8 pages forward; PF7 pages backward.
  • Row selector accepts S (view) or U (update). Only one row may be selected at a time.
  • Multiple selections in a single submit produce a validation error.

Card Update Validation Rules

  • Account ID: Required, must be 11-digit numeric (not spaces, not zero). Error: "ACCOUNT FILTER,IF SUPPLIED MUST BE A 11 DIGIT NUMBER".
  • Card Number: Required, must be 16-digit numeric. Error: "CARD ID FILTER,IF SUPPLIED MUST BE A 16 DIGIT NUMBER".
  • Embossed Name: Required, must contain only alphabetic characters and spaces. Validated by INSPECT CONVERTING alpha-to-space, then checking if result trims to empty. Error displayed if non-alpha characters found.
  • Active Status: Must be Y (active) or N (inactive). Error: "CARD STATUS MUST BE Y OR N".
  • Expiry Month: Must be numeric, in range 01–12. Error on blank or out-of-range.
  • Expiry Year: Must be numeric, in range 1950–2099. Error on blank or out-of-range.
  • Expiry Day: Must be numeric and valid. Stored as DD portion of YYYY-MM-DD.
  • CVV Code: Displayed but not editable on the update screen; preserved as-is from the existing record.

Concurrency Control

  • COCRDUPC implements optimistic concurrency: it stores a snapshot of the record's key fields (CCUP-OLD-DETAILS) at initial read.
  • Before writing, it re-reads the record with UPDATE and compares all fields with the snapshot in 9300-CHECK-CHANGE-IN-REC.
  • If any field differs (another user modified the record between reads), the update is aborted and the screen is refreshed with the latest disk values.
  • The CICS READ UPDATE holds an exclusive lock; the lock is released by REWRITE or transaction end.

Data Integrity

  • Card number is a 16-character VSAM primary key; it cannot be changed through the update screen.
  • Account ID cannot be changed through the update screen (field is protected).
  • CVV code is stored in plain text (PCI-DSS consideration — see Readiness).
  • Embossed name is automatically converted to uppercase via INSPECT CONVERTING (lowercase→uppercase) before storing.
  • Expiry date is stored as a single string in YYYY-MM-DD format, composed from separate year/month/day input fields using STRING.

Error Handling

VSAM Response Codes

All three programs evaluate WS-RESP-CD (mapped from CICS RESP) after every file I/O:

CICS Response Meaning Program Action
DFHRESP(NORMAL) Successful read/write Continue processing
DFHRESP(NOTFND) Card record not found Set INPUT-ERROR, display "DID NOT FIND ACCT/CARD COMBO" message
DFHRESP(ENDFILE) End of VSAM browse reached Set CA-NEXT-PAGE-NOT-EXISTS, display "NO MORE RECORDS TO SHOW"
DFHRESP(DUPREC) Duplicate record (browse) Treated as NORMAL during browse — next page still exists
Other Unexpected I/O error Set READ-LOOP-EXIT, populate error message with opname + file + RESP codes; display file error message

Concurrency Errors

  • COULD-NOT-LOCK-FOR-UPDATE: CICS READ UPDATE returned non-NORMAL response — displayed as error, update not attempted.
  • DATA-WAS-CHANGED-BEFORE-UPDATE: Snapshot comparison failed — screen refreshed with current disk values and user informed to review changes.
  • LOCKED-BUT-UPDATE-FAILED: REWRITE returned non-NORMAL after successful lock — displayed as error.

Abend Handling

All programs include an ABEND-ROUTINE paragraph that sends a plain-text error message to the screen before issuing EXEC CICS ABEND ABCODE('9999'). The culprit program name (LIT-THISPGM) is included in the abend data.

Code Examples

Example 1: VSAM Browse (Forward) — from COCRDLIC

*> Start browse at a given card number key
EXEC CICS STARTBR
     DATASET(LIT-CARD-FILE)
     RIDFLD(WS-CARD-RID-CARDNUM)
     KEYLENGTH(LENGTH OF WS-CARD-RID-CARDNUM)
     GTEQ                          *> Start at key >= given value
     RESP(WS-RESP-CD)
     RESP2(WS-REAS-CD)
END-EXEC

*> Read up to 7 records
PERFORM UNTIL READ-LOOP-EXIT

    EXEC CICS READNEXT
         DATASET(LIT-CARD-FILE)
         INTO(CARD-RECORD)
         LENGTH(LENGTH OF CARD-RECORD)
         RIDFLD(WS-CARD-RID-CARDNUM)
         KEYLENGTH(LENGTH OF WS-CARD-RID-CARDNUM)
         RESP(WS-RESP-CD)
         RESP2(WS-REAS-CD)
    END-EXEC

    EVALUATE WS-RESP-CD
        WHEN DFHRESP(NORMAL)
        WHEN DFHRESP(DUPREC)
            PERFORM 9500-FILTER-RECORDS
            IF WS-DONOT-EXCLUDE-THIS-RECORD
                ADD 1 TO WS-SCRN-COUNTER
                MOVE CARD-NUM         TO WS-ROW-CARD-NUM(WS-SCRN-COUNTER)
                MOVE CARD-ACCT-ID     TO WS-ROW-ACCTNO(WS-SCRN-COUNTER)
                MOVE CARD-ACTIVE-STATUS TO WS-ROW-CARD-STATUS(WS-SCRN-COUNTER)
            END-IF
        WHEN DFHRESP(ENDFILE)
            SET READ-LOOP-EXIT TO TRUE
            SET CA-NEXT-PAGE-NOT-EXISTS TO TRUE
        WHEN OTHER
            SET READ-LOOP-EXIT TO TRUE
            MOVE WS-FILE-ERROR-MESSAGE TO WS-ERROR-MSG
    END-EVALUATE

END-PERFORM

EXEC CICS ENDBR FILE(LIT-CARD-FILE)
END-EXEC

Example 2: Record Filtering (Account/Card) — from COCRDLIC 9500-FILTER-RECORDS

9500-FILTER-RECORDS.
    SET WS-DONOT-EXCLUDE-THIS-RECORD TO TRUE

    *> Filter by account ID if provided
    IF FLG-ACCTFILTER-ISVALID
        IF CARD-ACCT-ID = CC-ACCT-ID
            CONTINUE
        ELSE
            SET WS-EXCLUDE-THIS-RECORD TO TRUE
            GO TO 9500-FILTER-RECORDS-EXIT
        END-IF
    END-IF

    *> Filter by card number if provided
    IF FLG-CARDFILTER-ISVALID
        IF CARD-NUM = CC-CARD-NUM-N
            CONTINUE
        ELSE
            SET WS-EXCLUDE-THIS-RECORD TO TRUE
            GO TO 9500-FILTER-RECORDS-EXIT
        END-IF
    END-IF
    .
9500-FILTER-RECORDS-EXIT.
    EXIT.

Example 3: Optimistic Lock and Rewrite — from COCRDUPC 9200-WRITE-PROCESSING

9200-WRITE-PROCESSING.
    MOVE CC-CARD-NUM TO WS-CARD-RID-CARDNUM

    *> Read with exclusive update lock
    EXEC CICS READ
         FILE(LIT-CARDFILENAME)
         UPDATE
         RIDFLD(WS-CARD-RID-CARDNUM)
         KEYLENGTH(LENGTH OF WS-CARD-RID-CARDNUM)
         INTO(CARD-RECORD)
         LENGTH(LENGTH OF CARD-RECORD)
         RESP(WS-RESP-CD)
         RESP2(WS-REAS-CD)
    END-EXEC

    IF WS-RESP-CD NOT EQUAL TO DFHRESP(NORMAL)
        SET COULD-NOT-LOCK-FOR-UPDATE TO TRUE
        GO TO 9200-WRITE-PROCESSING-EXIT
    END-IF

    *> Detect concurrent modification
    PERFORM 9300-CHECK-CHANGE-IN-REC
    IF DATA-WAS-CHANGED-BEFORE-UPDATE
        GO TO 9200-WRITE-PROCESSING-EXIT
    END-IF

    *> Build update record
    INITIALIZE CARD-UPDATE-RECORD
    MOVE CCUP-NEW-CARDID         TO CARD-UPDATE-NUM
    MOVE CC-ACCT-ID-N            TO CARD-UPDATE-ACCT-ID
    MOVE CCUP-NEW-CRDNAME        TO CARD-UPDATE-EMBOSSED-NAME
    STRING  CCUP-NEW-EXPYEAR '-' CCUP-NEW-EXPMON '-' CCUP-NEW-EXPDAY
            DELIMITED BY SIZE
       INTO CARD-UPDATE-EXPIRAION-DATE
    END-STRING
    MOVE CCUP-NEW-CRDSTCD        TO CARD-UPDATE-ACTIVE-STATUS

    *> Write updated record
    EXEC CICS REWRITE
         FILE(LIT-CARDFILENAME)
         FROM(CARD-UPDATE-RECORD)
         LENGTH(LENGTH OF CARD-UPDATE-RECORD)
         RESP(WS-RESP-CD)
         RESP2(WS-REAS-CD)
    END-EXEC

    IF WS-RESP-CD NOT EQUAL TO DFHRESP(NORMAL)
        SET LOCKED-BUT-UPDATE-FAILED TO TRUE
    END-IF
    .

Example 4: Embossed Name Validation — from COCRDUPC 1230-EDIT-NAME

1230-EDIT-NAME.
    *> Check for blank/empty
    IF CCUP-NEW-CRDNAME EQUAL LOW-VALUES
    OR CCUP-NEW-CRDNAME EQUAL SPACES
        SET INPUT-ERROR TO TRUE
        SET FLG-CARDNAME-BLANK TO TRUE
        GO TO 1230-EDIT-NAME-EXIT
    END-IF

    *> Only alphabetic characters and spaces allowed
    *> Strategy: copy name, convert all alpha to space, check if remainder is empty
    MOVE CCUP-NEW-CRDNAME TO CARD-NAME-CHECK
    INSPECT CARD-NAME-CHECK
      CONVERTING LIT-ALL-ALPHA-FROM   *> 'ABCDEFGHIJKLMNOPQRSTUVWXYZ '
              TO LIT-ALL-SPACES-TO    *> '                           '

    IF FUNCTION LENGTH(FUNCTION TRIM(CARD-NAME-CHECK)) = 0
        SET FLG-CARDNAME-ISVALID TO TRUE   *> All chars were alpha or space
    ELSE
        SET INPUT-ERROR TO TRUE
        SET FLG-CARDNAME-NOT-OK TO TRUE    *> Non-alpha character found
    END-IF
    .

User Stories

US-CCRD-01 · Simple (1–2 pts)

As a cardholder, I want to list all credit cards linked to my account so I can see which cards are active and manage each one.

Acceptance Criteria:

  • GIVEN I am signed in as a regular user with account context
  • WHEN I navigate to CCLI (from main menu)
  • THEN the screen displays up to 7 cards filtered to my account
  • AND each row shows account number, card number, and status
  • AND PF8/PF7 pages forward/backward through additional cards

US-CCRD-02 · Simple (1–2 pts)

As a cardholder, I want to view my card's full details including expiration date so I know when my card expires.

Acceptance Criteria:

  • GIVEN I am on the card list screen
  • WHEN I enter S in a row's selector and press ENTER
  • THEN I am taken to the card detail screen (CCDL)
  • AND I can see card number, CVV, embossed name, expiry date, and status
  • AND the fields are protected (read-only)

US-CCRD-03 · Medium (3–5 pts)

As a cardholder, I want to update my card's embossed name and active status so I can correct mistakes and activate or deactivate a card.

Acceptance Criteria:

  • GIVEN I am on the card update screen (CCUP)
  • WHEN I change the embossed name to valid alphabetic characters
  • AND I change the status to Y or N
  • AND I press ENTER twice (validate then confirm)
  • THEN the CARDDATA VSAM record is updated via CICS REWRITE
  • AND a success confirmation message is displayed
  • AND if another user modified the record between my reads, I am shown the latest values and asked to retry

US-CCRD-04 · Simple (1–2 pts)

As an administrator, I want to list all credit cards in the system so I can perform system-wide card management.

Acceptance Criteria:

  • GIVEN I am signed in as an admin user
  • WHEN I navigate to CCLI
  • THEN all CARDDATA records are displayed (not filtered to a single account)
  • AND I can apply an optional account filter or card filter via the search fields

Development Guidelines and Patterns

CICS Transaction Pattern (Standard for this Module)

  • Each program checks EIBCALEN on entry. If 0, it is a fresh transaction start — initialize all COMMAREA fields.
  • Programs re-enter themselves via EXEC CICS RETURN TRANSID(...) COMMAREA(...) to maintain state across user interactions.
  • Navigation to another program uses EXEC CICS XCTL PROGRAM(...) COMMAREA(...).
  • Always set CDEMO-FROM-PROGRAM, CDEMO-LAST-MAP, and CDEMO-LAST-MAPSET before transferring control.
  • PF key mapping is handled by the shared CSSTRPFY copybook (COPY inline) which reads EIBAID and sets CCARD-AID-* flags.

Input Validation Pattern

  • Each field has a dedicated edit paragraph (e.g., 1210-EDIT-ACCOUNT, 1220-EDIT-CARD, 1230-EDIT-NAME).
  • Flag fields (e.g., WS-EDIT-ACCT-FLAG) use level-88 conditions for readable state: FLG-ACCTFILTER-ISVALID, FLG-ACCTFILTER-NOT-OK, FLG-ACCTFILTER-BLANK.
  • On any validation failure, set INPUT-ERROR and the specific flag. The first error message wins (guarded by WS-RETURN-MSG-OFF).
  • Screen attribute bytes (DFHRED for error, DFHBMFSE for normal input) are set in screen attribute paragraphs after validation.

Screen Layout Conventions

  • Screen titles are stored in COTTL01Y literals (CCDA-TITLE01, CCDA-TITLE02).
  • Current date and time are populated from FUNCTION CURRENT-DATE into CSDAT01Y work area fields.
  • Transaction name and program name are shown on each screen for user context.
  • Error messages use ERRMSGO field; informational messages use INFOMSGO field.

VSAM Browse Pattern

*> Standard pattern for paginated VSAM browse in this module:

*> 1. Position RID at desired start key
MOVE start-key TO WS-CARD-RID-CARDNUM

*> 2. STARTBR (GTEQ = start at or after the key)
EXEC CICS STARTBR DATASET(file) RIDFLD(rid) GTEQ ... END-EXEC

*> 3. Loop READNEXT / READPREV until page full or ENDFILE
PERFORM UNTIL READ-LOOP-EXIT
    EXEC CICS READNEXT/READPREV DATASET(file) ... END-EXEC
    EVALUATE WS-RESP-CD ...
END-PERFORM

*> 4. Always ENDBR to release browse cursor
EXEC CICS ENDBR FILE(file) END-EXEC

Adding a New Editable Field to COCRDUPC

  1. Add the field to CVACT02Y.cpy (if new to the VSAM record — must adjust RECLN and FILLER).
  2. Add CCUP-OLD-xxx and CCUP-NEW-xxx fields to the update working storage.
  3. Add the new field to COCRDUP.bms BMS mapset and regenerate COCRDUP.CPY.
  4. Add an edit paragraph (e.g., 1270-EDIT-NEWFIELD) following the existing validation pattern.
  5. Call the new edit paragraph from 1200-EDIT-MAP-INPUTS.
  6. Include the new field in the 9300-CHECK-CHANGE-IN-REC comparison.
  7. Include the new field in the 9200-WRITE-PROCESSING update record construction.

Readiness Considerations

⚠️ PCI-DSS Risk: CVV Storage
CARD-CVV-CD (3-digit CVV code) is stored in plain text within the CARDDATA VSAM file. Payment Card Industry Data Security Standard (PCI-DSS) prohibits storing CVV after authorization. In any production or modernization deployment, this field must be removed from persistent storage or replaced with a tokenization strategy.
⚠️ No Card Number Masking
The full 16-digit card number (CARD-NUM) is displayed on the 3270 screen in CCLI and CCDL without masking. In a modernized interface, card numbers should be masked (e.g., **** **** **** 1234).

Technical Debt

  • RECLN spelling typo: CARD-EXPIRAION-DATE (missing T) is consistent across all copybooks and programs — do not fix without a coordinated update.
  • No soft delete: Deactivating a card (status N) is the only way to prevent use; there is no deletion from VSAM.
  • No card creation online: The module supports list/view/update only. New card records must be added via JCL batch load (CARDFILE.jcl) or a custom batch program.
  • No Alternate Index on ACCT-ID: Filtering cards by account number requires a full browse of CARDDATA with in-memory filtering — potential performance issue for large datasets.

Modernization Considerations

  • The three-transaction flow (list → detail → update) maps naturally to a RESTful API pattern: GET /cards?accountId=X, GET /cards/{cardNum}, PUT /cards/{cardNum}.
  • The COMMAREA-based state management can be replaced with stateless JWT tokens carrying account/card context.
  • VSAM KSDS can be migrated to a relational table with indexes on both card_num (PK) and acct_id (FK, indexed) to eliminate the full-browse filter workaround.
  • The optimistic concurrency control pattern (9300-CHECK-CHANGE-IN-REC) is directly equivalent to a row-version or ETag check in a REST API and should be preserved in modernized code.
