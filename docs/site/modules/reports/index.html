📊 Reports

User Story Development Guide — CardDemo Application

📋 Overview

Purpose: Allow users to request transaction reports by date range. The online program (CORPT00C) accepts report parameters and submits a batch JCL job via an extra-partition TDQ to produce printed output.

Business Context: Reports enable cardholders and admins to review transaction history for a billing period. Statement generation is also driven by batch report programs.

CORPT00C TranID: CR00 CBSTM03A/B TDQ-based batch submit

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Report Request: As a cardholder, I want to request a transaction report for a date range so that I can review my spending history.
  • Date Validation: As the system, I want to validate report date range so that end date is after start date and format is correct.
  • Statement Generation: As a batch operator, I want to generate monthly statements so that all cardholders receive their account summaries.

📊 Complexity Guidelines

ComplexityPointsExamples

🔧 Technical Foundation

🏗️ Key Components

  • CORPT00C.cbl: Online report request. Accepts date range parameters, validates format (YYYY-MM-DD), writes JCL to extra-partition TDQ for batch job submission. TranID: CR00.
  • CBSTM03A.CBL: Statement generation part A — reads TRANSACT, formats statement lines.
  • CBSTM03B.CBL: Statement generation part B — sorts and produces final statement output.
  • CORPT00.bms: Report request BMS screen (date range, report name inputs).

🔗 Dependencies

  • TRANSACT VSAM: Source of transaction data for all reports.
  • ACCTDAT VSAM: Account header data for statements.
  • Extra-partition TDQ: CICS mechanism for submitting JCL to internal reader.
  • Batch JCL (TRANREPT.jcl, CREASTMT.JCL): Actual report/statement batch jobs.

📊 Report Submission Pattern

User enters: Start Date (YYYY-MM-DD), End Date (YYYY-MM-DD)
CORPT00C validates dates
Builds JCL record string:
  "//TRANREPT JOB ..."
  "//PARM '&STARTDT=yyyy-mm-dd,&ENDDT=yyyy-mm-dd'"
Writes JCL to CICS TDQ (Internal Reader)
Batch job executes → SYSOUT report produced

✅ Acceptance Criteria Patterns

Report Request:
  • Given valid start and end dates in YYYY-MM-DD format
  • When user submits report request
  • Then JCL submitted to batch; confirmation message shown
Date Validation:
  • Given invalid date format or end date before start date
  • When user submits
  • Then error displayed; no batch job submitted
