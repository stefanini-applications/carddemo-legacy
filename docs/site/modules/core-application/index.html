📦 Core Application

Guide for Developing User Stories

📋 Overview

Purpose: Power the baseline CardDemo experience—sign-on, account/card lifecycle screens, transaction entry, bill payments, reports, and batch refreshes.

Business Context: This module must stay reliable under modernization tests; every CICS screen (CC00/CM00/CT00/CAUP, etc.) hooks into VSAM/KSDS data that downstream MQ or DB2 modules consume.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Management: As a customer, I want to view and edit the account record via CAUP so that my billing address stays current.
  • Transaction Processing: As a cardholder, I want to submit a transaction through CT02 so that it immediately writes to the daily transaction file.
  • Reporting: As a support analyst, I want the CREASTMT job to generate statements so that I can email monthly summaries to customers.

⚡ Development Acceleration Factors

  • Copybooks: Reuse app/cpy/CVACT01Y.cpy and CVTRA06Y.cpy for consistent record layouts.
  • BMS Patterns: Extend existing map sets (e.g., COACTUP, COTRN00) to add new fields; the COBOL SEND MAP logic can remain intact.
  • JCL Templates: Leverage jobs in app/jcl/ for dataset refresh and combine steps (ACCTFILE → CARDFILE → TRANBKP).

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new display or validation field on an existing CICS map without changing persistence.
  • Medium (3-5 pts): Hook a new field into the VSAM dataset, update copybooks, and adjust JCL copy steps.
  • Complex (5-8 pts): Change transaction routing plus add an MQ call or DB2 sync so downstream modules see new data.

🔧 Technical Foundation

🏗️ Key Components

  • CICS Transactions: CC00, CM00, CT00–CT02, CAUP, CCUP, CB00, CR00.
  • Copybooks: VSAM structures defined in app/cpy/ (account, card, transaction, user/security).
  • BMS Maps: Layout definitions in app/bms/ with map sets such as COSGN00, COACTUP, COCRDLI.
  • Batch Jobs: JCL files in app/jcl/ orchestrate dataset loads (ACCTFILE, CARDFILE, TRANBKP, CREASTMT).

🔗 Dependencies

  • VSAM Datasets: AWS.M2.CARDDEMO.USRSEC.PS, AWS.M2.CARDDEMO.ACCTDATA.PS, AWS.M2.CARDDEMO.DALYTRAN.PS.
  • RACF Security: User IDs (ADMIN001, USER0001) defined in CSUSR01Y.cpy.
  • Scheduler/Control: Jobs in app/scheduler/ and macros under app/maclib/.

🌐 Public Interfaces

CICS CC00 → COSGN00C (sign-on/menu navigation)
CICS CAUP → COACTUPC (account updates)
CICS CT00/CT01/CT02 → COTRN00C/COTRN01C/COTRN02C (transaction list/view/add)
CICS CB00 → COBIL00C (bill payment)
JCL ACCTFILE/CARDFILE/TRANBKP/CREASTMT → VSAM refresh and statement generation

✅ Acceptance Criteria Examples

  • Given ADMIN001 is on CA00, when they update account address, then AWS.M2.CARDDEMO.ACCTDATA.PS reflects the new address and subsequent CT02 entries respect the inactive flag.
  • Given a transaction is entered via CT02, when the job completes, then the same TRANSACTION-ID exists in DALYTRAN-RECORD and displays on CT01.
