📦 Core Online Experience

Guide for writing user stories that enhance the base CardDemo BMS/CICS experience for regular and administrative journeys.

📋 Overview

Purpose: Deliver the legacy CardDemo 3270 experience—user signon, menus, account/card management, transaction browsing, bill payments, and statements—driven by BMS maps and COBOL programs.

Business Context: Every customer-facing story touches the `CC00` signon flow, tab-delivers to `CM00`, and ultimately hits VSAM masters (`USERSEC`, `ACCTDATA`, `CARDDATA`, `DALYTRAN`). Consistency is enforced by copybooks such as `CVACT01Y`, `CVACT02Y`, `CVCUS01Y`, and `CVTRA05Y`.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Balance & Transactions: As a regular user, I want to view my account balance via `CAVW` and scroll with PF7/PF8 so that I can decide if I have room to pay a bill.
  • Payments: As a regular user, I want to create a bill payment through `CB00` that respects card status so that the daily `POSTTRAN` job posts accurate transactions.
  • Admin Ops: As a customer service agent, I want to review statements via `CBSTM03A` so that I can explain charges during calls.

⚡ Development Acceleration Factors

  • Copybook Reuse: Update `CVACT01Y`, `CVACT02Y`, `CVTRA05Y`, and `CVACT03Y` to propagate field changes across all screens.
  • BMS Map Library: `app/bms/COCRDLI`, `COACTVW`, `COPAU00`, etc., share layouts—duplicate and tweak only segments that change labels or PF key assignments.
  • PF Key Patterns: PF3 to exit, PF5 to search/mark, PF7/PF8 to page. Follow existing message-area and `MSGAREA` definitions in `CSMSG01Y`.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new label/field to `COACTVWC`/`COACTUPC` and update the corresponding copybook.
  • Medium (3-5 pts): Introduce a new transaction type that requires `COTRN02C` validation logic plus a downstream `TRANEXTR` feed to `transaction-type`.
  • Complex (5-8 pts): Merge a new MQ-based data source into an existing form (e.g., embed an authorization status panel guided by `authorization` module messaging).

🔧 Technical Foundation

🏗️ Key Components

  • BMS maps: `COSGN00C`, `COMEN01C`, `COACTVWC`, `COCRDUP`, `COTRN00C`, `CBIL00C`, `CBSTM03A`.
  • COBOL programs: `COSGN00C`, `COACTUPC`, `COCRDUPC`, `CBTRN02C`, `CBTRN03C` coordinate VSAM reads/writes.
  • Copybooks: `CVACT01Y`, `CVACT02Y`, `CVACT03Y`, `CVTRA05Y`, `CVCUS01Y` define record layouts.
  • VSAM masters: `USERSEC`, `ACCTDATA`, `CARDDATA`, `CUSTDATA`, `DALYTRAN` store live customer data.

🔗 Dependencies

  • core-batch: Relies on `TRANEXTR`, `POSTTRAN`, `CBSTM03A` to keep data fresh.
  • transaction-type: Uses `TRANSACTION_TYPE` metadata synchronized via `TRANEXTR`.
  • authorization: Optional `CPVS`/`CPVD` screens tie pending authorizations into account views.
  • scripts/run_full_batch.sh: Orchestrates dataset refresh before online signon.

🌐 Public Interfaces

CC00 / COSGN00C - Handle signon authentication (PF keys drive login flows)
CM00 / COMEN01C - Main menu dispatcher
CAVW / COACTVWC - Account view with field-level consistency checks
CAUP / COACTUPC - Account update (VSAM writeback)
COTRN0xC - Transaction list/view/add (DALYTRAN updates)
CB00 / COBIL00C - Bill payment and card validation
CBSTM03A - Statement generation per customer

🧪 Acceptance Criteria & Patterns

  • Data integrity: Every update must preserve `EXP-ACCT-CURR-BAL` vs. credit limit and succeed in the daily `POSTTRAN` run.
  • Validation: PF keys must trigger the `MSGAREA` defined in `CSMSG01Y` and display the correct BMS message text.
  • Notifications: Use BMS message area instead of global toast; update `CSMSG01Y` to change severity text.
