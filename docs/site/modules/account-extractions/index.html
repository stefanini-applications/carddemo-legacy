account-extractions

COBOL / CICS IBM MQ VSAM Optional Extension

Overview

The account-extractions module is an optional extension for the CardDemo credit card management platform. It demonstrates a MQ-driven request/response pattern that bridges asynchronous IBM MQ messaging with synchronous VSAM data access inside a CICS environment.

Two COBOL programs — CODATE01 (CICS transaction CDRD) and COACCT01 (CICS transaction CDRA) — act as MQ listeners. Each program reads incoming messages from a named MQ request queue, processes the request (either returning the current system date/time, or reading an account record from VSAM), and puts a reply on the corresponding MQ response queue.

This module is located under app/app-vsam-mq/ and is entirely independent of the core CardDemo online menus. It is intended for cloud integration scenarios where external clients (cloud services, data-lake pipelines, etc.) need to query mainframe data asynchronously without direct 3270 terminal sessions.

Architectural Context

The module sits at the MQ-bridge layer of the CardDemo architecture: 

External Client / Cloud Service
         │
         │  PUT message to CARDDEMO.REQUEST.QUEUE
         ▼
   IBM MQ Queue Manager
         │
         │  CICS trigger fires CDRD or CDRA transaction
         ▼
   CICS Transaction Server
         │
         ├─► CODATE01 ──► CICS ASKTIME / FORMATTIME ──► reply queue CARD.DEMO.REPLY.DATE
         │
         └─► COACCT01 ──► CICS READ (ACCTDAT VSAM file) ──► reply queue CARD.DEMO.REPLY.ACCT
                                    │
                                    ▼
                            VSAM KSDS (ACCTDAT)
                            sourced from CVACT01Y copybook

Both programs share the same structural skeleton:

  1. Open an error queue (CARD.DEMO.ERROR) for fault reporting.
  2. Retrieve the triggering queue name from the CICS trigger mechanism (EXEC CICS RETRIEVE INTO(MQTM)).
  3. Open the input queue (shared read, save-all-context).
  4. Open the output reply queue (output, pass-all-context).
  5. Loop: MQGET with a 5-second wait, process, MQPUT reply, SYNCPOINT.
  6. Close all queues and return via EXEC CICS RETURN.
Design note: The programs use MQGMO-SYNCPOINT and MQPMO-SYNCPOINT so that each GET/PUT pair is unit-of-work protected. CICS SYNCPOINT is issued at the top of the main loop to commit the previous message before fetching the next one.

CODATE01 — System Date Inquiry (Transaction CDRD)

Processing logic: When a message arrives, the program calls EXEC CICS ASKTIME to obtain the current absolute time, then EXEC CICS FORMATTIME to produce a MM-DD-YYYY date and HH:MM:SS time string. The formatted values are concatenated into the reply buffer before being put back onto the reply queue. 

* CODATE01 — 4000-PROCESS-REQUEST-REPLY (excerpt)
     EXEC CICS ASKTIME
          ABSTIME (WS-ABS-TIME)
     END-EXEC

     EXEC CICS FORMATTIME
          ABSTIME(WS-ABS-TIME)
          MMDDYYYY(WS-MMDDYYYY)
          DATESEP('-')
          TIME(WS-TIME)
          TIMESEP
     END-EXEC

     STRING  'SYSTEM DATE : ' WS-MMDDYYYY
             'SYSTEM TIME : ' WS-TIME
             DELIMITED BY SIZE
             INTO REPLY-MESSAGE
     END-STRING
     PERFORM 4100-PUT-REPLY

Processing logic: The request message is parsed into REQUEST-MSG-COPY which exposes WS-FUNC (4 chars) and WS-KEY (11-digit account number). If WS-FUNC = 'INQA' and WS-KEY > ZEROES, a direct VSAM READ is issued against the ACCTDAT dataset using the account ID as the record key. The account fields from the CVACT01Y copybook are moved into a formatted WS-ACCT-RESPONSE structure and put onto the reply queue. 

* COACCT01 — 4000-PROCESS-REQUEST-REPLY (excerpt)
     IF WS-FUNC = 'INQA' AND WS-KEY > ZEROES
        MOVE WS-KEY TO WS-CARD-RID-ACCT-ID

        EXEC CICS READ
             DATASET   (LIT-ACCTFILENAME)      *> 'ACCTDAT '
             RIDFLD    (WS-CARD-RID-ACCT-ID-X)
             KEYLENGTH (LENGTH OF WS-CARD-RID-ACCT-ID-X)
             INTO      (ACCOUNT-RECORD)
             LENGTH    (LENGTH OF ACCOUNT-RECORD)
             RESP      (WS-RESP-CD)
             RESP2     (WS-REAS-CD)
        END-EXEC

        EVALUATE WS-RESP-CD
            WHEN DFHRESP(NORMAL)
                 MOVE ACCT-ID               TO WS-ACCT-ID
                 MOVE ACCT-ACTIVE-STATUS    TO WS-ACCT-ACTIVE-STATUS
                 MOVE ACCT-CURR-BAL         TO WS-ACCT-CURR-BAL
                 MOVE ACCT-CREDIT-LIMIT     TO WS-ACCT-CREDIT-LIMIT
                 MOVE ACCT-CASH-CREDIT-LIMIT TO WS-ACCT-CASH-CREDIT-LIMIT
                 MOVE ACCT-OPEN-DATE        TO WS-ACCT-OPEN-DATE
                 MOVE ACCT-EXPIRAION-DATE   TO WS-ACCT-EXPIRAION-DATE
                 MOVE ACCT-REISSUE-DATE     TO WS-ACCT-REISSUE-DATE
                 MOVE ACCT-CURR-CYC-CREDIT  TO WS-ACCT-CURR-CYC-CREDIT
                 MOVE ACCT-CURR-CYC-DEBIT   TO WS-ACCT-CURR-CYC-DEBIT
                 MOVE ACCT-GROUP-ID         TO WS-ACCT-GROUP-ID
                 MOVE WS-ACCT-RESPONSE      TO REPLY-MESSAGE
                 PERFORM 4100-PUT-REPLY
            WHEN DFHRESP(NOTFND)
                 STRING 'INVALID REQUEST PARAMETERS '
                        'ACCT ID : ' WS-KEY
                        DELIMITED BY SIZE INTO REPLY-MESSAGE
                 PERFORM 4100-PUT-REPLY
        END-EVALUATE
     END-IF

Both programs use the same request buffer layout. A caller places a 1000-byte message on the configured input queue. The program reads the first 15 bytes to determine the function and the key, then replies on the corresponding reply queue.

Business Rules

  1. Function code validation (COACCT01 only): The request is only processed if WS-FUNC = 'INQA' AND WS-KEY > ZEROES. Any other combination returns an INVALID REQUEST PARAMETERS error reply rather than raising an exception.
  2. VSAM key lookup: The account ID from the request is used as a direct VSAM RIDFLD key. A DFHRESP(NOTFND) response is reported gracefully in the MQ reply; other VSAM errors route to the error queue and terminate.
  3. MQ wait interval: Both programs set MQGMO-WAITINTERVAL = 5000 (5 seconds). If no message arrives in that window, MQRC-NO-MSG-AVAILABLE causes the loop to exit gracefully via NO-MORE-MSGS.
  4. Correlation / reply-to: The program captures MQMD-MSGID, MQMD-CORRELID, and MQMD-REPLYTOQ from the inbound message descriptor. These are preserved in SAVE-MSGID, SAVE-CORELID, SAVE-REPLY2Q and propagated to the outbound MQMD so the caller can correlate responses.
  5. Syncpoint discipline: A CICS SYNCPOINT is issued at the start of each loop iteration (4000-MAIN-PROCESS) to commit the prior GET+PUT pair before the next message is dequeued, preventing duplicate processing on restart.
  6. Error routing: MQ API failures (MQOPEN, MQGET, MQPUT) are written to CARD.DEMO.ERROR and cause immediate program termination. VSAM errors other than NOTFND also route to the error queue.
  7. Date format: Dates returned by CODATE01 use MM-DD-YYYY format with DATESEP('-') and times use HH:MM:SS with the default CICS time separator.

CODATE01 End-to-End Flow

Client
  │
  ├─ 1. PUT message to CARDDEMO.REQUEST.QUEUE (any content, REPLYTOQ set)
  │
MQ Trigger Monitor
  │
  ├─ 2. Trigger fires CICS transaction CDRD
  │
CODATE01 (1000-CONTROL)
  │
  ├─ 3. Open CARD.DEMO.ERROR queue (error sink)
  ├─ 4. EXEC CICS RETRIEVE → get input queue name from trigger message
  ├─ 5. Set reply queue = 'CARD.DEMO.REPLY.DATE'
  ├─ 6. MQOPEN input queue  (MQOO-INPUT-SHARED + MQOO-SAVE-ALL-CONTEXT)
  ├─ 7. MQOPEN output queue (MQOO-OUTPUT + MQOO-PASS-ALL-CONTEXT)
  │
  └─ Loop (4000-MAIN-PROCESS):
       ├─ 8. EXEC CICS SYNCPOINT  (commit previous UoW)
       ├─ 9. MQGET with 5-second wait
       │      ├─ if MQCC-OK → save MSG-ID, CORRELID, REPLYTOQ
       │      │                PERFORM 4000-PROCESS-REQUEST-REPLY
       │      │                   ├─ ASKTIME → WS-ABS-TIME
       │      │                   ├─ FORMATTIME → WS-MMDDYYYY, WS-TIME
       │      │                   ├─ STRING reply text into REPLY-MESSAGE

       │      └─ if MQRC-NO-MSG-AVAILABLE → SET NO-MORE-MSGS TO TRUE → exit loop
       │
  ├─ 10. Close input, output, error queues
  └─ 11. EXEC CICS RETURN

Development Guidelines

Adding a New Request Type

  1. Add a new 88 level or EVALUATE branch in 4000-PROCESS-REQUEST-REPLY for the new function code.
  2. Populate REPLY-MESSAGE with the response text.
  3. Call PERFORM 4100-PUT-REPLY.
  4. If the new type requires a different reply queue, parameterize the reply queue name rather than hardcoding it.

Changing Queue Names

Reply and error queue names are hardcoded constants in the PROCEDURE DIVISION. To change them, update the MOVE '...' TO REPLY-QUEUE-NAME and MOVE '...' TO ERROR-QUEUE-NAME statements, then recompile. Consider externalizing these as CICS DEFINE PROGRAM attributes or a configuration record in the future.

Performance Considerations

  • The 5-second MQGMO-WAITINTERVAL means a quiet queue will hold a CICS task for up to 5 seconds before exiting. Tune this value based on expected message rates.
  • The MQ buffer is fixed at 1000 bytes. Account response payloads are well under this limit; no dynamic allocation is needed.
  • Target MQ round-trip: < 1 second under normal load (per system performance budgets).

Error Handling Patterns

  • MQ failures immediately route to 9000-ERROR (writes to error queue) then 8000-TERMINATION (closes queues, issues CICS RETURN).
  • VSAM NOTFND is handled gracefully — a plain-text error is returned to the caller rather than causing program termination.
  • Do not add DISPLAY statements for production — they route to the CICS system log. Use the error queue pattern instead.

📦 Account Extractions

Guide for Developing User Stories

📋 Overview

Purpose: Demonstrate MQ↔VSAM integrations for system date inquiries and account-data requests through the CDRD and CDRA transactions.

Business Context: Supports modernization scenarios where downstream services ask the mainframe for either time synchronization or account details without logging into a screen.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • System Clocks: As an integration tester, I want to send a DATE MQ request so that the target system can reconcile cross-platform timestamps.
  • Account Lookups: As a partner API, I want to provide an account number to CDRA and receive structured customer data so that I can keep loyalty programs in sync.
  • Async Tracking: As an operations owner, I want correlation IDs logged for every request so retries do not duplicate processing.

⚡ Development Acceleration Factors

  • Message Schemas: Request/response templates (DATE and ACCT structures) are documented and easy to replicate for new MQ payloads.
  • MQ Resources: `CARDDEMO.REQUEST.QUEUE`/`CARDDEMO.RESPONSE.QUEUE` definitions under app/app-vsam-mq/csd/ provide ready-to-use connection info.
  • BMS & COBOL: Programs CODATE01 and COACCT01 already show how to map MQ responses back into BMS displays.

📊 Complexity Guidelines

  • Simple: Adjust the account response format to include one new field.
  • Medium: Add a new MQ listener that handles an additional request type and enriches VSAM data.
  • Complex: Introduce additional MQ queues plus correlation handling and integrate the response with another module (e.g., authorization).

🔧 Technical Foundation

🏗️ Key Components

  • Transactions: CDRD for system date inquiries, CDRA for account details.
  • COBOL Programs: CODATE01 and COACCT01 implement MQ request/response logic.
  • MQ Configuration: Queues CARDDEMO.REQUEST.QUEUE and CARDDEMO.RESPONSE.QUEUE defined under app/app-vsam-mq/csd/ plus MQCONN/MQQUEUE resources.
  • VSAM Lookup: Programs read from `AWS.M2.CARDDEMO.ACCTDATA.PS` (account copybook `CVACT01Y`) to build account responses.

🔗 Dependencies

  • IBM MQ: Requires an MQ channel accessible to CICS; refer to app/app-vsam-mq/README.md for queue names.
  • VSAM Data: The account detail payload relies on `CVACT01Y` fields for values such as `ACCT-CURR-BAL`.
  • CICS Maps: Responses map back into `app/bms/` screens via standard `SEND MAP` logic.

🌐 Public Interfaces

CICS CDRD → CODATE01 (system date MQ request)
CICS CDRA → COACCT01 (account detail MQ request)
MQ CARDDEMO.REQUEST.QUEUE → handles DATE/ACCT request payloads
MQ CARDDEMO.RESPONSE.QUEUE → returns SYSTEM-DATE or ACCOUNT-DATA records

✅ Acceptance Criteria Examples

  • Given an MQ request labeled DATE, when CODATE01 sends the response, then the response contains SYSTEM-DATE and the original REQUEST-ID.
  • Given a valid account number in an ACCT request, when COACCT01 runs, then the response returns fields such as ACCT-CURR-BAL and ACCT-ACTIVE-STATUS as defined in `CVACT01Y.cpy`.
