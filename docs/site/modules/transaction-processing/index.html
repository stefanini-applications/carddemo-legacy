💸 Transaction Processing Module

User Story Development Guide — List, View, Add & Report Transactions

📋 Overview

Purpose: View transaction history by account, view individual transaction details, add new manual transactions, and generate transaction reports from the CICS online environment.

Business Context: Transactions are the core financial events in CardDemo. This module handles the online (real-time) side — cardholders browsing their history and adding manual entries. The batch side (POSTTRAN, INTCALC) posts daily transactions and calculates interest. Reports are submitted to batch from the online CR00 transaction.

CICS Transactions: CT00 (list), CT01 (view), CT02 (add), CR00 (report)

VSAM Dataset: AWS.M2.CARDDEMO.TRANSACT.VSAM.KSDS (KSDS with AIX, 350-byte records)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Transaction List: As a cardholder, I want to view my transaction history so that I can track and reconcile my spending.
  • Transaction Detail: As a cardholder, I want to view the details of a specific transaction (merchant, amount, date) so that I can verify charges on my account.
  • Manual Entry: As a cardholder, I want to add a manual transaction entry so that out-of-band purchases are recorded in the system.
  • Report: As a cardholder, I want to generate a transaction report for a date range so that I can review my activity for a billing period.
  • Dispute: As a cardholder, I want to flag a suspicious transaction so that I can initiate a dispute investigation.

⚡ Development Acceleration Factors

  • CVTRA05Y copybook: Pre-defined 350-byte TRAN-RECORD — 16-char TRAN-ID as primary key
  • VSAM KSDS with AIX: Transaction lookup by ID or alternate index (card number) — both access modes available
  • Existing screen BMS maps: COTRN00–02.bms cover all three online screens
  • Transaction type + category codes: Reference VSAM files TRANTYPE and TRANCATG for descriptions
  • Report job already exists: TRANREPT (CBTRN03C) batch job handles report generation — online just submits it

📊 Complexity Guidelines

  • 1–2 pts List/view transactions by account (read VSAM, display)
  • 3–5 pts Add manual transaction with type/category validation; paged list with navigation
  • 5–8 pts Transaction dispute workflow, cross-reference to merchant data, multi-file date-range filtering, report generation trigger

🗄️ Transaction Record Structure

01  TRAN-RECORD.
    05  TRAN-ID            PIC X(16).      -- Primary key
    05  TRAN-TYPE-CD       PIC X(02).      -- e.g. "PR" (purchase), "CR" (credit)
    05  TRAN-CAT-CD        PIC 9(04).      -- Category (e.g. 0001=groceries)
    05  TRAN-SOURCE        PIC X(10).      -- Source system identifier
    05  TRAN-DESC          PIC X(100).     -- Description
    05  TRAN-AMT           PIC S9(09)V99.  -- Amount (+ debit, - credit)
    05  TRAN-MERCHANT-ID   PIC 9(09).
    05  TRAN-MERCHANT-NAME PIC X(50).
    05  TRAN-MERCHANT-CITY PIC X(50).
    05  TRAN-MERCHANT-ZIP  PIC X(10).
    05  TRAN-CARD-NUM      PIC X(16).      -- Card used (links to CARDXREF)
    05  TRAN-ORIG-TS       PIC X(26).      -- YYYY-MM-DD HH:MM:SS.mmmmmm
    05  TRAN-PROC-TS       PIC X(26).      -- Processing timestamp

🔗 Dependencies

  • card-management: TRAN-CARD-NUM must resolve via CARDXREF to an account
  • account-management: Transaction amounts update ACCT-CURR-BAL (via batch POSTTRAN)
  • batch-processing: POSTTRAN reads DALYTRAN and writes to TRANSACT.VSAM.KSDS; INTCALC uses category balances
  • transaction-type-management (optional): TRAN-TYPE-CD validated against TRANTYPE reference data

✅ Acceptance Criteria Patterns

  • Given a valid account, when accessing CT00, then transactions are listed in reverse chronological order
  • Given a transaction ID, when accessing CT01, then all 350-byte record fields are displayed correctly
  • Given new transaction via CT02, when card number is invalid (not in CARDXREF), then system rejects with error
  • Given new transaction via CT02, when amount is zero or negative (for a debit), then system rejects with validation error
  • Given valid transaction add via CT02, then TRAN-ORIG-TS is set to current system timestamp
  • Given report request CR00, when submitted, then batch job TRANREPT is triggered and completion message displayed
