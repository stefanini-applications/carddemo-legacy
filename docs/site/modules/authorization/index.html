🔒 Authorization

User Story Development Guide — CardDemo Application (Extension Module)

📋 Overview

Purpose: Real-time credit card authorization processing. External POS/systems send authorization requests via IBM MQ. COPAUA0C processes requests, applies business rules (credit check, fraud detection), stores decisions in IMS DB, and sends responses via MQ reply queue.

Business Context: Authorization is the gatekeeper for all card-present and card-not-present transactions. It decides in real-time whether a transaction is approved or declined, integrating with VSAM account data, IMS DB for audit, and DB2 for fraud analytics.

COPAUA0C TranID: CP00 IBM MQ IMS DB DB2: AUTHFRDS

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Authorization Request: As a merchant terminal, I want to submit a card authorization request via MQ so that I can verify the card before completing a sale.
  • Authorization Response: As a merchant terminal, I want to receive an approval or decline response so that I can complete or reject the transaction at the point of sale.
  • View Pending Authorizations: As a cardholder, I want to view pending authorizations on my account so that I can track recent transactions before they post.
  • Fraud Detection: As a fraud analyst, I want authorization decisions stored in DB2 so that I can analyze patterns and detect suspicious activity.
  • Auth Purge: As a batch operator, I want expired authorizations purged so that the IMS database does not grow unbounded.

🔧 Technical Foundation

🏗️ Key Components

  • COPAUA0C.cbl: Core authorization decision program. MQ-triggered (CP00). Reads up to 500 messages per trigger. Validates card/account, applies rules, writes to IMS, sends MQ response.
  • COPAUS0C.cbl: Authorization summary view screen for accounts.
  • COPAUS1C.cbl: Authorization detail view.
  • COPAUS2C.cbl: Authorization navigation/search.
  • CBPAUP0C.cbl: Batch purge of expired authorization records from IMS DB.

🔗 Dependencies

  • IBM MQ: Request queue (CP00 triggered by message arrival), reply queue for responses.
  • IMS DB (DBPAUTP0): Hierarchical storage for authorization records (parent: account, child: auth detail).
  • DB2 (AUTHFRDS): Fraud detection data store.
  • ACCTDAT / CUSTDAT / CARDDAT / CCXREF VSAM: Account, customer, card lookup during authorization.

📊 Authorization Decision Logic

Receive MQ auth request → extract card number + amount
Lookup CCXREF → resolve ACCT-ID + CUST-ID
Read ACCTDAT → get ACCT-ACTIVE-STATUS, ACCT-CURR-BAL, ACCT-CREDIT-LIMIT
Check fraud flags in IMS / DB2

Decision:
  APPROVED if:
    ACCT-ACTIVE-STATUS = 'Y'
    AND (ACCT-CREDIT-LIMIT - ACCT-CURR-BAL) >= requested amount
    AND no fraud flags
  DECLINED otherwise (with reason code)

Write auth record to IMS DB (DBPAUTP0)
Write fraud data to DB2 (AUTHFRDS) if applicable
Send response to MQ reply queue (APPROVED/DECLINED + amount + timestamp)

✅ Acceptance Criteria Patterns

Approved Authorization:
  • Given active account with sufficient available credit and no fraud flags
  • When MQ auth request received
  • Then APPROVED response sent to reply queue; auth record written to IMS DB
Declined — Insufficient Credit:
  • Given requested amount exceeds available credit
  • When MQ auth request received
  • Then DECLINED response sent with reason "Insufficient Credit"; auth record written to IMS DB
Auth View:
  • Given account has authorization records in IMS DB
  • When user navigates to auth view (COPAUS0C)
  • Then list of authorizations displayed with status, amount, and timestamp
