🛡️ Authorization Module

User Story Development Guide — Real-Time Card Authorization via MQ/IMS/DB2  Optional Module

📋 Overview

Purpose: Simulates real-time credit card authorization processing. Authorization requests arrive via IBM MQ, are processed by COPAUA0C, stored in IMS hierarchical database, and logged in DB2 for fraud analytics. Includes online views for pending authorizations and a batch purge job.

Business Context: This optional module extends CardDemo to showcase multi-technology integration (MQ + IMS + DB2 + VSAM + CICS) in a realistic authorization workflow. It demonstrates two-phase commit patterns across IMS and DB2, as well as MQ trigger-based program invocation.

Prerequisites: IMS DB configured (DBPAUTP0), DB2 table AUTHFRDS created, MQ queues defined, base CardDemo operational.

CICS Transactions: CP00 (MQ processor), CPVS (pending auth summary), CPVD (pending auth detail)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Authorization Request: As a merchant system, I want to submit card authorization requests via MQ so that transactions can be approved or declined in real time.
  • Pending View: As a cardholder, I want to view pending authorizations on my account so that I can see which charges are holds.
  • Authorization Detail: As a cardholder, I want to see the details of a pending authorization (merchant, amount, date) so that I can verify or dispute it.
  • Fraud Analytics: As a fraud analyst, I want authorization events logged in DB2 so that I can run queries to identify suspicious patterns.
  • Batch Purge: As a system operator, I want expired authorizations automatically purged from IMS so that storage is managed and data remains current.

⚡ Development Acceleration Factors

  • MQ trigger pattern established: COPAUA0C demonstrates the complete MQ request/response pattern
  • IMS DLI patterns: IMSFUNCS.cpy provides IMS function call patterns
  • Existing BMS maps: COPAU00.bms and COPAU01.bms define the pending auth screens
  • DB2 schema ready: AUTHFRDS.ddl defines the fraud reporting table
  • IMS DB defined: DBPAUTP0.dbd and PSB files already configured

📊 Complexity Guidelines

  • 1–2 pts View pending authorization list (read IMS, display CPVS screen)
  • 3–5 pts View authorization detail (CPVD); update IMS status; insert DB2 fraud record
  • 5–8 pts Full MQ authorization pipeline (request→validate→IMS insert→MQ response); batch purge; new fraud rule in DB2

🔄 Authorization Flow

1. Merchant system → sends auth request to CARDDEMO.REQUEST.QUEUE (MQ)
2. MQ trigger → invokes COPAUA0C (CICS CP00 transaction)
3. COPAUA0C validates card via VSAM CARDXREF
4. COPAUA0C inserts authorization record into IMS DB (DBPAUTP0)
5. COPAUA0C sends approval/decline response to CARDDEMO.RESPONSE.QUEUE
6. Cardholder views pending auths via CPVS (COPAUS0C)
7. Admin/cardholder views detail via CPVD (COPAUS1C)
   → COPAUS1C updates IMS status + inserts fraud record to DB2 AUTHFRDS
8. Nightly: CBPAUP0J runs CBPAUP0C to purge expired IMS records

🔗 Dependencies

  • IBM MQ: CARDDEMO.REQUEST.QUEUE and CARDDEMO.RESPONSE.QUEUE must be defined
  • IMS DB: DBPAUTP0 database and PSBs loaded; IMS region running
  • DB2: AUTHFRDS table created (AUTHFRDS.ddl)
  • card-management: VSAM CARDXREF used for card validation in authorization request
  • account-management: VSAM ACCTDATA used to verify account status and available credit

✅ Acceptance Criteria Patterns

  • Given valid auth request on MQ, when CP00 processes it, then IMS record is inserted and response sent to reply queue
  • Given invalid card number in auth request, when processed, then decline response sent to reply queue and record flagged in IMS
  • Given pending authorizations exist, when cardholder accesses CPVS, then all pending records for their account are listed
  • Given authorization detail viewed via CPVD, then DB2 AUTHFRDS record is created with merchant, amount, and decision data
  • Given authorization is older than retention period, when CBPAUP0J runs, then IMS record is deleted
  • Given IMS unavailable, when auth request processed, then error response sent to MQ reply queue; no partial commits
