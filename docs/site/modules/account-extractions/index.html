📦 Account Extractions

Guide for Developing User Stories

📋 Overview

Purpose: Demonstrate MQ↔VSAM integrations for system date inquiries and account-data requests through the CDRD and CDRA transactions.

Business Context: Supports modernization scenarios where downstream services ask the mainframe for either time synchronization or account details without logging into a screen.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • System Clocks: As an integration tester, I want to send a DATE MQ request so that the target system can reconcile cross-platform timestamps.
  • Account Lookups: As a partner API, I want to provide an account number to CDRA and receive structured customer data so that I can keep loyalty programs in sync.
  • Async Tracking: As an operations owner, I want correlation IDs logged for every request so retries do not duplicate processing.

⚡ Development Acceleration Factors

  • Message Schemas: Request/response templates (DATE and ACCT structures) are documented and easy to replicate for new MQ payloads.
  • MQ Resources: `CARDDEMO.REQUEST.QUEUE`/`CARDDEMO.RESPONSE.QUEUE` definitions under app/app-vsam-mq/csd/ provide ready-to-use connection info.
  • BMS & COBOL: Programs CODATE01 and COACCT01 already show how to map MQ responses back into BMS displays.

📊 Complexity Guidelines

  • Simple: Adjust the account response format to include one new field.
  • Medium: Add a new MQ listener that handles an additional request type and enriches VSAM data.
  • Complex: Introduce additional MQ queues plus correlation handling and integrate the response with another module (e.g., authorization).

🔧 Technical Foundation

🏗️ Key Components

  • Transactions: CDRD for system date inquiries, CDRA for account details.
  • COBOL Programs: CODATE01 and COACCT01 implement MQ request/response logic.
  • MQ Configuration: Queues CARDDEMO.REQUEST.QUEUE and CARDDEMO.RESPONSE.QUEUE defined under app/app-vsam-mq/csd/ plus MQCONN/MQQUEUE resources.
  • VSAM Lookup: Programs read from `AWS.M2.CARDDEMO.ACCTDATA.PS` (account copybook `CVACT01Y`) to build account responses.

🔗 Dependencies

  • IBM MQ: Requires an MQ channel accessible to CICS; refer to app/app-vsam-mq/README.md for queue names.
  • VSAM Data: The account detail payload relies on `CVACT01Y` fields for values such as `ACCT-CURR-BAL`.
  • CICS Maps: Responses map back into `app/bms/` screens via standard `SEND MAP` logic.

🌐 Public Interfaces

CICS CDRD → CODATE01 (system date MQ request)
CICS CDRA → COACCT01 (account detail MQ request)
MQ CARDDEMO.REQUEST.QUEUE → handles DATE/ACCT request payloads
MQ CARDDEMO.RESPONSE.QUEUE → returns SYSTEM-DATE or ACCOUNT-DATA records

✅ Acceptance Criteria Examples

  • Given an MQ request labeled DATE, when CODATE01 sends the response, then the response contains SYSTEM-DATE and the original REQUEST-ID.
  • Given a valid account number in an ACCT request, when COACCT01 runs, then the response returns fields such as ACCT-CURR-BAL and ACCT-ACTIVE-STATUS as defined in `CVACT01Y.cpy`.
