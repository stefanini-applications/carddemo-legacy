🛡️ Authorization Optional Module

Real-Time Credit Card Authorization — MQ + IMS DB + DB2

📋 Overview

Module ID: authorization

Purpose: Optional extension that adds real-time credit card authorization processing via IBM MQ. Authorization requests arrive via MQ, are processed by COPAUA0C, decisions stored in IMS DB, fraud analytics logged to DB2. Users can view pending authorizations (CPVS/CPVD).

ℹ️ Installation required: This module requires IMS DB setup, DB2 CARDAUTHDB, and MQ configuration. See app/app-authorization-ims-db2-mq/README.md for setup instructions.

🎯 User Story Templates

As a merchant POS system, I want to send authorization requests via MQ so that card transactions can be approved in real-time.
As a regular user, I want to view pending authorizations on my account (CPVS) so that I can see pre-authorized charges.
As a regular user, I want to view the details of a specific authorization (CPVD) so that I have full information about a pending charge.
As a fraud analyst, I want declined authorizations logged to DB2 so that fraud patterns can be identified and analyzed.
As a batch system, I want expired authorizations purged (CBPAUP0J) so that old pending records don't accumulate.

✅ Acceptance Criteria Patterns

MQ Processing (CP00): Given authorization request message on MQ input queue, when COPAUA0C is triggered, then: account validated via VSAM XREF, business rules applied (credit limit, card status), approval/decline decision stored in IMS DB, response sent to MQ reply queue.
Pending Auth List (CPVS): Given ACCT-ID in COMMAREA, when CPVS loads, then all pending authorizations for account are listed from IMS DB showing: merchant, amount, date, status.
Authorization Detail (CPVD): Given user selects authorization, when CPVD loads, then full authorization details are shown; viewing updates IMS DB status and inserts audit record to DB2.
Auth Expiry Purge (CBPAUP0J): Given expired authorization records in IMS DB, when CBPAUP0C runs, then records past expiry date are deleted from IMS DB.
Decline Logging: Given authorization is declined (insufficient credit, inactive card), then decline reason is logged to DB2 for fraud analytics.

🔗 Dependencies

  • accounts: ACCTFILE/XREFFILE used for account validation
  • credit-cards: CARDFILE checked for card active status
  • IMS DB: Must be configured with CARDAUTHDB database
  • DB2: Must have fraud analytics tables provisioned
  • MQ: Queue manager and request/reply queues must exist
  • VSAM XREFFILE: Card-to-account cross-reference for auth lookup
