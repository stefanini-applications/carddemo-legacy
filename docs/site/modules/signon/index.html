🔐 Signon Module

User Story Development Guide — Authentication & Session Management

📋 Overview

Purpose: Authenticates users against the VSAM user security file and routes them to the appropriate menu based on their role (regular user or admin).

Business Context: Entry point for all CardDemo interactions. Every user — cardholder or administrator — must authenticate before accessing any function. The signon module determines which capabilities are exposed based on the SEC-USR-TYPE field.

CICS Transactions: CC00 (signon), CM00 (main menu)

VSAM Dataset: AWS.M2.CARDDEMO.USRSEC.PS (keyed sequential, 80-byte records)

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Authentication: As a [cardholder / admin], I want to sign in with my user ID and password so that I can access the system securely.
  • Role Routing: As an admin, I want to be directed to the Admin Menu after login so that I can access administrative functions without extra navigation.
  • Security: As a security officer, I want invalid credentials to be rejected with an error message so that unauthorized access is prevented.
  • Session Exit: As a user, I want to exit the application from the main menu so that my session is properly terminated.

⚡ Development Acceleration Factors

  • CSUSR01Y copybook: Pre-defined 80-byte user record structure — no new data model needed
  • VSAM KSDS access pattern: Standard COBOL READ with key lookup — well-established pattern across the codebase
  • CICS COMMAREA: Session state passed via COMMAREA (COCOM01Y) — follow existing pattern
  • BMS screen already defined: COSGN00.bms provides the signon screen layout

📊 Complexity Guidelines

  • 1–2 pts Display signon screen, read user record, validate credentials, route to menu
  • 3–5 pts Add failed login counter, add CAPTCHA logic, add session timeout warning
  • 5–8 pts Integrate external identity provider, implement MFA, add audit logging to DB2

🗄️ Data Structure

01 SEC-USER-DATA.
   05 SEC-USR-ID      PIC X(08).   -- User ID (key)
   05 SEC-USR-FNAME   PIC X(20).   -- First name
   05 SEC-USR-LNAME   PIC X(20).   -- Last name
   05 SEC-USR-PWD     PIC X(08).   -- Password (plain text)
   05 SEC-USR-TYPE    PIC X(01).   -- 'U'=user, 'A'=admin
   05 SEC-USR-FILLER  PIC X(23).

🔗 Dependencies

  • VSAM USRSEC file: Must be open in CICS (JCL: OPENFIL) before signon works
  • All other modules: Depend on signon for session establishment
  • COMMAREA (COCOM01Y): Carries user context to all downstream transactions

✅ Acceptance Criteria Patterns

  • Given valid credentials, when user signs in, then system routes type=U to CM00 and type=A to CA00
  • Given invalid credentials, when user signs in, then system displays an error message and remains on CC00
  • Given blank user ID or password, when user attempts to sign in, then system displays field-level validation error
  • Given USRSEC file is unavailable, when signon is attempted, then system displays a file I/O error code
  • Given a successful login, when user navigates away and returns, then session context (user ID, type) is preserved in COMMAREA
