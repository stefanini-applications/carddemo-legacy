Optional Module CP00 · CPVS · CPVD · IMS · DB2 · MQ

📋 Overview

The Authorization module is an optional IMS/DB2/MQ extension that adds real-time card authorization processing to CardDemo. It receives authorization requests from a cloud POS emulator via IBM MQ, processes them against VSAM account data, stores results in an IMS HIDAM database, and supports fraud marking in DB2.

Business Context

A cloud POS emulator sends CSV-formatted authorization requests to the MQ queue AWS.M2.CARDDEMO.PAUTH.REQUEST. COPAUA0C (CP00) is MQ-triggered, processes each request by looking up the account via the CARDXREF VSAM AIX, applies configurable business rules (credit limit check, card status, expiry), and sends an approval/decline response to the AWS.M2.CARDDEMO.PAUTH.REPLY queue. The authorization summary is stored in the IMS HIDAM database for review. Fraud analysts can mark suspicious authorizations via CPVD (PF5), which triggers a DB2 insert in AUTHFRDS.

Story Templates

POS System Sends Authorization Request via MQ
As a point-of-sale system, I want to send a card authorization request via MQ and receive an approval or decline response so that I can complete or refuse the transaction.

COPAUA0C triggered by MQ message on PAUTH.REQUEST queue; performs VSAM CARDXREF lookup, applies business rules, sends response to PAUTH.REPLY queue, stores summary in IMS PAUTSUM0 segment
Cardholder Views Pending Authorizations
As a cardholder, I want to see a list of recent authorizations on my card so that I can verify activity.

COPAUS0C (CPVS) — reads IMS PAUTSUM0 root segments for the account; displays authorization summary list
Fraud Analyst Marks Suspicious Authorization
As a fraud analyst, I want to flag a suspicious authorization so that it is recorded in the fraud database for investigation.

COPAUS1C (CPVD) — view auth detail; press PF5 to trigger COPAUS2C, which inserts a record into DB2 AUTHFRDS table with CARD_NUM, AUTH_TS, AUTH_FRAUD='Y', TRANSACTION_AMT
Batch Purges Expired Authorizations
As a batch operations manager, I want expired and declined authorizations purged nightly so that IMS storage is managed and unused credit holds are released.

CBPAUP0C — reads IMS PAUTSUM0/PAUTDTL1 segments, identifies expired records, deletes from IMS, restores ACCT-CASH-CREDIT-LIMIT to ACCTDATA VSAM for declined/expired holds
Risk Manager Views Fraud Cases
As a risk manager, I want to query all authorizations flagged as fraudulent so that I can manage risk exposure.

DB2 SELECT from AUTHFRDS table filtered by AUTH_FRAUD='Y'; new report program or DB2 query tool

Acceleration Factors

  • COPAUA0C MQ read pattern (MQGET) — complete MQ receive, parse, and respond pattern; reusable for any MQ-triggered processing story
  • IMS GU/GN call patterns — COPAUS0C/1C demonstrate IMS Get Unique and Get Next for HIDAM navigation
  • COPAUS2C DB2 INSERT pattern — static embedded SQL insert with SQLCA error handling; template for any DB2 write story
  • CSV parsing from MQ message — field extraction from comma-delimited MQ message body is reusable for any CSV-over-MQ story
  • CBPAUP0C purge/credit-restore pattern — IMS delete + VSAM update in same batch step is reusable for any cleanup/restore story

✅ Acceptance Criteria Patterns

MQ Authorization Processing (CP00)

  • GIVEN a valid authorization request on the MQ queue, WHEN COPAUA0C processes it, THEN an approval or decline response is placed on PAUTH.REPLY within 5 seconds
  • GIVEN an authorization request, WHEN COPAUA0C processes it, THEN a PAUTSUM0 IMS record is created with the authorization details
  • GIVEN a card number in the request, WHEN COPAUA0C performs the VSAM xref lookup, THEN the associated account ID and status are retrieved correctly via CARDXREF AIX
  • GIVEN a request exceeding the account credit limit, WHEN COPAUA0C applies business rules, THEN AUTH-RESP-CODE indicates decline and APPROVED-AMT is zero in the response

Fraud Marking (CPVD / COPAUS2C)

  • GIVEN a fraud analyst presses PF5 on the authorization detail screen, WHEN COPAUS2C processes the fraud mark, THEN a record is inserted into DB2 AUTHFRDS with AUTH_FRAUD='Y' and the correct CARD_NUM and AUTH_TS
  • GIVEN a DB2 insert error, WHEN COPAUS2C handles the SQLCA, THEN a clear error message is displayed and the CICS transaction rolls back

Batch Purge (CBPAUP0C)

  • GIVEN expired authorization records in IMS, WHEN CBPAUP0C runs, THEN all expired records are deleted from IMS PAUTSUM0/PAUTDTL1
  • GIVEN a purged authorization that had a credit hold, WHEN CBPAUP0C completes, THEN the corresponding ACCT-CASH-CREDIT-LIMIT in ACCTDATA VSAM is restored
  • GIVEN the purge batch run, WHEN it completes, THEN return code is 0 and a summary count of purged records is written to SYSOUT
