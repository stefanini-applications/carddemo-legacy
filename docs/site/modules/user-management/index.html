👥 User Management

User Story Development Guide · CardDemo Mainframe Application

Core Module CA00 · CU00-03 · COUSR00-03C

📋 Overview

The User Management module provides admin-only CRUD operations for system user accounts. Access is restricted to users with type 'A' (Administrator). The module covers the full lifecycle: listing, adding, updating, and deleting users in the USRSEC VSAM file.

CICS Transactions

Transaction IDProgramFunction
CA00COADM01CAdmin Menu — entry point; routes to user management and DB2 options
CU00COUSR00CList users — paginated (10 per screen), forward/backward
CU01COUSR01CAdd new user — validates unique ID, type, inserts to USRSEC VSAM
CU02COUSR02CUpdate user — password, first/last name, type change
CU03COUSR03CDelete user — immediate hard-delete from USRSEC VSAM

Data Store

USRSEC VSAM (AWS.M2.CARDDEMO.USRSEC.PS) — 80-byte fixed-block records keyed on SEC-USR-ID (8 chars), defined in copybook CSUSR01Y. Initialized by the DUSRSECJ JCL job.

Business Context

Only users with SEC-USR-TYPE = 'A' can reach the CA00 Admin Menu and the CU00–CU03 screens. This is enforced at the COADM01C entry point by checking the session's user type. Password is stored in plain text in the base application — this is a known modernization target. There is no soft-delete, audit trail, or password history in the base implementation.

Business Rules

  • Admin-only access (SEC-USR-TYPE = 'A' required for CA00)
  • User ID must be unique (8-char primary key)
  • SEC-USR-TYPE must be 'A' (Administrator) or 'U' (User)
  • Admin cannot delete their own session user (session risk)
  • Delete is immediate hard-delete — no confirmation step in base app
  • No audit trail in base implementation

🎯 User Story Development Guide

Story Templates

List All System Users
As a system administrator, I want to see a paginated list of all user accounts so that I can monitor who has access to the system.

Program: COUSR00C (CU00) — sequential scan of USRSEC VSAM, 10 records per screen, PF7/PF8 paging
Add New User with Type and Credentials
As a system administrator, I want to create a new user account with a specified type and password so that I can grant system access to a new employee.

Program: COUSR01C (CU01) — validates unique SEC-USR-ID, validates SEC-USR-TYPE (A or U), inserts to USRSEC VSAM
Update User Password
As a system administrator, I want to reset a user's password so that I can restore access for users who are locked out.

Program: COUSR02C (CU02) — keyed read-for-update on USRSEC, update SEC-USR-PWD, rewrite record
Delete Departed User
As a system administrator, I want to delete a user account for a departed employee so that access is removed immediately.

Program: COUSR03C (CU03) — keyed delete from USRSEC VSAM; no soft-delete; immediate effect
View All Active Admin Accounts
As an auditor, I want to see all accounts with administrator type so that I can verify privileged access is appropriately controlled.

CU00 list filtered client-side on SEC-USR-TYPE = 'A'; or new filtered scan story (Medium complexity)

Acceleration Factors

  • COUSR00-03C CRUD template — these four programs provide a complete VSAM CRUD template reusable for any new VSAM-backed entity in the system
  • CU00 paging pattern — 10-records-per-screen with forward/backward navigation is directly reusable for any list story
  • CSUSR01Y copybook — user security record shared with authentication (COSGN00C); ensures consistent user record layout across modules
  • DUSRSECJ JCL — provides USRSEC initialization and load pattern for bulk user setup stories

✅ Acceptance Criteria Patterns

Access Control

  • GIVEN a user with SEC-USR-TYPE='U' (non-admin), WHEN they attempt to access CA00, THEN access is denied with an authorization error message
  • GIVEN an unauthenticated session, WHEN CA00 is invoked directly, THEN the user is redirected to the sign-on screen

User List (CU00)

  • GIVEN an authenticated admin, WHEN CU00 is invoked, THEN up to 10 user records are displayed per screen from USRSEC VSAM
  • GIVEN more than 10 users exist, WHEN the admin presses PF8, THEN the next 10 users are displayed; PF7 returns to previous page

Add User (CU01)

  • GIVEN a new unique user ID, WHEN CU01 processes the add, THEN the record is inserted into USRSEC VSAM and a confirmation is displayed
  • GIVEN a duplicate user ID, WHEN CU01 validates, THEN the insert is rejected with a "user ID already exists" error
  • GIVEN an invalid user type (not A or U), WHEN CU01 validates, THEN the insert is rejected with a type validation error

Delete User (CU03)

  • GIVEN a valid user ID, WHEN CU03 deletes the user, THEN the record is immediately removed from USRSEC VSAM
  • GIVEN the admin attempts to delete their own user ID, WHEN CU03 validates, THEN the delete is rejected with a self-delete protection error
