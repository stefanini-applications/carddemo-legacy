👥 User Management Module

User Story Development Guide — Admin User CRUD Operations

📋 Overview

Purpose: Admin-only module enabling full CRUD (Create, Read, Update, Delete) operations on user accounts stored in the VSAM user security file.

Business Context: User management controls who can access CardDemo and with what privileges. Only admin users (type = A) can access this module via the Admin Menu (CA00). Regular users cannot see or access these functions. This is the primary access control mechanism for the application.

CICS Transactions: CU00 (list), CU01 (add), CU02 (update), CU03 (delete)

VSAM Dataset: AWS.M2.CARDDEMO.USRSEC.PS (KSDS, 80-byte records, keyed by SEC-USR-ID)

⚠️ Security Note: Passwords are stored as plain text (8 chars) in the current implementation. Password encryption should be addressed in any modernization effort.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • List Users: As an admin, I want to list all system users so that I can audit who has access.
  • Add User: As an admin, I want to create a new user account with role assignment so that new employees can access the system.
  • Update User: As an admin, I want to update a user's name or password so that account information stays current.
  • Delete User: As an admin, I want to delete a user account so that terminated employees immediately lose system access.
  • Role Assignment: As an admin, I want to set a user's type (regular/admin) so that access privileges match their responsibilities.

⚡ Development Acceleration Factors

  • CSUSR01Y copybook: Pre-defined 80-byte record — all user fields readily accessible
  • Four existing BMS maps: COUSR00–03.bms cover all four CRUD screens completely
  • VSAM KSDS direct access: User ID is the key — O(1) lookup for add/update/delete
  • Admin access control established: CA00 already gates access to this module

📊 Complexity Guidelines

  • 1–2 pts List users (paginated VSAM read), read/display single user
  • 3–5 pts Add user (duplicate check + WRITE), update user (READ + REWRITE), delete user (READ + DELETE)
  • 5–8 pts Password hashing integration, role-based permission matrix, audit log to DB2, self-service password reset

🗄️ User Record Structure

01 SEC-USER-DATA.
   05 SEC-USR-ID      PIC X(08). -- Primary key; must be unique
   05 SEC-USR-FNAME   PIC X(20). -- First name
   05 SEC-USR-LNAME   PIC X(20). -- Last name
   05 SEC-USR-PWD     PIC X(08). -- Password (plain text, 8 chars)
   05 SEC-USR-TYPE    PIC X(01). -- 'U'=regular user, 'A'=admin
   05 SEC-USR-FILLER  PIC X(23). -- Reserved

🔗 Dependencies

  • signon module: USRSEC is the authentication source — changes here affect who can log in
  • Admin access: Only accessible after admin login via CA00 (COADM01C)
  • VSAM USRSEC file: Must be open in CICS (JCL: OPENFIL)
  • transaction-type-management (optional): Also accessible from admin menu alongside user management

✅ Acceptance Criteria Patterns

  • Given admin is authenticated, when accessing CU00, then all existing users are listed (paginated, 10 per page)
  • Given a new unique user ID, when adding via CU01, then VSAM WRITE succeeds and confirmation is displayed
  • Given a duplicate user ID, when adding via CU01, then system rejects with "user ID already exists" error
  • Given user ID exists, when updating via CU02, then fields are updated and VSAM REWRITE succeeds
  • Given user ID exists, when deleting via CU03, then VSAM DELETE succeeds and user can no longer log in
  • Given blank user ID or password, when adding via CU01, then system rejects with field validation error
  • Given user type not 'U' or 'A', when saving, then system rejects with invalid type error
