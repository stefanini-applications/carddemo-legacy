🔐 Authentication

User Story Development Guide — CardDemo Application

📋 Overview

Purpose: Control user access to the CardDemo application via signon screen, credential validation against the USRSEC VSAM file, and routing to the appropriate menu based on user type.

Business Context: Every session starts with authentication. User type (Admin vs Regular) gates all subsequent operations. This module is the foundation of the application's security model.

CICS VSAM COSGN00C TranID: CC00

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Regular User Login: As a cardholder, I want to log in with my user ID and password so that I can securely access my account information.
  • Admin Login: As an admin, I want to log in and be routed to the admin menu so that I can manage users and system configuration.
  • Login Failure Feedback: As a user, I want to receive a clear error message when my credentials are invalid so that I know my login was unsuccessful.
  • User Type Routing: As the system, I want to route authenticated users to the correct menu so that each user sees only their permitted options.

⚡ Development Acceleration Factors

  • COSGN00C: Complete signon program already implemented — use as pattern for additional authentication checks.
  • COCOM01Y (COMMAREA): Session state structure reused across all programs — carries user ID, user type, navigation state.
  • USRSEC VSAM: User credential store is pre-defined with standard record layout (CSUSR01Y).

🔧 Technical Foundation

🏗️ Key Components

  • COSGN00C.cbl: Main signon program. Validates user ID + password against USRSEC. Sets COMMAREA user type on success. TranID: CC00.
  • COCOM01Y.cpy: COMMAREA copybook — carries session state across all programs (user ID, user type, from/to program, account/card/customer info).
  • CSUSR01Y.cpy: User security record layout (8-char ID, 8-char password, 1-char type A/U).
  • COSGN00.bms: BMS screen definition for signon screen.

🔗 Dependencies

  • USRSEC VSAM File: Must be loaded with user records before authentication works.
  • COMEN01C (CM00): Target for regular user post-authentication.
  • COADM01C (CA00): Target for admin user post-authentication.

📊 Authentication Flow

EIBCALEN = 0 → Initialize signon screen (first entry)
User enters UserID + Password → Press Enter
COSGN00C reads USRSEC(UserID)
  → Match found AND password matches:
      Set CDEMO-USER-ID = UserID
      Set CDEMO-USER-TYPE = 'A' or 'U'
      If 'A' → XCTL to COADM01C (CA00)
      If 'U' → XCTL to COMEN01C (CM00)
  → Not found OR password mismatch:
      Display error: "Invalid credentials"
      Re-present signon screen

📁 Key Files

  • app/cbl/COSGN00C.cbl — Signon program
  • app/cpy/COCOM01Y.cpy — COMMAREA definition
  • app/cpy/CSUSR01Y.cpy — User security record
  • app/bms/COSGN00.bms — Signon BMS map
  • app/data/ASCII/usrsec.txt — Sample user data

✅ Acceptance Criteria Patterns

Successful Login:
  • Given a valid user ID and password exist in USRSEC
  • When user enters correct credentials and presses Enter
  • Then user is routed to CM00 (regular) or CA00 (admin) with user ID set in COMMAREA
Failed Login:
  • Given invalid or non-existent credentials
  • When user presses Enter
  • Then error message displayed on signon screen; user is NOT routed to menu
User Type Routing:
  • Given user type = 'A' in USRSEC
  • When login succeeds
  • Then admin menu (CA00) is displayed, not user menu
