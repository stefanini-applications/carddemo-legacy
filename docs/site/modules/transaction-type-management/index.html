📦 Transaction Type Management

Guide for DBA and admin stories that expand DB2-backed metadata and synchronize VSAM reference files.

📋 Overview

Purpose: Manage transaction-type reference data through DB2 CRUD screens (`CTTU`, `CTLI`) while keeping the VSAM copy in sync for core transaction flows.

Business Context: Centralizing types in DB2 allows admins to govern financial metadata while nightly extracts (`TRANEXTR`, `MNTTRDB2`) keep the production engine aware of new codes.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Metadata CRUD: As an admin, I want to update transaction descriptions via `CTLI` so agents see accurate text in the `CT00` list.
  • Batch sync: As an integration lead, I want `TRANEXTR` to export DB2 rows nightly so VSAM has the latest statically compiled transaction types.
  • Governance: As compliance, I want `CTTU` to block deletes when category children exist so we keep referential integrity.

⚡ Development Acceleration Factors

  • DB2 precompiler artifacts: `dcl/` and `ddl/` directories already contain host variable definitions and schema scripts.
  • Cursor patterns: `CTLI` demonstrates forward/backward paging; reuse it when introducing new result sets.
  • Batch jobs: `TRANEXTR` and `MNTTRDB2` can be parameterized with the same control files.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add an additional column to `TRANSACTION_TYPE` and expose it on `CTTU`.
  • Medium (3-5 pts): Add a new category table field that validates in `CTLI` and is exported via `TRANEXTR`.
  • Complex (6-8 pts): Introduce a new DB2-backed reference data flow that requires new binding, `TRANEXTR` transformation, and VSAM file rekeying.

🔧 Technical Foundation

🏗️ Key Components

  • Transactions `CTTU` (add/edit) and `CTLI` (list/update/delete).
  • DB2 tables `CARDDEMO.TRANSACTION_TYPE` and `..._CATEGORY` with cursor logic.
  • Batch jobs `TRANEXTR`, `MNTTRDB2`, `CREADB21` for initial load.

🔗 Dependencies

  • DB2 subsystem with proper plan/bind for static SQL.
  • VSAM files refreshed from `TRANEXTR` so online transactions keep local copies.
  • JCL control files (`ctl/`) for batch parameterization.

🌐 Public Interfaces


CTTU - Transaction type add/edit
CTLI - Transaction type list/update/delete
TRANEXTR - Export DB2 metadata to VSAM
MNTTRDB2 - Batch maintenance

✅ Acceptance Criteria Examples

  • `CTTU` inserts/updates `TRANSACTION_TYPE` with SQLCA zero and allows same-field values after the commit.
  • `CTLI` paging works with forward/backward cursors and displays the current row number along with type code.
  • `TRANEXTR` writes files consumed by `CT00` so the new type is visible to the main menu within one nightly job.
