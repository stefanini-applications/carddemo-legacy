📋 Module Overview

Purpose: Real-time credit card authorization processing. Receives authorization requests via MQ, validates them against account/customer data, stores results in IMS DB, supports fraud marking with DB2 integration, and provides CICS screens for viewing pending authorizations.

Business Context: When a cardholder makes a purchase, the merchant POS system sends an authorization request via MQ. CardDemo processes this request in real time and returns an approve/decline decision. Fraud analysts can then review pending authorizations and flag suspicious ones for investigation.

Transactions: CP00 (process) | CPVS (summary) | CPVD (detail)
Programs: COPAUA0C / COPAUS0C / COPAUS1C / COPAUS2C

🎯 User Story Templates

US-AUTH-003: As a fraud analyst, I want to view the details of a specific authorization so that I can evaluate its legitimacy.

US-AUTH-005: As a batch operator, I want to run CBPAUP0J to purge expired authorizations so that IMS DB storage is managed.

✅ Acceptance Criteria Patterns

Authorization Processing (CP00):
GIVEN an MQ authorization request arrives in the request queue
WHEN COPAUA0C processes the message
THEN account and customer data is validated, approve/decline is determined, response is sent to reply queue, and result is stored in IMS DB
Authorization Summary (CPVS):
GIVEN an authenticated user with account access
WHEN CPVS is executed
THEN a summary list of pending authorizations is displayed from IMS DB
Fraud Mark (CPVD):
GIVEN a fraud analyst selects an authorization and marks it as fraud
WHEN the action is submitted
THEN IMS DB record is updated and a DB2 fraud record is inserted
Batch Purge (CBPAUP0J):
GIVEN expired authorization records exist in IMS DB
WHEN CBPAUP0J runs CBPAUP0C
THEN expired records are deleted, return code = 0

Authorization Processing Flow

  1. External POS/cloud client sends authorization request to MQ request queue
  2. MQ trigger activates CICS transaction CP00 running COPAUA0C
  3. Program reads MQ message and extracts card number and amount
  4. Looks up account via VSAM CARDXREF cross-reference
  5. Validates: account active, balance within credit limit, card active
  6. Stores authorization result (approve/decline) in IMS DB
  7. Sends response message to MQ reply queue
  8. Fraud analyst can view via CPVS (COPAUS0C) and CPVD (COPAUS1C)
  9. Fraud marking via COPAUS2C inserts record into DB2
  10. Batch CBPAUP0J periodically purges expired IMS DB records

Prerequisites for Installation

  • Base CardDemo application installed and operational
  • IBM MQ configured with request and reply queues
  • IMS DB configured with authorization segment definitions
  • DB2 fraud table created (DDL in app/app-authorization-ims-db2-mq/ddl/)
  • CICS resources defined for CP00, CPVS, CPVD transactions
  • IMS DB resources defined (PSB, DBD in ims/ directory)

Dependencies

  • authentication: CICS session required for CPVS/CPVD screens
  • credit-cards: CARDXREF VSAM used for card-to-account lookup
  • accounts: ACCTFILE used for balance and limit validation
  • IBM MQ Infrastructure: External dependency - MQ must be running
  • IMS DB Infrastructure: IMS DB must be configured
  • DB2 Infrastructure: DB2 fraud table must exist

⚡ Development Acceleration Factors

  • MQ trigger pattern: COPAUA0C demonstrates MQ-triggered CICS execution - reuse for other event-driven patterns
  • IMS DL/I calls: Established IMS hierarchical DB access patterns in COPAUS0C/1C/CBPAUP0C
  • Two-phase commit pattern: COPAUS1C demonstrates IMS+DB2 coordinated updates
  • CARDXREF lookup: Card-to-account cross-reference pattern reused across base app and this module
