🗄️ Transaction Type DB2

User Story Development Guide · CardDemo Mainframe Application

Optional Module CTTU · CTLI · DB2 · COBTUPDT

📋 Overview

The Transaction Type DB2 module is an optional extension that maintains transaction type and category reference data in DB2 tables. It demonstrates static embedded SQL, cursor-based browsing with forward/backward paging, and proper COMMIT/ROLLBACK with SQLCA error handling in a CICS/DB2 environment.

Business Context

The module maintains CARDDEMO.TRANSACTION_TYPE and CARDDEMO.TRANSACTION_TYPE_CATEGORY DB2 tables. These tables are the authoritative source for transaction type codes. The TRANEXTR batch job extracts the DB2 data to the VSAM TRANTYPE file, which is then used by the core transaction programs (COTRN02C) for validation. Admin Menu CA00 options 5 (CTLI) and 6 (CTTU) provide access to these screens.

This module serves as a DB2 patterns reference for modernization teams — it demonstrates all key DB2/CICS interaction patterns in one focused module.

Story Templates

Admin Adds New Transaction Type
As a system administrator, I want to add a new transaction type code so that it is available for use in transaction entry (CT02).

COTRTUPC (CTTU) — INSERT INTO CARDDEMO.TRANSACTION_TYPE (TR_TYPE, TR_DESCRIPTION); SQLCA checked; EXEC CICS SYNCPOINT on success
Admin Updates Transaction Type Description
As a system administrator, I want to update the description of an existing transaction type so that it accurately reflects current business usage.

COTRTUPC (CTTU) — UPDATE CARDDEMO.TRANSACTION_TYPE SET TR_DESCRIPTION=:WS-DESC WHERE TR_TYPE=:WS-TYPE; SQLCA error handling
Admin Deletes Transaction Type (Referential Integrity Check)
As a system administrator, I want to delete an unused transaction type code so that the reference table stays clean.

COTRTLIC (CTLI) — DELETE FROM CARDDEMO.TRANSACTION_TYPE WHERE TR_TYPE=:WS-TYPE; FK RESTRICT on TRANSACTION_TYPE_CATEGORY prevents delete if categories exist; SQLCODE -530 shown as user error
Batch Synchronize DB2 to VSAM
As a batch operations manager, I want TRANEXTR to extract transaction types from DB2 to the VSAM TRANTYPE file nightly so that online programs validate against current data.

TRANEXTR job: SELECT all rows from TRANSACTION_TYPE → write sequential file → IDCAMS REPRO to TRANTYPE VSAM
Browse Transaction Types with Paging
As an administrator, I want to browse the list of all transaction types with forward and backward navigation so that I can review the full reference set.

COTRTLIC (CTLI) — DECLARE cursor; FETCH NEXT for forward; reopen cursor + FETCH n times for backward; 10 rows per screen

Acceleration Factors

  • COTRTLIC cursor paging pattern — forward/backward DB2 cursor navigation is directly reusable for any DB2 browsing transaction in the system
  • COTRTUPC INSERT/UPDATE pattern — static embedded SQL with SQLCA handling is the template for any DB2 write story
  • COBTUPDT batch DB2 update pattern — batch DB2 maintenance with COMMIT/ROLLBACK is reusable for any batch DB2 update story
  • SQLCA error handling template — consistent SQLCODE → user message mapping pattern applicable to all DB2 programs

Cursor Paging Pattern (COTRTLIC)

-- Forward paging:
EXEC SQL DECLARE TRT-CURSOR CURSOR FOR
    SELECT TR_TYPE, TR_DESCRIPTION
    FROM CARDDEMO.TRANSACTION_TYPE
    ORDER BY TR_TYPE
END-EXEC.

EXEC SQL OPEN TRT-CURSOR END-EXEC.
PERFORM VARYING WS-ROW FROM 1 BY 1
    UNTIL WS-ROW > 10 OR SQLCODE = +100
    EXEC SQL FETCH TRT-CURSOR INTO :WS-TYPE, :WS-DESC END-EXEC
    IF SQLCODE = 0
        MOVE WS-TYPE TO SCREEN-TYPE(WS-ROW)
        MOVE WS-DESC TO SCREEN-DESC(WS-ROW)
    END-IF
END-PERFORM.

-- Backward paging: close cursor, reopen, FETCH to (current-page-start - 10)

✅ Acceptance Criteria Patterns

Add/Update (CTTU)

  • GIVEN a new unique type code, WHEN CTTU processes the add, THEN a row is inserted into CARDDEMO.TRANSACTION_TYPE and EXEC CICS SYNCPOINT commits the change
  • GIVEN a duplicate type code, WHEN CTTU attempts the INSERT, THEN SQLCODE indicates duplicate key and a user-friendly error message is displayed
  • GIVEN a DB2 error during INSERT, WHEN CTTU handles the SQLCA, THEN EXEC CICS SYNCPOINT ROLLBACK is issued and the error is displayed

Browse/Delete (CTLI)

  • GIVEN multiple transaction types exist, WHEN CTLI is invoked, THEN rows are displayed using the DB2 cursor with correct forward and backward paging
  • GIVEN a type code with associated category rows, WHEN CTLI attempts to delete it, THEN the DB2 FK RESTRICT constraint fires (SQLCODE -530) and the user sees a referential integrity error
  • GIVEN a type code with no categories, WHEN CTLI deletes it, THEN the row is removed from TRANSACTION_TYPE and the change is committed

Batch Synchronization

  • GIVEN a MNTTRDB2 run, WHEN COBTUPDT completes, THEN return code is 0 and DB2 TRANSACTION_TYPE reflects all intended changes
  • GIVEN a TRANEXTR run, WHEN CBEXPORT completes, THEN TRANTYPE VSAM contains all rows from CARDDEMO.TRANSACTION_TYPE and COTRN02C validates against the updated data
