🔑 Authorization Module

CardDemo — User Story Development Guide

📋 Overview

Purpose: Credit card authorization request processing using MQ for messaging, IMS DB for customer/authorization data, and DB2 for finalized authorization records.

Business Context: Cardholders can view pending authorizations before they are finalized. Authorization requests arrive via MQ, customer data is retrieved from IMS, and finalized authorizations are stored in DB2. Expired authorizations are purged by a batch job.

CICS Transactions: CPVS (auth summary), CPVD (auth detail), CP00 (process auth — MQ triggered)

🎯 User Story Development Guide

📝 US Templates

  • As a cardholder, I want to view a summary of pending authorizations so that I can see charges being processed
  • As a cardholder, I want to view the details of a specific pending authorization so that I can verify a charge before it posts
  • As a system, I want authorization requests processed via MQ so that the authorization service decouples from the cardholder-facing system
  • As a batch operator, I want expired authorizations purged by CBPAUP0J so that IMS and DB2 do not accumulate stale records

⚡ Development Acceleration Factors

  • MQ request/response pattern: Established in COPAUA0C — reusable for other MQ integrations
  • IMS DL/I patterns: COPAUS0C, COPAUS1C demonstrate IMS read and update patterns (PSBs provided)
  • DB2 insert pattern: Authorization finalization in COPAUS1C demonstrates CICS DB2 insert
  • CCPAUERY/CCPAURLY/CCPAURQY: Authorization query/reply/request data structures in copybooks

📊 Complexity Guidelines

  • 3–5 pts Pending Auth Summary (CPVS) — read IMS and VSAM, display list
  • 5–8 pts Auth Detail (CPVD) — IMS update + DB2 insert in single CICS transaction
  • 8–13 pts Authorization processing (CP00) — MQ trigger, IMS insert/update, full request/response cycle

🔄 Authorization Flow

1. Authorization request arrives on MQ queue
2. COPAUA0C triggered (transaction CP00)
3. Retrieve customer data from IMS DB (DL/I GET UNIQUE)
4. Process authorization decision
5. Insert/Update authorization record in IMS
6. Insert finalized record in DB2 AUTHFRDS table
7. Send response back via MQ
8. Cardholder views pending auths via CPVS (reads IMS + VSAM)
9. Cardholder views detail via CPVD (reads IMS, updates status, inserts DB2)
10. Batch CBPAUP0J purges expired records from IMS and DB2

✅ Acceptance Criteria Examples

  • Given pending authorizations in IMS for account → CPVS displays summary list with amount and merchant
  • Given authorization selected from summary → CPVD shows full authorization details
  • Given MQ authorization request arrives → CP00 processes it; IMS and DB2 updated; MQ response sent
  • Given expired authorization records → CBPAUP0J deletes them from IMS and DB2 without affecting active records
