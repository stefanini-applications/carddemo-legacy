🧱 Core Batch Orchestration

Guidelines for writing stories that keep VSAM datasets, statements, and analytics feeds in sync via JCL and scripting.

📋 Overview

Purpose: Automate dataset refresh, interest calculations, posting, indexing, and purge operations that power the online experience.

Business Context: Batch jobs under `app/jcl/` and `samples/jcl/` refresh VSAM files and generate derived data that users consume; stories here ensure operational readiness.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Post-run Refresh: As a batch operator, I want `run_full_batch.sh` to submit `ACCTFILE` → `CARDFILE` → `TRANBKP` in order so that all VSAM sources are fresh before posting.
  • Index Rebuild: As a release coordinator, I want `TRANIDX` to run after `TRANEXTR` so that alternate indexes reflect new cards before the new day.
  • Purge Strategy: As a platform architect, I want `CBPAUP0J` to delete expired auth records after 30 days so that IMS storage stays under 2 GB.

⚡ Development Acceleration Factors

  • JCL Sequencing: Mirror `scripts/run_full_batch.sh` order when creating new jobs; reuse the `CLOSEFIL` / `OPENFIL` pattern for file gating.
  • Job Templates: `samples/jcl/CICCMP.jcl`, `CBPAUP0J`, `IMSMQCMP.jcl` provide proven structures—copy and adjust parameters rather than building from scratch.
  • Automation Scripts: `scripts/run_interest_calc.sh`, `scripts/run_posting.sh`, and `remote_refresh.sh` illustrate how to wrap FTP submissions for Jenkins or remote CI.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a parameterized variable to an existing JCL (e.g., dataset name) with validation in the script.
  • Medium (3-5 pts): Introduce a new batch job that feeds an existing VSAM dataset and update `run_full_batch.sh` to schedule it.
  • Complex (5-8 pts): Develop a new asynchronous job that interacts with MQ before/after a Db2 operation, updating the automation script and documentation.

🔧 Technical Foundation

🏗️ Key Components

  • JCL jobs: `ACCTFILE.jcl`, `CARDFILE.jcl`, `TRANBKP.jcl`, `TRANEXTR.jcl`, `CREASTMT.jcl`, `TRANIDX.jcl`, `CBPAUP0J`.
  • COBOL batch: `CBTRN02C`, `CBTRN03C`, `CBSTM03A`, `CBPAUP0C` for posting and purges.
  • Automation scripts: `run_full_batch.sh`, `run_posting.sh`, `remote_compile.sh`, `remote_refresh.sh`.

🔗 Dependencies

  • core-online: Relies on fresh VSAM data from these jobs.
  • transaction-type: Depends on `TRANEXTR` data to supply Db2 transaction metadata.
  • authorization: `CBPAUP0J` must run before MQ-based flows reuse IMS space.
  • scripts: `scripts/run_full_batch.sh` orchestrates dataset refresh.

🌐 Public Interfaces

ACCTFILE/CARDFILE/CUSTFILE - Reload VSAM datasets with new sample data
TRANBKP - Backup DALYTRAN before recombining transactions
CREASTMT/CBSTM03A - Build customer statements based on posted transactions
TRANEXTR/TRANIDX - Export metadata to VSAM and rebuild indexes before trading day
POSTTRAN/INTCALC/COMBTRAN - Posting, interest, and merge jobs that produce the daily state
CBPAUP0J - Expired authorization purge

🧪 Acceptance Criteria & Patterns

  • Sequencing: Dataset refresh jobs must finish before `POSTTRAN`; automation scripts should log each submission (see `run_full_batch.sh`).
  • Instrumentation: Every JCL must write RC codes that script monitors (e.g., `tnftp` echo statements) and fail fast on RC>0.
  • Automation: Use `sleep` statements between jobs when manipulating files that are still open to avoid dataset locks.
