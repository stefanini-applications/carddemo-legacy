📦 authorization-ims-db2-mq

Guide for user stories that follow the MQ → IMS → DB2 authorization flow.

📋 Overview

Purpose: Emulate real-world POS authorizations by receiving MQ requests, validating via VSAM/IMS, and tracking fraud in DB2.

Business Context: This optional module represents an integration template for partner MQ clients that need to route authorization decisions back through the core CardDemo data model.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • MQ Approval: As a merchant emulator, I want to send a CSV-style MQ request to PAUTH.REQUEST and receive a response so I know whether the card was approved.
  • Fraud Investigation: As a Fraud Analyst, I want to use CPVD to mark an authorization and persist the flag in the DB2 AUTHFRDS table.
  • Batch Maintenance: As a Batch Operator, I want CBPAUP0J to purge authorizations older than 30 days to keep IMS performance stable.

⚡ Development Acceleration Factors

  • Predefined programs: COBOL programs COPAUA0C, COPAUS0C, COPAUS1C, COPAUS2C, CBPAUP0C live under app/app-authorization-ims-db2-mq/.
  • MQ + IMS artifacts: Queue names and DBD/PSB definitions are already defined in the extension directory; reuse the same names when building new MQ channels.
  • DB2 schema: AUTHFRDS and XAUTHFRD are ready, so new reporting screens can join on the same table.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new MQ field to the `PAUTH.REQUEST` message and display it on the CPVS screen.
  • Medium (3-5 pts): Add a fraud tagging screen that links IMS detail to DB2 rows via the existing copybooks.
  • Complex (5-8 pts): Introduce a new MQ consumer or integrate a downstream analytics service that consumes AUTHFRDS data.

✅ Acceptance Criteria Examples

  • MQ requests/replies honor the CSV format and replicate the `AUTH-ID-CODE` + `AUTH-RESP-CODE` fields.
  • IMS records older than 30 days are purged only by CBPAUP0J and logged for audit.
  • Any fraud marking updates both IMS/DB2 and logs the event timestamp for analytics.

🔧 Technical Foundation

🏗️ Key Components

  • CICS programs COPAUA0C, COPAUS0C, COPAUS1C, COPAUS2C, CBPAUP0C.
  • MQ queues AWS.M2.CARDDEMO.PAUTH.REQUEST and .REPLY.
  • IMS DBDs, PSBs, and JCL templates under app/app-authorization-ims-db2-mq/.
  • DB2 schema with AUTHFRDS and XAUTHFRD indexes.

🔗 Dependencies

  • Requires MQ infrastructure to be reachable from CICS.
  • IMS HIDAM database must exist with PSBs/DBDs deployed before CICS transactions are installed.
  • DB2 subsystem must be bound (DB201PLN) with the `AUTHFRDS` table ready.
  • Relies on VSAM datasets for account cross-reference lookups.

🌐 Public Interfaces

CP00 (MQ-triggered authorization), CPVS/CPVD (summary/detail), CBPAUP0J (purge); MQ queues: PAUTH.REQUEST, PAUTH.REPLY; DB2 table: AUTHFRDS; IMS DBD/PSB: DBPAUTP0/DBPAUTX0 & PSBPAUTB/PSBPAUTL.
