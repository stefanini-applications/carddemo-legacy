📨 MQ Integration Optional Module

MQ-Based Account & System Date Inquiry — CDRD / CDRA

📋 Overview

Module ID: mq-integration

Purpose: Optional extension demonstrating IBM MQ request/response messaging patterns. Provides two transactions: CDRD (system date inquiry) and CDRA (account details inquiry). External systems can query the mainframe without direct VSAM access.

ℹ Installation required: Requires IBM MQ setup with configured queue manager and queues. See app/app-vsam-mq/README.md for setup instructions.

Business Context: Demonstrates asynchronous integration pattern for mainframe service exposure. Useful for service enablement scenarios where external systems (cloud, distributed) need to query mainframe data via messaging.

🎯 User Story Templates

As an external system, I want to query the mainframe system date via MQ (CDRD) so that date-sensitive operations are synchronized with the mainframe clock.
As an external system, I want to retrieve account details via MQ (CDRA) so that I can integrate without requiring direct VSAM file access.
As a cloud application, I want asynchronous account inquiry via MQ so that mainframe availability doesn't block my request processing.
As a distributed system, I want standard MQ message formats so that I can integrate with the mainframe using any MQ-compatible client.

✅ Acceptance Criteria Patterns

Date Inquiry (CDRD): Given request message on input queue, when CODATE01 processes, then response message with current mainframe system date (YYYY-MM-DD format) is placed on reply queue identified by MQMD CorrelId.
Account Inquiry (CDRA): Given request message with account ID, when COACCT01 processes, then: ACCTFILE is read by account ID, account fields are formatted into response message, response is placed on MQ reply queue.
Account Not Found: Given request with non-existent account ID, then error response (with appropriate error code) is returned on reply queue.
Message Format: Request and response messages must conform to defined MQ message format; malformed messages generate error response with format error code.
Correlation: Response message MQMD CorrelId must match the request message MsgId for proper correlation by the requesting system.

🔧 Technical Foundation

ComponentTransactionFunction
CODATE01.cblCDRDSystem date inquiry — reads mainframe date, formats MQ response message
COACCT01.cblCDRAAccount inquiry — reads ACCTFILE, formats account data in MQ response
MQ input queue—Receives incoming request messages from external systems
MQ reply queue—Identified by MQMD ReplyToQ in request; target for response messages
ACCTFILE VSAM—Read by COACCT01 for account data (CDRA)

MQ Request/Response Pattern

-- Incoming MQ message structure:
MQMD CorrelId / MsgId (for correlation)
MQMD ReplyToQ (reply queue name)
Message data: ACCOUNT-ID or inquiry parameters

-- Processing:
EXEC CICS GET QUEUE(input-queue) ...
-- Process request, read VSAM if needed
-- Format response data
EXEC CICS PUT QUEUE(reply-queue) CORRELID(MsgId) ...

-- Response: account data or date in defined format

🔗 Dependencies

  • IBM MQ: Queue manager and queues must be configured
  • CICS-MQ bridge: Required for EXEC CICS MQ commands
  • accounts: ACCTFILE accessed by COACCT01 (CDRA)
  • CSD definitions: CICS resource definitions in app/app-vsam-mq/csd/

⚡ Development Acceleration

  • MQ request/response pattern: Fully established in CODATE01/COACCT01 — copy for any new MQ service
  • MQMD correlation: Correlation ID pattern documented and implemented
  • Error response format: Error handling pattern established — reuse for all MQ services
  • CSD template: CICS resource definitions in csd/ provide template for new MQ transactions
