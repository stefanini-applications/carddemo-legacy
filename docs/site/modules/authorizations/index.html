📦 Authorizations

User story guide for the IMS/DB2/MQ pending-authorization system.

📋 Overview

Purpose: Model real-time authorization request/response flows for modernization demos by wiring MQ to IMS/DB2-backed CICS programs.

Business Context: This module demonstrates two-phase commit patterns, fraud tagging, and MQ-driven triggers; it allows product teams to showcase how a cloud MQ client can securely call into a mainframe authorization workflow.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Request Validation: As an MQ integrator, I want `CP00` to reject malformed `AWS.M2.CARDDEMO.PAUTH.REQUEST` messages so that downstream DB2 fraud analytics stay accurate.
  • Fraud Review: As a fraud analyst, I want to view pending authorizations in `CPVS`, drill into `CPVD`, and flag fraud via `COPAUS2C` so that DB2 `AUTHFRDS` records capture the status.

⚡ Development Acceleration Factors

  • Shared copybooks: `CIPAUSMY`, `CIPAUDTY`, `CCPAURQY`, `CCPAURLY` hold request/response layouts that are reused across `COPAUA0C`, `COPAUS0C`, and batch purge programs.
  • MQ patterns: Reuse the existing MQCONNECT/MQOPEN/MQPUT/MQGET choreography from `COPAUA0C`; changes typically only touch queue names or correlation data.

📊 Complexity Guidelines

  • Simple: Add an additional field to the comma-delimited MQ request and display it on `CPVS`.
  • Medium: Extend `COPAUA0C` to call another DB2 table before responding and update `AUTHFRDS` with that context.
  • Complex: Introduce a new MQ queue bundle or encrypt the payload, requiring updates to `IMS` DBD/PSB definitions plus `DB2` table updates.

🔧 Technical Foundation

🏗️ Key Components

  • Programs: `COPAUA0C` (MQ-triggered processor), `COPAUS0C`/`COPAUS1C` (summary/detail), `COPAUS2C` (fraud tagging), `CBPAUP0C` (expiration purge).
  • IMS Artifacts: DBDs `DBPAUTP0`, `DBPAUTX0`; segments `PAUTSUM0`, `PAUTDTL1`, index `PAUTINDX`; PSBs `PSBPAUTB`, `PSBPAUTL`.
  • DB2 Table: `AUTHFRDS` and index `XAUTHFRD` for fraud records.

🔗 Dependencies

  • IBM MQ queues `AWS.M2.CARDDEMO.PAUTH.REQUEST` and `.REPLY` declared under `app/app-authorization-ims-db2-mq/jcl` and wired via `app/app-authorization-ims-db2-mq/csd`.
  • IMS HIDAM for authorization storage (optional but essential for certification demos).
  • DB2 for fraud tracking and analytics; requires schema updates reflected in `COPAUS2C`.

🌐 Public Interfaces


MQ Queues:
  AWS.M2.CARDDEMO.PAUTH.REQUEST  (comma-separated values matching CCCPAURQY layout)
  AWS.M2.CARDDEMO.PAUTH.REPLY    (response layout CCCPAURLY)
CICS Transactions:
  CP00 - Authorization request processor
  CPVS - Pending authorization summary
  CPVD - Authorization detail / fraud tagging
Batch:
  CBPAUP0J (calls CBPAUP0C) - Purge expired authorizations

✅ Acceptance Criteria Examples

  • Every MQ request that reaches `COPAUA0C` results in either a DB2 insert or an `AUTHFRDS` record flagged as fraud.
  • `CPVS` shows the latest summary from IMS with PF5 enabling fraud tagging and updates `AUTHFRDS` inside the same CICS unit-of-work.
  • `CBPAUP0J` (batch) removes expired authorizations older than the configured retention window without leaving locked VSAM files.
