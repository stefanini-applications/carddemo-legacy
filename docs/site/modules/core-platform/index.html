Module Overview and Purpose

core-platform is the mandatory baseline module for CardDemo. It defines the runtime behavior that every optional module relies on: sign-on, CICS menu navigation, core VSAM entities, online transaction entry/update flows, and nightly/monthly batch jobs that keep balances, statements, and indexes consistent.

The module is centered around three execution styles:

  • Online CICS flows: 3270 BMS screens + COBOL transaction programs.
  • Batch pipelines: JCL chains invoking COBOL and utilities (IDCAMS, SORT).
  • Shared contracts: copybooks defining commarea and data record layouts.

Business Goals

  • Authenticate users and route to correct menu context.
  • Maintain account/card/customer entities safely.
  • Capture and post transactions into persistent master data.
  • Process bill payments with confirmation and audit trail transaction.
  • Run repeatable batch jobs for posting, interest, and statements.

Primary User Journeys

  • CC00 sign-on for cardholder/admin identity.
  • CM00 menu option navigation to account/card/transaction functions.
  • CAUP/CCUP controlled update with confirm-save semantics.
  • CT02 transaction capture with strict key/date/amount validation.
  • CB00 full-balance payment with transaction write + account update.

Architectural Context

Core platform follows a transaction-oriented CICS architecture where each transaction ID is mapped to a COBOL program and optional BMS mapset. Request context is passed in CARDDEMO-COMMAREA (app/cpy/COCOM01Y.cpy) and used for routing, identity propagation, and back-navigation.

Execution Model

Layer Mechanism Examples Notes
UI + Transaction CICS + BMS + COBOL COSGN00C, COMEN01C, COACTUPC, COTRN02C, COBIL00C PF key/AID driven state transitions; map redisplay on validation failure.
Persistence VSAM KSDS and AIX paths ACCTDAT, CARDDAT, CCXREF, CXACAIX, TRANSACT Read/update with CICS RESP/RESP2 handling; optimistic re-check in update flows.
Batch JCL + COBOL + utilities POSTTRAN, INTCALC, COMBTRAN, CREASTMT Nightly/monthly sequences managed by CA7 and Control-M definitions.

High-Level Data Flow

CC00 (COSGN00C) -> validate USRSEC -> XCTL COMEN01C/COADM01C
CM00 (COMEN01C) -> option to CAUP/CCUP/CT02/CB00/CR00
CAUP/CCUP -> read XREF + ACCT + CUST/CARD -> validate -> lock -> rewrite -> return
CT02 -> validate keys/data -> resolve account/card via xref -> write TRANSACT
CB00 -> read/update ACCTDAT -> write TRANSACT (payment) -> rewrite ACCTDAT

Batch:
POSTTRAN (CBTRN02C) -> DALYTRAN -> validate against XREF/ACCT -> write TRANSACT
INTCALC (CBACT04C) -> TCATBALF + DISCGRP -> compute interest -> update ACCT + write SYSTRAN
COMBTRAN -> merge BKUP + SYSTRAN -> reload TRANSACT
CREASTMT (CBSTM03A) -> produce text and HTML statements

API Example: Sign-On Validation Path

READ-USER-SEC-FILE.
    EXEC CICS READ
         DATASET   (WS-USRSEC-FILE)
         INTO      (SEC-USER-DATA)
         RIDFLD    (WS-USER-ID)
         KEYLENGTH (LENGTH OF WS-USER-ID)
         RESP      (WS-RESP-CD)
         RESP2     (WS-REAS-CD)
    END-EXEC.

    EVALUATE WS-RESP-CD
      WHEN 0
        IF SEC-USR-PWD = WS-USER-PWD
          IF CDEMO-USRTYP-ADMIN
            EXEC CICS XCTL PROGRAM('COADM01C') ...
          ELSE
            EXEC CICS XCTL PROGRAM('COMEN01C') ...
          END-IF
        ELSE
          MOVE 'Wrong Password. Try again ...' TO WS-MESSAGE
        END-IF
      WHEN 13
        MOVE 'User not found. Try again ...' TO WS-MESSAGE
    END-EVALUATE.

Business Rules Confirmed in Code

  • Authentication: CC00 requires non-blank user/password and matches against USRSEC; routes by user type.
  • Menu authorization: user role is checked against option metadata; admin options are blocked for user role.
  • Account lookup path: account id must resolve in CXACAIX, then ACCTDAT, then associated CUSTDAT.
  • CAUP validation: validates dates, signed currency fields, SSN rules, FICO range, state code, zip-state consistency, and required text fields.
  • CCUP validation: card name alpha/space only, status only Y/N, expiry month 1-12, expiry year 1950-2099.
  • Update concurrency: CAUP/CCUP perform update-lock reads and compare previously fetched values before rewrite to detect stale updates.
  • Bill payment: rejects when current balance is not positive; on confirm writes payment transaction and updates account balance.
  • CT02 validation: enforces amount mask [-|+]99999999.99, date format YYYY-MM-DD, numeric merchant id, non-empty required fields.
  • POSTTRAN validation: rejects transactions when xref/account not found, overlimit, or account expired relative to transaction origin timestamp.
  • INTCALC behavior: reads disclosure rate by account group/type/category; computes monthly interest and writes synthetic system transactions.

Development Guidelines and Patterns

Pattern: CICS Screen Roundtrip

IF first-entry
  initialize map
  send screen
ELSE
  receive screen
  evaluate EIBAID
    WHEN ENTER -> process inputs
    WHEN PF3   -> route back
    WHEN PF4/PF5/PF12 -> action specific
    WHEN OTHER -> message + redisplay
  END-EVALUATE
END-IF

EXEC CICS RETURN
  TRANSID(...)
  COMMAREA(...)
END-EXEC

Pattern: Update With Stale-Data Check

  1. Fetch and cache old values into program commarea/work area.
  2. Validate user edits.
  3. READ ... UPDATE to acquire lock(s).
  4. Re-compare current persisted data with cached old values.
  5. If changed, reject and ask user to review latest state.
  6. If unchanged, build update record and REWRITE.

Pattern: PF Key Semantics

PF Key Typical Meaning Observed In
PF3 Back/Exit to previous program/menu CC00, CM00, CAUP, CCUP, CT02, CB00
PF4 Clear current form CT02, CB00
PF5 Confirm save/execute action CAUP, CCUP, CT02 copy-last flow
PF12 Cancel edits and reload fetched details CAUP, CCUP

Quality Guardrails

  • Preserve copybook compatibility for shared record layouts.
  • Keep message-area based feedback deterministic and user-actionable.
  • When changing validation rules, align map hints and COBOL edits together.
  • For batch updates, maintain idempotent restart behavior and explicit reject outputs.

Operational Runbook Notes

Typical Daily Chain (Observed in Scheduler Artifacts)

  1. CLOSEFIL (close online files in CICS)
  2. POSTTRAN (post daily input)
  3. WAITSTEP (cooldown/wait)
  4. OPENFIL (re-open online files)

Typical Monthly Chain

  1. CLOSEFIL
  2. INTCALC
  3. COMBTRAN
  4. WAITSTEP
  5. OPENFIL

Failure Signals to Watch

  • CICS RESP/RESP2 non-normal codes in online updates.
  • Unexpected increase in DALYREJS output volume.
  • Missing alternate index/path for transaction browse/reporting flows.
  • Scheduler out-condition not cleared causing chain blocks.

Fast Validation Checklist

  • All required files present and enabled in CSD group CARDDEMO.
  • OPENFIL has run after maintenance cycles.
  • CC00 login works for ADMIN001 and USER0001.
  • CT02 can insert and show generated transaction ID message.
  • CB00 posts payment and updates account balance.
