📦 Authorization

Guide for user stories that span MQ-driven authorizations, IMS pending records, and DB2 fraud analytics.

📋 Overview

Purpose: Add a credit card authorization workflow that ingests MQ requests, stores pending records in IMS HIDAM, and marks fraud cases in DB2 for analytics.

Business Context: Modernization efforts need an MQ contract that mirrors live POS interactions, tracks pending authorizations, and surfaces fraud decisions to reviewers.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Real-time processing: As an integration partner, I want to send `AUTH` requests to `AWS.M2.CARDDEMO.PAUTH.REQUEST` and get an IMS-backed response through `AWS.M2.CARDDEMO.PAUTH.REPLY` so my POS mirrors CardDemo decisions.
  • Fraud marking: As a fraud analyst, I want to flag a suspicious authorization from `CPVD` and have `AUTHFRDS` record `AUTH_FRAUD='Y'` immediately.
  • Retention: As an operations manager, I want `CBPAUP0J` to purge IMS records older than the retention window so the HIDAM performance stays predictable.

⚡ Development Acceleration Factors

  • MQ templates: The existing request/response copybooks (e.g., `CCPAURQY`, `CCPAURLY`) define field order; reuse them to build new queue interop.
  • DB2 table/index: `AUTHFRDS`/`XAUTHFRD` already handle fraud attributes, so augmenting them only requires additional columns or indexes.
  • IMS segments: `PAUTSUM0`/`PAUTDTL1` support summary and detail navigation; screens `CPVS` and `CPVD` already scroll through them.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new field (e.g., merchant category) to the MQ request copybook and surface it on `CPVS`.
  • Medium (3-5 pts): Extend fraud scoring so `CPVD` writes a new indicator to `AUTHFRDS` and reruns the existing DB2 update flow.
  • Complex (6-8 pts): Introduce a new MQ response code that requires IMS updates, DB2 writes, and new UX messaging across `CPVS/CPVD`.

🔧 Technical Foundation

🏗️ Key Components

  • Programs `COPAUA0C`, `COPAUS0C`, `COPAUS1C`, `COPAUS2C`, `CBPAUP0C`.
  • IMS artifacts `DBPAUTP0`, `DBPAUTX0`, `PSBPAUTB`, `PSBPAUTL`.
  • MQ queues `AWS.M2.CARDDEMO.PAUTH.REQUEST`/`REPLY`.

🔗 Dependencies

  • IBM MQ configured with CICS-MQ connection resources.
  • DB2 schema binding for `AUTHFRDS` plus static SQL programs.
  • VSAM cross-reference files (e.g., `CARDXREF`) for account lookups.

🌐 Public Interfaces


CP00 - MQ-triggered authorization request parser
CPVS - Pending authorization summary
CPVD - Authorization details + fraud marking
CBPAUP0J - Batch purge of expired IMS authorizations
<MQ> AWS.M2.CARDDEMO.PAUTH.REQUEST/REPLY

✅ Acceptance Criteria Examples

  • MQ requests write to IMS (PAUTSUM0/PAUTDTL1) before the reply queue is opened.
  • `CPVD` PF5 sets `AUTH_FRAUD='Y'` and logs the same flag in `AUTHFRDS` with the reason code.
  • `CBPAUP0J` removes entries older than 30 days while releasing VSAM locks.
