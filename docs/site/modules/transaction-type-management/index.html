1. Overview & Purpose

The transaction-type-management module is an optional DB2 extension for the CardDemo mainframe application. It replaces historically flat-file-driven transaction-type reference data with a relational DB2 approach, enabling administrators to maintain transaction type codes and categories through interactive CICS screens and JCL batch jobs.

The module lives entirely under app/app-transaction-type-db2/ and exposes two CICS transactions (CTLI and CTTU) alongside three batch jobs (CREADB21, TRANEXTR, MNTTRDB2). It serves as a reference architecture for embedding static DB2 SQL in COBOL/CICS programs and bridging DB2 relational data back into the VSAM-based core application via nightly extraction.

Key Value Proposition: Centralises transaction-type reference data in DB2, allowing admin users to add, update, and delete codes through a governed UI while the nightly TRANEXTR job keeps the VSAM pipeline consistent.

Module Scope

  • Transaction-type code lifecycle (add / edit / delete)
  • Category sub-code management
  • Forward and backward cursor-based paging
  • Nightly DB2 → VSAM data extraction
  • Batch bulk-load / maintenance of DB2 rows

Out of Scope

  • Core transaction posting (core-application module)
  • Authorization or fraud analytics (authorization module)
  • MQ messaging integration
  • IMS persistence

2. Architectural Context

Position in the CardDemo Stack

CardDemo follows a transaction-centric mainframe architecture where CICS programs serve as the application tier, VSAM KSDS files as the primary operational database, and DB2 as an optional relational layer for metadata and analytics. This module occupies the DB2 metadata tier alongside the authorization module's fraud tables.

Dual Persistence Strategy

The module intentionally maintains two copies of transaction-type data:

  1. DB2 (authoritative): relational tables enable referential integrity, governed UI edits, and SQL querying.
  2. VSAM flat files (operational): high-throughput reads for the core transaction-listing screen (COTRN00C). Refreshed by TRANEXTR.
Sync lag: VSAM flat files reflect DB2 changes only after the next TRANEXTR run. Changes made via CTTU/CTLI are immediately durable in DB2 but appear in online transaction lists only after the nightly batch.

             D = Delete (DELETE row)
             * = Comment (ignored)
Cols 2-3   : Transaction type code (2-char)
Cols 4-53  : Transaction description (50-char)

CREATE TABLE CARDDEMO.TRANSACTION_TYPE

    TR_DESCRIPTION VARCHAR(50)  NOT NULL,   -- Human-readable label

DB2 Table: CARDDEMO.TRANSACTION_TYPE_CATEGORY

CREATE TABLE CARDDEMO.TRANSACTION_TYPE_CATEGORY
(
    TRC_TYPE_CODE     CHAR(2)      NOT NULL,   -- FK to TRANSACTION_TYPE.TR_TYPE
    TRC_TYPE_CATEGORY CHAR(4)      NOT NULL,   -- Category sub-code

7. Business Rules & Validation

Referential Integrity

Rule RI-01: TRANSACTION_TYPE_CATEGORY rows reference TRANSACTION_TYPE.TR_TYPE with ON DELETE RESTRICT. Attempting to delete a transaction type that has associated category rows will receive a negative SQLCODE and must be rejected with a user-facing error message.

Rule IDFieldRuleError
IV-01TR_TYPEMandatory; blank triggers BLANK flagFLG-TRANFILTER-BLANK
IV-02TR_TYPEMust be alphanumeric only (INSPECT vs LIT-ALL-ALPHANUM-FROM-X)FLG-ALPHNANUM-NOT-OK
IV-03TR_DESCRIPTIONMandatory on save; blank triggers BLANK flagFLG-DESCRIPTION-BLANK
IV-04TR_DESCRIPTIONMust be alphanumeric characters onlyWS-NAME-MUST-BE-ALPHA
IV-05BothOptimistic-lock: if record modified between read and PF5, reject with stale-data messageDATA-WAS-CHANGED-BEFORE-UPDATE

Input Validation — CTLI (COTRTLIC)

Rule IDRule
IV-06Action column accepts only U (update) or D (delete). Any other non-blank value triggers WS-ROW-SELECT-ERROR for that row.
IV-07Multiple rows may be marked U simultaneously; each is processed in a loop. Delete confirmation is single-row per screen refresh.
IV-08Updated description fields undergo the same alphanumeric validation as CTTU.

Batch MNTTRDB2 (COBTUPDT)

Rule IDRule
BT-01Records with operation code * are silently skipped (comment lines).
BT-02Any unrecognised operation code triggers an abend (9999-ABEND paragraph).
BT-03Negative SQLCODE on any DML statement causes immediate abend with formatted SQLCODE in return message.
BT-04SQLCODE +100 (NOT FOUND) on UPDATE is treated as a hard error and causes abend.

Data Synchronisation Rules

Rule IDRule
DS-01TRANEXTR must run after any online or batch DB2 changes to keep VSAM files current for core transaction screens.
DS-02Steps 10 and 20 of TRANEXTR back up existing VSAM files to GDG before replacement. Failure in backup steps (COND=(0,NE)) causes subsequent extraction steps to be skipped.

9. Development Guidelines & Patterns

DB2 Static SQL Pattern

All DB2 access uses static embedded SQL compiled by the DB2 precompiler. The canonical pattern:

* Include host variables via DCLGEN:
     EXEC SQL INCLUDE DCLTRTYP END-EXEC.

* Execute a parameterised query:
     EXEC SQL
          SELECT TR_TYPE, TR_DESCRIPTION
            INTO :DCL-TR-TYPE, :DCL-TR-DESCRIPTION
            FROM CARDDEMO.TRANSACTION_TYPE
           WHERE TR_TYPE = :WS-SEARCH-KEY
     END-EXEC.

* Check SQLCA immediately after every EXEC SQL:
     MOVE SQLCODE TO WS-DISP-SQLCODE
     EVALUATE TRUE
         WHEN SQLCODE = ZERO
              CONTINUE
         WHEN SQLCODE = +100
              SET WS-RECORD-NOT-FOUND TO TRUE
              PERFORM 9000-RETURN-TO-CALLER
         WHEN SQLCODE < 0
              PERFORM 9999-FORMAT-DB2-MESSAGE
              PERFORM 9999-ABEND
     END-EVALUATE.

Cursor-Based Paging Pattern (COTRTLIC)

Forward/backward paging re-opens the cursor with a WHERE anchor based on the last-seen type code:

* Declare cursor (declared before precompile, referenced at runtime):
     EXEC SQL
          DECLARE TRTCURS CURSOR FOR
          SELECT TR_TYPE, TR_DESCRIPTION
            FROM CARDDEMO.TRANSACTION_TYPE
           WHERE TR_TYPE >= :WS-PAGE-ANCHOR
           ORDER BY TR_TYPE
     END-EXEC.

* Fetch loop for one page (7 rows):
     EXEC SQL OPEN TRTCURS END-EXEC.
     PERFORM VARYING I FROM 1 BY 1
             UNTIL I > WS-MAX-SCREEN-LINES
                  OR SQLCODE = +100
         EXEC SQL
              FETCH TRTCURS
               INTO :DCL-TR-TYPE, :DCL-TR-DESCRIPTION
         END-EXEC
         * populate screen row I from host variables...
     END-PERFORM.
     EXEC SQL CLOSE TRTCURS END-EXEC.

DB2 Error Handling (CSDB2RPY.cpy)

The shared copybook CSDB2RPY.cpy provides:

  • 9998-PRIMING-QUERY: Verifies DB2 connectivity via SELECT 1 FROM SYSIBM.SYSDUMMY1.
  • 9999-FORMAT-DB2-MESSAGE: Calls DSNTIAC to extract human-readable error text from SQLCA and formats it into WS-LONG-MSG.
9999-FORMAT-DB2-MESSAGE.
     CALL LIT-DSNTIAC USING DFHEIBLK
                            DFHCOMMAREA
                            SQLCA
                            WS-DSNTIAC-FORMATTED
                            WS-DSNTIAC-LRECL
     END-CALL.
     STRING
          FUNCTION TRIM(WS-DB2-CURRENT-ACTION)
                   ' SQLCODE:'
                   WS-DISP-SQLCODE
                   WS-DSNTIAC-FMTD-TEXT
          DELIMITED BY SIZE
     INTO WS-LONG-MSG
     END-STRING.
     MOVE WS-LONG-MSG TO WS-RETURN-MSG.

BMS Screen Design Conventions

  • Row 1: Transaction ID (TRNNAME), screen title (TITLE01), current date (CURDATE).
  • Row 2: Program name (PGMNAME), subtitle (TITLE02), current time (CURTIME).
  • Info message field: INFOMSG (ASKIP, used for success confirmations).
  • Error message: ERRMSGO (highlighted, used for validation failures).
  • All editable input fields use HILIGHT=UNDERLINE and ATTRB=(UNPROT).

COBOL Coding Conventions

  • Flag fields use 88-level condition names with clear semantics (e.g., FLG-ALPHNANUM-ISVALID, FLG-ALPHNANUM-NOT-OK, FLG-ALPHNANUM-BLANK).
  • All DB2 operations are wrapped in EVALUATE TRUE blocks immediately checking SQLCODE.
  • Screen array data (7 rows) stored in contiguous OCCURS table, traversed with I subscript.
  • CICS COMMAREA used to pass state between pseudo-conversational interactions.
  • INSPECT statements use literal tables (LIT-ALL-ALPHANUM-FROM-X) for character-class validation.

11. Integration with Core Application

Admin Menu Hook

The admin menu program COADM01C (transaction CA00) provides options 5 and 6 which XCTL to COTRTLIC (CTLI) and COTRTUPC (CTTU) respectively. Both programs XCTL back to COADM01C on PF3 exit using literal LIT-ADMINPGM = 'COADM01C' and LIT-ADMINTRANID = 'CA00'.

Nightly Data Pipeline

DB2 CARDDEMO.TRANSACTION_TYPE
         |
         |  TRANEXTR (nightly JCL)
         v
AWS.M2.CARDDEMO.TRANTYPE.PS  ------>  COTRN00C
AWS.M2.CARDDEMO.TRANCATG.PS  ------>  (transaction list screen)
                                       resolves DALYTRAN-TYPE-CD
                                       to human-readable labels

VSAM Field Alignment

The DALYTRAN-TYPE-CD PIC X(02) field in daily transaction records (copybook CVTRA06Y) maps directly to TR_TYPE in DB2 and to bytes 1-2 of the TRANTYPE.PS record. This alignment ensures transaction list screens resolve type codes to current descriptions after each TRANEXTR run.

12. Performance Considerations

OperationBudgetNotes
CTLI page flip (PF7/PF8) P95 < 300 ms DB2 cursor re-open per page; indexed on TR_TYPE primary key. No full table scan expected for typical ref-data table sizes (<200 rows).
CTTU add/update (PF5) P95 < 500 ms Single-row INSERT or UPDATE with PK access path.
CICS overall transaction response P95 < 1.5 s Platform-wide SLA defined in system overview for all CICS interactions.
TRANEXTR nightly job < 5 min Bounded by table row count; typically <100 rows in reference tables.
MNTTRDB2 bulk load < 1 min per 1000 rows Sequential file read + individual DML statements. For large initial loads, consider LOAD utility instead.
