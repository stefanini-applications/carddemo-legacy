ЁЯУж core-batch

Guide for crafting offline jobs that fuel the CardDemo datasets.

ЁЯУЛ Overview

Purpose: Maintain VSAM data accuracy, post transactions, and produce statements via orchestrated JCL flows.

Business Context: Batch jobs keep the demo environment consistent for every user story; they are the source of truth for `core-online` screens.

ЁЯОп User Story Development Guide

ЁЯУЭ Module-Specific US Templates

  • Data Refresh: As a Batch Operator, I want to refresh `ACCTFILE` before posting so that account data is complete for front-end validation.
  • Posting Sequence: As a Compliance Analyst, I want `POSTTRAN` to follow `TRANFILE` and `TRANBKP` so that the ledger reflects all activity.
  • Reporting: As a Reporting Owner, I want `CREASTMT` to create a statement file immediately after `COMBTRAN` so that statements include the posting window.

тЪб Development Acceleration Factors

  • JCL templates: Jobs named after their function (ACCTFILE, TRANEXTR, POSTTRAN) already exist under app/jcl/.
  • Automation scripts: `scripts/run_full_batch.sh` and `scripts/remote_submit.sh` wrap the job dependencies.
  • Control cards: `app/ctl/` and `samples/jcl/` provide ready-to-use IDCAMS/SORT steps.

ЁЯУК Complexity Guidelines

  • Simple (1-2 pts): Update an existing job parameter or add a new dataset to `ACCTFILE` without touching shared VSAM files.
  • Medium (3-5 pts): Introduce a new dependent job (e.g., a variant of `TRANBKP`) that runs after posting steps and updates indexes.
  • Complex (5-8 pts): Reorder the entire posting pipeline or integrate a new optional module (authorization purge) while keeping job sequencing intact.

тЬЕ Acceptance Criteria Examples

  • Each job must end with IDCAMS condition code 0000 and update the correct VSAM dataset.
  • Job dependencies (ACCTFILE тЖТ TRANFILE тЖТ POSTTRAN тЖТ CREASTMT) are recorded in release notes.
  • Scripting wrappers handle job submission order and HLQ substitution before hitting the mainframe.

ЁЯФз Technical Foundation

ЁЯПЧя╕П Key Components

  • JCL jobs under app/jcl/ (ACCTFILE, CARDFILE, TRANFILE, POSTTRAN, COMBTRAN, CREASTMT, TRANIDX, INTCALC, etc.).
  • Control cards in app/ctl/ and helper samples under samples/jcl/.
  • Batch COBOL programs (`CBTRN02C`, `CBSTM03A`, `CBACT04C`, `CBPAUP0C`).
  • Scheduler definitions (`app/scheduler/CardDemo.ca7`, `.controlm`).

ЁЯФЧ Dependencies

  • Tight coupling to `core-online` (VSAM files must be ready before CICS screens read them).
  • Feeds optional modules: approval/purge for `authorization-ims-db2-mq`, transaction type refresh for `transaction-type-db2`.
  • Relies on RACF identity (CC00) because jobs run under specific HLQ owners.
