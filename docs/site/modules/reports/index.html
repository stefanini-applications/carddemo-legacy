📋 Module Overview

Purpose: Generate transaction reports and account statements via batch job submission from CICS, plus standalone batch report generation.

Business Context: Reporting closes the financial cycle – cardholders and finance teams need summarized views of transactions. CICS submits batch jobs through the Internal Reader; batch programs then generate reports using GDG for versioned output.

Transactions: CR00 (CICS report submission)  |  Online Program: CORPT00C
Batch Programs: CBTRN03C (transaction report), CBSTM03A/CBSTM03B (statement)

COBOL CICS COBOL Batch JCL Internal Reader GDG VSAM TRANSACT

🎯 User Story Templates

US-RPT-001: As a cardholder, I want to submit a transaction report request from the CICS screen so that a report is generated for my account.
US-RPT-002: As a finance team member, I want to run the TRANREPT batch job so that a detailed transaction report is produced.
US-RPT-003: As a finance team member, I want to run the CREASTMT batch job so that monthly account statements are generated.
US-RPT-004: As a batch operator, I want statements stored in GDG datasets so that historical versions are retained automatically.

✅ Acceptance Criteria Patterns

CICS Report Submission:
GIVEN an authenticated cardholder on the CR00 screen
WHEN the cardholder submits the report request
THEN a JCL job is submitted via Internal Reader and a submission confirmation is displayed
Batch Transaction Report:
GIVEN the TRANREPT job runs CBTRN03C
WHEN the job completes
THEN a transaction detail report is produced with all transactions from TRANSACT, return code = 0
Statement Generation:
GIVEN the CREASTMT job runs CBSTM03A
WHEN the job completes
THEN statements are written to the GDG dataset, return code = 0, previous generations retained

Report Generation Architecture

  1. Cardholder or admin selects CR00 in CICS
  2. CORPT00C collects parameters (date range, account ID)
  3. CICS submits JCL to Internal Reader (INTRDR dataset)
  4. z/OS job scheduler picks up and runs the batch job
  5. CBTRN03C or CBSTM03A reads TRANSACT sequentially
  6. Report output written to sysout or GDG dataset

GDG (Generation Data Groups)

  • Statements use GDG for automatic versioning – each run creates a new generation
  • DEFGDGB job defines GDG bases before first use
  • Previous statement generations are automatically retained per GDG LIMIT setting

Dependencies

  • authentication: CICS session required for online report submission
  • transactions: TRANSACT VSAM is the data source for all reports
  • batch-processing: COMBTRAN must run before CREASTMT to merge transaction files
  • DEFGDGB: GDG base must be defined before first statement run

⚡ Development Acceleration Factors

  • CORPT00C Internal Reader pattern: Established CICS-to-batch submission mechanism; reuse for other on-demand batch triggers
  • CBTRN03C sequential read pattern: Standard VSAM sequential read for reporting – reuse as base for new report programs
  • PROC TRANREPT: Reusable JCL procedure – parameterize for different report types
