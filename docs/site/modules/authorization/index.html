🛡️ Authorization Extension

User story playbook for MQ-driven authorization flows that touch IMS, Db2, and VSAM.

📋 Overview

Purpose: Handle pending authorization requests from MQ, apply business rules, persist them in IMS/Db2, and give analysts screens for review.

Business Context: The MQ request queue `AWS.M2.CARDDEMO.PAUTH.REQUEST` is triggered by cloud clients; CICS programs `COPAUA0C`, `COPAUS0C`, and `COPAUS1C` keep IMS segments and `AUTHFRDS` table synchronized.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Authorization Processing: As a fraud analyst, I want PF5 on CPVD to mark an authorization as fraudulent so that the record writes to Db2 `AUTHFRDS` and downstream reports flag it.
  • MQ Format: As a cloud integrator, I want queue-to-program correlation to rely on the documented comma-separated layout so that APIs can parse responses to the `AUTH-ID-CODE` field.
  • Purge Workflow: As an operator, I want `CBPAUP0J` to delete authorizations older than 30 days so that IMS disk usage stays within budget.

⚡ Development Acceleration Factors

  • Copybooks: `CIPAUSMY`, `CIPAUDTY`, `CCPAURQY`, `CCPAURLY` centralize message layouts for IMS and MQ.
  • MQ Setup: Follow the queue names in the README; use `COPAUA0C` to inspect message handling before replicating the pattern.
  • Db2 Fraud Store: Db2 table `AUTHFRDS` includes `AUTH_FRAUD`, `FRAUD_RPT_DATE`; reusing these columns accelerates analytics stories.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new MQ field to the request layout and surface it in the summary screen `CPVS`.
  • Medium (3-5 pts): Extend `CBPAUP0J` to purge IMS segments based on a new status column and update the MQ response.
  • Complex (5-8 pts): Build a new MQ consumer that evaluates fraud rules, updates IMS, and triggers a Db2 insert for every event.

🔧 Technical Foundation

🏗️ Key Components

  • CICS programs: `COPAUA0C` (MQ trigger processor), `COPAUS0C` (summary), `COPAUS1C` (detail), `COPAUS2C` (fraud tagging).
  • Batch job: `CBPAUP0J` (expired purge) and `CBPAUP0C` COBOL logic.
  • Databases: IMS segments `PAUTSUM0`, `PAUTDTL1` plus Db2 table `AUTHFRDS`.
  • MQ: Queues `AWS.M2.CARDDEMO.PAUTH.REQUEST` / `PAUTH.REPLY`.

🔗 Dependencies

  • VSAM: Account data from `COACTVWC` ensures authorization context.
  • Db2: Fraud records replicate into analytics pipelines (optional integration).
  • core-batch: Purge job runs every night; ensure `CBPAUP0J` appears in `run_full_batch.sh`.

🌐 Public Interfaces

MQ Request to AWS.M2.CARDDEMO.PAUTH.REQUEST
COPAUA0C (CP00) - Validates request, writes IMS segments, replies on PAUTH.REPLY
CPVS/COPAUS0C - Summary screen for pending authorizations
CPVD/COPAUS1C - Detail screen + PF actions (PF5 to mark fraud)
CBPAUP0J/CBPAUP0C - Batch purge of IMS authorizations older than 30 days
DB2 AUTHFRDS - Records flagged fraudulent for analytics

🧪 Acceptance Criteria & Patterns

  • Message Integrity: Requests must follow the documented comma-separated layout and responses must align with correlation IDs.
  • Error Handling: MQ errors go to `CCPAUERY`; BMS message area (defined in `CSMSG01Y`) must show failure text.
  • Performance: MQ round trip < 5 seconds; purges cannot block the nightly `run_full_batch.sh` sequence.
